# 입력자료(inputs/) 규칙

> 하네스 에이전트가 분석·설계할 때 참조하는 **외부 입력물**을 배치하는 규약. 작업 전에 사용자가 미리 채워둠.

## 1. 위치

### 단일 통합 입력 폴더
```
inputs/
```
프로젝트 루트(vault 밖)에 위치. 모든 표준 분석 시 공통 참조.
표준 원문·법규·해설서·AsIs·산업가이드를 단일 구조로 관리.

## 2. 하위 폴더 구조 (권장)

```
inputs/
├── 01_표준원문/         ← ISO/IEC/KS 원본 PDF·TXT (라이선스 보유 시)
├── 02_법규/             ← 국내 법령 PDF·MD (law.go.kr 발췌)
├── 03_해설서/           ← 인증기관·심사원 교재·산업협회 가이드
├── 04_AsIs/             ← 기존 조직 정책·SOP·절차서 + 개정 큐 파일
├── 05_산업가이드/       ← 업종별 best practice (NIST·ENISA·식약처 등)
└── README.md            ← 입력물 목록·출처·라이선스 상태 요약
```

> 서브폴더가 없어도 동작. 단, 혼재 시 에이전트가 분류 정확도 낮을 수 있음.

## 3. 에이전트 동작 규약

**모든 분석 에이전트는 inputs/ 선행 스캔을 의무화**:

1. `inputs/` 존재 확인 → 없으면 사용자에게 경고
2. 있으면 전수 스캔 (PDF·MD·TXT 모두 읽기)
3. 각 파일을 카테고리로 분류 후 요약
4. 분석 결과에 근거로 인용 (source_citation 필수)
5. 입력물에 없는 사실 추정 시 `[inputs 미제공 — LLM 추정]` 꼬리표

## 4. source_citation 스키마

모든 요구사항/산출물 본문에 근거를 명시:

```yaml
source_citation:
  - type: standard_original    # standard_original | law | guide | as_is | industry | llm_inference | web
    file: "inputs/01_표준원문/ISO9001_2015.pdf"
    locator: "Section 7.5.3 p.18"
    retrieved_at: "2026-04-17"
    license: "ISO copyright — 내부 참조 only, 원문 인용 금지"
    paraphrase_only: true
  - type: law
    file: "inputs/02_법규/개인정보보호법.md"
    locator: "제29조"
    retrieved_at: "2026-04-17"
    license: "공공저작물"
    paraphrase_only: false
```

본문 마크다운에서는 축약 가능: `(출처: ISO9001 §7.5.3)` 하되, 상응 frontmatter 또는 각주에 전체 스키마 보존.

## 5. 출처 우선순위

에이전트가 상충 정보를 받았을 때 신뢰도 순위:

```
1. 국내 법규 (법적 강제)
2. 라이선스 보유 표준 원문 (`inputs/01_표준원문/`)
3. 공식 해설서·인증기관 가이드 (`inputs/03_해설서/`)
4. 산업 가이드 (NIST, ENISA 등)
5. 기존 자산 As-Is (`inputs/04_AsIs/`)  ← 존중하되 표준 대비 gap 지적
6. LLM 내부 지식
7. Web fetch (가장 마지막, 출처 URL 필수 기록)
```

## 6. 저작권 가드레일 (CRITICAL)

| 입력물 유형 | 가능 | 금지 |
|---|---|---|
| ISO/IEC 원문 | 요지 paraphrase, 조항번호·제목 인용 | **원문 문구 직접 복사** (침해) |
| KS 표준 | 동일 (라이선스 조건 확인) | 동일 |
| 국내 법규 | 원문 인용 가능 (공공저작물) | — |
| 공개 가이드(NIST 등) | 원문 인용 가능 (저작권 표기) | — |
| 고객사 As-Is | 외부 배포 불가, 내부 참조만 | 타 고객 산출물에 유출 |

→ qa-reviewer 가 생성물 내 `inputs/01_표준원문/` 파일과 **20단어 이상 연속 일치** 검사, 탐지 시 부적합.

## 7. 사용자 워크플로우

```bash
# ① 작업 전: inputs/ 에 입력물 투하
cp ~/Downloads/KS_Q_ISO9001.pdf inputs/01_표준원문/
cp ~/Documents/개인정보보호법.pdf inputs/02_법규/
cp ~/Documents/현_품질매뉴얼_v3.pdf inputs/04_AsIs/

# ② 편입 실행
/process-plan "OOO사 품질경영체계"

# ③ 결과물은 inputs/ 를 근거로 생성되며 source_citation 포함
# ④ 적용요건.md → vault/02_적용요건/{슬러그}/적용요건.md 자동 생성
```

## 8. 검증 규칙 (qa-reviewer 체크 포인트)

- [ ] `inputs/README.md` 에 각 파일 라이선스 상태 명시
- [ ] 생성된 POL/PRO/WI 중 `source_citation` 누락 문서 없음
- [ ] 라이선스 없는 표준 원문 문구 직접 복사 없음 (20단어 일치 검사)
- [ ] `[inputs 미제공 — LLM 추정]` 꼬리표 있는 요구사항은 작업노트에 이슈로 등록

## 9. 초기 구조 (inputs/ 폴더)

`inputs/` 는 프로젝트 루트에 이미 생성되어 있으며 5개 카테고리 폴더와 README 를 포함한다.

### inputs/ 구조
```
inputs/
├── README.md                    ← 인벤토리 템플릿 + git 정책
├── 01_표준원문/README.md        ← 카테고리별 안내
├── 02_법규/README.md
├── 03_해설서/README.md
├── 04_AsIs/README.md            ← AsIs 자산 + 개정 큐 파일
└── 05_산업가이드/README.md
```

### inputs/README.md 인벤토리 스텁
```markdown
## 현재 입력물 인벤토리
| 파일 | 카테고리 | 출처 | 라이선스 | 우선순위 | content_mode | 비고 |
|---|---|---|---|---|---|---|
| (추가 시 기록) | | | | ⭐⭐⭐ | paraphrase | |
```

## 10. 하이브리드 구조 (PDF + MD 변환본) — 권장

### 10.1 왜 MD 변환본을 함께 두는가
PDF 원본만 있으면 에이전트가:
- 페이지 지정 Read 필요 (10페이지 초과 시 필수)
- Grep 불가 (조항 검색 어려움)
- 토큰 낭비 (레이아웃·폰트 오염)

→ **같은 이름 폴더** 에 MD 로 변환한 서브볼트를 두면 위 문제 모두 해소.

### 10.2 폴더 패턴

```
inputs/01_표준원문/
├── ISO9001_2015.pdf              ← 원본(내부 대조용 보관)
└── ISO9001_2015/                  ← 변환 MD 서브볼트 (같은 이름 폴더)
    ├── MOC_ISO9001.md             ← 조항 인덱스
    ├── 00_개요.md
    ├── 04_조직_상황.md
    ├── 05_리더십.md
    ├── 06_기획.md
    ├── 07_지원/
    │   ├── 07.1_자원.md
    │   ├── 07.2_역량.md
    │   └── 07.5_문서화된_정보.md
    ├── 08_운영.md
    ├── 09_성과평가.md
    └── 10_개선.md
```

법규·해설서도 동일 패턴 (`개인정보보호법.pdf` + `개인정보보호법/` 폴더).

### 10.3 변환 MD 의 frontmatter
반드시 `type: input_document` 로 구분. 템플릿: [[T15_입력문서_INP]]

주요 필드:
- `source_type`: standard_original | law | guide | as_is | industry
- `source_file`: 원본 PDF 파일명
- `source_pages`: 원본 페이지 범위
- `clause`, `clause_title`: 조항 정보
- `license`: 라이선스 상태
- **`content_mode`**: `verbatim` | `paraphrase` | `summary`

### 10.4 content_mode 3단계

| 모드 | 용도 | 저작권 |
|---|---|---|
| `verbatim` | 원문 그대로 텍스트 추출 (pandoc·marker 등) | 원본 PDF 와 동일 보호 — 산출물에 직접 인용 금지 |
| `paraphrase` | 조항별 요지 재작성 | **산출물 본문에 조항번호와 함께 인용 가능** |
| `summary` | 핵심만 1~2줄 요약 | 가장 안전 |

→ 에이전트는 `content_mode: verbatim` 인 MD 를 원본 PDF 와 **동일 저작권 수준** 으로 취급.

### 10.5 에이전트 우선순위
`inputs/01_표준원문/` (변환본) 을 **PDF 원본보다 먼저** 스캔. 같은 조항에 두 소스가 있으면 MD 우선, PDF 는 페이지 대조용.

### 10.6 변환 방법 가이드

| 대상 | 권장 방법 | 저작권 |
|---|---|---|
| ISO/IEC 표준 원문 | 수동 `paraphrase/summary` MD 변환 | 안전 |
| 국내 법규 (law.go.kr) | `pandoc` 또는 `marker` 로 verbatim MD | 공공저작물 — verbatim OK |
| 인증기관 해설서 | `paraphrase` 변환 | 발행처 라이선스 확인 |
| 고객사 As-Is | 원본 그대로 복사 (MD 있으면 MD, 없으면 PDF) | 내부 자산 |

### 10.7 MOC 파일 (변환 서브볼트용)
서브볼트 루트에 `MOC_{표준}.md` 를 두어 조항 인덱스 제공:

```markdown
---
type: input_moc
title: ISO 9001 조항 인덱스
---

# ISO 9001:2015 — 조항 인덱스

| 조항 | 제목 | MD 파일 | content_mode |
|---|---|---|---|
| 4.1 | 조직 상황 | [[04_조직_상황]] | paraphrase |
| 7.5 | 문서화된 정보 | [[07_지원/07.5_문서화된_정보]] | paraphrase |
```

에이전트가 이 MOC 를 먼저 읽고 필요한 조항만 선별 Read 가능.
