# 표준 분류 레지스트리

모든 편입 대상 표준의 **레이어·구조·통합 모드**를 단일 출처로 관리. `standard-analyzer`·`process-designer`·`qa-reviewer` 가 편입·QA 시점에 이 레지스트리를 조회한다.

---

## 1. 분류 3축

### 1.1 Layer (적용 레이어)

| 값 | 의미 | 예 |
|---|---|---|
| `L1_management` | 경영시스템 (전사 거버넌스·지원) | ISO 9001, ISO 27001, IATF 16949 |
| `L2_engineering` | 제품/프로세스 엔지니어링 | ASPICE, ISO 26262, IEC 62304 |
| `L3_reference` | 참조·프레임워크 (별도 POL/PRO 생성 안 함) | ISO 31000 |

### 1.2 Structure (원본 구조)

| 값 | 의미 |
|---|---|
| `HLS` | Annex SL 10조항 표준 |
| `quasi_HLS` | HLS 이전 유사 8조항 (ISO 13485:2016) |
| `capability_model` | 성숙도 평가 모델 (ASPICE) |
| `part_lifecycle` | Part 기반 수명주기 (ISO 26262) |
| `phase_lifecycle` | Phase 기반 수명주기 (ISO/SAE 21434, IEC 81001-5-1) |
| `sw_lifecycle` | SW 수명주기 (IEC 62304) |
| `risk_lifecycle` | 리스크 관리 수명주기 (ISO 14971) |
| `risk_framework` | 리스크 프레임워크 가이드 (ISO 31000) |

### 1.3 Integration Mode (통합 방식)

| 값 | 동작 | 에이전트 처리 |
|---|---|---|
| `hls_merge` | HLS 공통 조항 기반 통합 | 기존 POL/PRO 확장 우선 (기본) |
| `quasi_hls_merge` | 핵심 공통(문서·경영검토 등)만 통합, 나머지 분리 | 영역코드 분리 + 일부 경계면 확장 |
| `interface_only` | 독립 체계 + 경계면 PRO 만 교차 | 영역코드 분리, 공급자/변경/구성관리만 통합 |
| `reference_only` | 별도 산출물 생성 안 함 | 기존 POL/PRO 에 주석·인용 + REF 만 생성 |

## 2. 등록된 표준

### 2.1 HLS 기반 경영시스템 (L1 · hls_merge)

| 표준 | 영역코드 | Structure | 상위/관계 |
|---|---|---|---|
| ISO 9001 | QMS | HLS | — |
| ISO 14001 | EMS | HLS | — |
| ISO 45001 | OHSMS | HLS | — |
| ISO/IEC 27001 | ISMS | HLS | — |
| ISO/IEC 27701 | PIMS | HLS | ISO 27001 확장 |
| ISO/IEC 20000 | ITSM | HLS | — |
| ISO 22301 | BCMS | HLS | — |
| ISO/IEC 42001 | AIMS | HLS | — |
| **IATF 16949** | **AUTO** | **HLS** | **ISO 9001 확장 (자동차)** |

### 2.2 의료기기 도메인

| 표준 | 영역코드 | Layer | Structure | Integration | 비고 |
|---|---|---|---|---|---|
| **ISO 13485** | MDQMS | L1 | **quasi_HLS** | **quasi_hls_merge** | 의료기기 QMS. 의도적 HLS 비준수 |
| **ISO 14971** | MDRM | L2 | risk_lifecycle | interface_only | 의료기기 리스크 관리. ISO 13485 §7.1 에서 참조 |
| **IEC 62304** | MDSW | L2 | sw_lifecycle | interface_only | 의료기기 SW 수명주기 (Safety class A/B/C) |
| **IEC 81001-5-1** | MDCS | L2 | phase_lifecycle (security) | interface_only | 헬스 SW/IT 시스템 사이버보안 |

### 2.3 자동차 도메인

| 표준 | 영역코드 | Layer | Structure | Integration | 비고 |
|---|---|---|---|---|---|
| **IATF 16949** | AUTO | L1 | HLS | hls_merge | ISO 9001 위에 자동차 특화 |
| **ASPICE** | ASPICE | L2 | capability_model | interface_only | SW 프로세스 성숙도. HLS 비준수 |
| **ISO 26262** | FUSA | L2 | part_lifecycle | interface_only | 기능안전. 12 Part. V-model |
| **ISO/SAE 21434** | VCSMS | L2 | phase_lifecycle | interface_only | 자동차 사이버보안 |

> ⚠️ `VCSMS` 는 기존 `CSMS`(침해사고 대응) 와 혼동 방지를 위한 별도 코드.

### 2.4 참조·특수

| 표준 | 영역코드 | Layer | Structure | Integration | 비고 |
|---|---|---|---|---|---|
| ISO 31000 | RM | L3 | risk_framework | reference_only | 리스크 관리 원칙·가이드 |
| ISO/IEC/IEEE 12207 | SWLC | L2 | sw_lifecycle | interface_only | SW 수명주기 프로세스 |
| ISO/IEC/IEEE 15288 | SYSLC | L2 | phase_lifecycle | interface_only | 시스템 수명주기 |
| ISO/IEC 29110 | VSE | L2 | sw_lifecycle | interface_only | 초소기업 SW 수명주기 |

### 2.5 프로세스 성숙도 모델

| 표준 | 영역코드 | Layer | Structure | Integration | 비고 |
|---|---|---|---|---|---|
| **ASPICE** (위 자동차에서 재게시) | ASPICE | L2 | capability_model | interface_only | SW 프로세스 성숙도 |
| **CMMI-DEV v3.0 (ML3)** | CMMI | L2 | capability_model | interface_only | 개발 조직 성숙도. 19+ PA, ML2/ML3 Practice |

## 3. 표준 간 Interface 예 (경계면 통합 지점)

`interface_only` 표준은 아래 경계면에서만 상위 L1 과 통합:

| 경계면 | L1 (HLS) 조항 | L2 표준 요구 |
|---|---|---|
| 문서관리 | §7.5 | ASPICE SUP.7, 26262 Part 8, 62304 §5.1.1 |
| 역량관리 | §7.2 | — (대부분 L1 으로 통합) |
| 공급자관리 | §8.4 | 26262 Part 8, 21434 §15, 62304 §5.1.5 |
| 변경관리 | §8.5 | ASPICE SUP.10, 26262 Part 6, 62304 §6 |
| 구성관리 | §7.5 확장 | ASPICE SUP.8, 26262 Part 6, 62304 §8 |
| 리스크 거버넌스 | §6.1 | 14971, 26262 Part 3, 21434 §6 |
| 내부심사 | §9.2 | — (L1 전용) |

→ MAT-07 Interface 테이블(향후) 에서 상세 관리.

## 4. 에이전트 동작 규칙

### 4.1 standard-analyzer
- Phase 0 선행 스캔 시 본 레지스트리에서 **대상 표준의 layer/structure/integration_mode** 조회
- `_state.yaml` 의 `phases.analyze.metrics` 에 이 메타 기록
- 레지스트리 미등록 표준 발견 시: 사용자에게 **"layer/integration_mode 기본값(L1_management/hls_merge)로 진행 OK? 레지스트리 등록 필요"** 질의

### 4.2 process-designer (핵심 분기)
레지스트리의 `integration_mode` 에 따라 설계 전략 분기:

```
integration_mode 조회
├─ hls_merge
│    → 기존 POL/PRO grep → 동일 목적 있으면 standards[] 배열 확장
│    → 영역코드 충돌 시 새 POL/PRO (기존 패턴 유지)
│
├─ quasi_hls_merge
│    → 경계면 조항(문서관리·경영검토 등)만 기존 확장
│    → 그 외는 전용 영역코드(MDQMS 등)로 독립 생성
│
├─ interface_only
│    → 독립 체계로 전용 영역코드 생성 (ASPICE/FUSA/VCSMS/MDRM/MDSW/MDCS 등)
│    → 상위 L1 의 경계면 PRO 만 "참조" 링크 추가 (통합 안 함)
│    → MAT-07 Interface 테이블에 경계면 기록
│
└─ reference_only
     → POL/PRO 생성 금지
     → REF 만 생성 (09_REF_참고자료/)
     → 기존 관련 POL/PRO 의 본문에 "인용" 주석 추가
```

### 4.3 qa-reviewer (§12 신규 체크)
- 모든 산출물 frontmatter 의 `standards[]` 배열 확인
- 각 표준의 `integration_mode` 대비 배치 위치 정합성 검증:
  - `interface_only` 표준이 HLS 영역코드 POL/PRO 에 섞여 있으면 **FAIL** (레이어 혼재)
  - `reference_only` 표준이 POL/PRO 에 `standards` 로 선언되어 있으면 **경고** (REF 만 허용)

---

## 5. 신규 표준 등록 프로세스

1. 표준 공식명·발행기관·발행년도 확보
2. 본 레지스트리 §2 해당 섹션에 Row 추가
3. 영역코드 할당 ([[02_문서번호체계]] §영역 코드 표에도 동반 등록)
4. `layer`/`structure`/`integration_mode` 판정
5. Interface 표준이면 §3 경계면 표에도 Row 추가
6. 변경은 PCB 심의 대상

## 6. 메타 모델 (YAML)

향후 자동화 시 파싱용 JSON/YAML 포맷 예시:

```yaml
standards:
  - code: ISO13485
    full_name: "Medical devices — Quality management systems"
    publisher: ISO
    year: 2016
    layer: L1_management
    structure: quasi_HLS
    integration_mode: quasi_hls_merge
    scope_codes: [MDQMS]
    parent_standard: null
    interface_with: [ISO14971, IEC62304, IEC81001-5-1]
    notes: "의도적으로 HLS 채택하지 않음. 구조는 ISO 9001:2008 기반."

  - code: ISO14971
    full_name: "Medical devices — Application of risk management"
    publisher: ISO
    year: 2019
    layer: L2_engineering
    structure: risk_lifecycle
    integration_mode: interface_only
    scope_codes: [MDRM]
    interface_with: [ISO13485_§7.1, IEC62304_§7]
    notes: "리스크 관리 수명주기. ISO 13485 §7.1 에서 참조 필수."

  - code: IEC62304
    full_name: "Medical device software — Software life cycle processes"
    publisher: IEC
    year: 2015  # +A1:2015 포함
    layer: L2_engineering
    structure: sw_lifecycle
    integration_mode: interface_only
    scope_codes: [MDSW]
    interface_with: [ISO13485, ISO14971, IEC81001-5-1]
    notes: "SW Safety class A/B/C 결정 필수."

  - code: IEC81001-5-1
    full_name: "Health software and health IT systems safety, effectiveness and security — Part 5-1: Security — Activities in the product life cycle"
    publisher: IEC
    year: 2021
    layer: L2_engineering
    structure: phase_lifecycle
    integration_mode: interface_only
    scope_codes: [MDCS]
    interface_with: [IEC62304, ISO13485]
    notes: "IEC 62443-4-1 구조 채택. FDA Premarket Cybersecurity, EU MDR 요구."

  - code: CMMI-DEV-ML3
    full_name: "Capability Maturity Model Integration for Development (CMMI-DEV) v3.0 — Maturity Level 3"
    publisher: ISACA / CMMI Institute
    year: 2023
    layer: L2_engineering
    structure: capability_model
    integration_mode: interface_only
    scope_codes: [CMMI]
    interface_with: [ISO9001, ISO12207, ISO15288, ASPICE]
    notes: "19+ Practice Areas (Core/Development/Supplier). ML3 = ML2 PA 전체 + ML3 PA. PA별 Required/Expected/Informative 구조. 평가 모델."
```
