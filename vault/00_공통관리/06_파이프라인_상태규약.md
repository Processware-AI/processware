# 파이프라인 상태 규약

체크포인트/재개 메커니즘과 자가수정(self-correction) 루프의 단일 규약.

## 0. 적용 범위

이 규약은 **4차원 PDCA 전 차원**에 공통으로 적용된다. 차원별 상태 파일 위치와 주관 에이전트가 다르다.

| 차원 | 커맨드 | 상태 파일 경로 | 비고 |
|---|---|---|---|
| **0 (Ingest)** | `/process-ingest` | `.claude/states/{표준코드}_ingest_state.yaml` | 9 Phase + HITL 확정 |
| **1 (Plan)** | `/process-plan` | `.claude/states/{슬러그}_state.yaml` | 본 규약 주 대상. 6 phase + 자가수정 루프 |
| **2 (Do)** | `/process-do` | `.claude/runs/run-{hex}/trace.jsonl` | JSONL trace. HITL 정지·재개는 hitl-gatekeeper |
| **2B (Backfill)** | `/process-backfill` | `.claude/runs/run-b{hex}/trace.jsonl` | do와 동일 구조 |
| **3 (Check)** | `/process-check` | `.claude/runs/run-a{hex}/trace.jsonl` | audit/kpi/gap 모두 trace 기반 |
| **3 (Audit)** | `/process-audit` | `.claude/runs/run-g{hex}/trace.jsonl` | GAP 분석 전용 |
| **4 (Act)** | `/process-act` | `.claude/runs/run-c{hex}/trace.jsonl` | PCB HITL 정지 포함 |

> **본 §1~§8은 차원 1(Plan) `_state.yaml` 기반 규약이 주체**이며, 차원 2~4는 trace.jsonl 기반으로 별도 구현됨. 공통 원칙(체크포인트, 재개, 실패 처리)은 동일하게 적용.

## 1. 상태 파일 위치
```
.claude/states/{슬러그}_state.yaml
```
적용요건 하나당 1개. 파이프라인 실행 시작 시 생성, 각 phase 종료 시 갱신.
슬러그는 `/process-plan "적용요건명"` 입력을 소문자·하이픈으로 변환한 식별자.

## 2. 스키마

```yaml
slug: "ooo-품질경영체계"        # 적용요건 슬러그 (자유 명칭 → 소문자·하이픈)
req_title: "OOO사 품질경영체계"  # 원본 적용요건 명칭
req_path: "vault/02_적용요건/OOO사_품질경영체계/적용요건.md"
pipeline_version: "1.0"
created: "2026-04-17T01:00:00Z"
updated: "2026-04-17T01:20:00Z"
overall_status: "running"      # pending | running | done | halted
current_phase: "design"         # preflight | analyze | design | write | trace | qa | done
attempts: 1                     # self-heal 루프 반복 횟수
max_attempts: 3

phases:
  preflight:
    status: done                # pending | running | done | failed | skipped
    agent: orchestrator
    started: "2026-04-17T01:00:00Z"
    completed: "2026-04-17T01:05:00Z"
    artifacts:
      - "inputs/README.md"
    metrics:
      input_files: 7
      categories: {standard_original: 1, law: 3, guide: 2, as_is: 1}
    notes: "입력자료 7개 확인, 라이선스 전수 OK"
  analyze:
    status: done
    agent: standard-analyzer
    started: "2026-04-17T01:05:00Z"
    completed: "2026-04-17T01:20:00Z"
    artifacts:
      - "vault/02_적용요건/OOO사_품질경영체계/적용요건.md"
    metrics:
      req_total: 85
      req_mandatory: 67
      req_recommended: 18
      citation_coverage: 0.94
  design:
    status: running
    agent: process-designer
    started: "2026-04-17T01:20:00Z"
  write:
    status: pending
    agent: wi-tmp-writer
  trace:
    status: pending
    agent: traceability-mapper
  qa:
    status: pending
    agent: qa-reviewer

history:
  - ts: "2026-04-17T01:00:00Z"
    event: pipeline_start
    attempt: 1
  - ts: "2026-04-17T01:05:00Z"
    event: phase_completed
    phase: preflight
  - ts: "2026-04-17T01:20:00Z"
    event: phase_completed
    phase: analyze

qa_failures: []   # QA fail 이슈 목록. 자가수정 루프용. 재QA 통과 시 비움.
```

## 3. 상태 값 정의

**phase.status**
| 값 | 의미 |
|---|---|
| pending | 아직 시작 안 됨 |
| running | 현재 실행 중 |
| done | 정상 완료 |
| failed | 실행 실패 (last_error 필드에 원인) |
| skipped | 사용자가 --skip 지정 또는 해당 단계 불필요 |

**overall_status**
| 값 | 의미 |
|---|---|
| pending | 시작 전 |
| running | 진행 중 |
| done | QA 통과 후 정상 종료 |
| halted | 사용자 중단 또는 max_attempts 초과 |

## 4. 에이전트 동작 규약 (모든 에이전트 공통)

### 4.1 시작 시 (Prerequisite Check)
1. `_state.yaml` 읽기
2. 자기 phase 의 **prerequisite phases** 가 모두 `status: done` 인지 확인
   - analyze 의 선행: preflight
   - design 의 선행: preflight, analyze
   - write 의 선행: preflight, analyze, design
   - trace 의 선행: preflight, analyze, design, write
   - qa 의 선행: preflight, analyze, design, write, trace
3. 선행 미완 시 **즉시 중단** + 사용자에게 보고 (`_state.yaml` 은 수정하지 않음)
4. OK 면 자기 phase 를 `status: running` + `started: <now>` 로 갱신

### 4.2 완료 시 (Done-marker)
`_state.yaml` 의 자기 phase 를 다음과 같이 갱신:
- `status: done`
- `completed: <now>`
- `artifacts:` 생성/수정한 파일 경로 목록
- `metrics:` 에이전트별 정의된 메트릭 (예시: analyze → req_total, design → pol_count, qa → pass_rate)
- `notes:` 1~2 줄 요약
- `updated:` (최상위) 현재 시각으로
- `current_phase:` 다음 phase 로 이동
- `history:` append (`event: phase_completed`)

### 4.3 실패 시
- `status: failed`
- `last_error: <간단한 원인>`
- 부분 생성물은 남겨둠 (재개 시 덮어쓰거나 보완)

### 4.4 재실행 시 (자가수정 모드)
오케스트레이터가 `qa_failures` 를 참조해 담당 에이전트를 재호출:
- 에이전트는 `qa_failures` 중 `assigned_to == <자기이름>` 인 이슈만 처리
- 처리 완료 시 해당 이슈를 `qa_failures` 목록에서 제거
- 자기 phase 는 `status: done` 유지(재실행되었다는 표시는 `history` 에 append)

## 5. 재개 규칙 (build-process 시작 시)

오케스트레이터는 파이프라인 시작 시 다음 순서로 판단:

```
_state.yaml 존재?
├─ NO  → Fresh 시작 (preflight 부터)
└─ YES → 
   ├─ overall_status == "done"    → 사용자에게 "이미 완료. 재실행?" 질문
   ├─ overall_status == "halted"  → "중단됨. 이어서? 새로 시작?" 질문
   └─ overall_status == "running" →
      ├─ current_phase 식별
      ├─ 사용자에게 "{current_phase}부터 재개?" 질문
      └─ 승인 시 해당 phase 부터 실행
```

**플래그**
- `--restart` : 기존 state.yaml 을 `{슬러그}_state_{YYYYMMDD}.yaml` 으로 아카이브 후 Fresh 시작
- `--resume`  : 질문 없이 자동 재개
- `--from <phase>` : 명시 phase 부터 강제 시작 (상위 phase status 덮어쓰기)

## 6. 자가수정(self-correction) 루프

```
Attempt N:
  orchestrator 파이프라인 전구간 실행
    → QA 실행
    → QA report: pass / fail
  
  if pass:
    overall_status = done
    halt
  
  if fail (attempt < max_attempts):
    qa_failures 파싱 → assigned_to 로 그룹화
    for each group:
      해당 agent 를 scoped 모드로 재호출 (해당 이슈만 수정)
    QA 재실행
    attempt += 1
  
  if fail (attempt >= max_attempts):
    overall_status = halted
    사용자에게 잔여 이슈 전체 보고 + 수동 조치 요청
```

## 7. QA Fail 라우팅 표

qa-reviewer 는 모든 Fail 항목에 `assigned_to` 와 `fix_scope` 를 기입한다.

| 부적합 카테고리 | assigned_to | 예 |
|---|---|---|
| frontmatter 누락·타입 오류 | 해당 유형 생성 에이전트 | POL/PRO → process-designer, WI/TMP/EX → wi-tmp-writer |
| 파일명 규칙 위반 | 해당 유형 생성 에이전트 | 동일 |
| 폴더 배치 오류 | 해당 유형 생성 에이전트 | 동일 |
| POL/PRO 구조 누락(§11) | process-designer | RACI·Mermaid·KPI 누락 등 |
| WI 구조 누락(§11) | wi-tmp-writer | 예외처리·완료조건 누락 등 |
| source_citation 누락 | **생성한 에이전트** (로그 추적) | — |
| 저작권 가드 위반(§10) | 해당 유형 생성 에이전트 | 20단어 일치 → 재작성 |
| 추적성 링크 누락·깨짐 | traceability-mapper | MAT 갱신 누락 |
| MAT 내부 이슈 | traceability-mapper | — |
| 기준 문서 위반 (§1~§8) | 해당 산출물 담당 에이전트 | 재작성 |

**예외**: `assigned_to` 가 불명확한 Fail 은 `manual` 로 기록 → 사용자에게 직접 에스컬레이션.

## 8. 메트릭 정의 (Phase별)

| Phase | 메트릭 |
|---|---|
| preflight | input_files, categories, license_unknown_count |
| analyze | req_total, req_mandatory, req_recommended, citation_coverage(0~1) |
| design | pol_count, pro_count, reused_count |
| write | wi_count, tmp_count, ex_count, inherited_from_asis |
| trace | req_covered, req_uncovered, mat_rows_updated |
| qa | total_checks, pass, fail, warn, attempt |

## 9. 구현 참고 (에이전트 구현자용)

- YAML 읽기: `Read` 후 문자열 파싱 (Claude 가 직접 처리)
- YAML 쓰기: `Edit` 로 특정 필드만 수정 (전체 재작성 대신 선택적 대체 권장)
- 동시성: 본 규약은 **직렬 실행** 가정. 병렬 실행 시 lock 없음 (현재 설계는 한 번에 하나의 파이프라인)
