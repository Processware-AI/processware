# 02-0. 표준 문서 전처리 — `/process-ingest`

> ISO PDF, 법규 문서, 가이드라인 등의 원문을 `/process-plan`이 사용할 수 있는  
> **정규화된 요구사항 패키지**로 변환하는 전처리 단계.  
> `/process-plan` 실행 전 선행 필수 단계다.

---

## 왜 전처리가 필요한가

표준 PDF를 AI에게 직접 주면 다음 문제가 생긴다:

| 문제 | 내용 |
|---|---|
| 조항 누락 | 200페이지 표준에서 중요 조항을 빠트릴 수 있음 |
| 구조 불안정 | 표준마다 구조가 달라 조항 경계 인식 오류 |
| 추적성 없음 | "이 요건이 원문 어느 조항에서 왔는가" 확인 불가 |
| 검증 불가 | "전부 추출됐는가?" AI 스스로 보장 못함 |

`/process-ingest`는 이 문제를 8단계 파이프라인으로 해결하고, **사람이 결과를 검토·승인**한 후에만 다음 단계로 넘어간다.

---

## 폴더 역할

```
sources/          ← 원본 파일 배치 (PDF·DOCX·PPTX 등)
    ↓ /process-ingest
inputs/           ← 정규화된 출력 (YAML·MD만, 바이너리 금지)
    ↓ /process-plan
vault/02_적용요건/  ← 최종 적용요건 + 프로세스 문서
```

> `inputs/`에 PDF를 직접 넣으면 `/process-plan`이 즉시 중단된다.  
> 반드시 `sources/`에 넣고 `/process-ingest`를 실행해야 한다.

---

## 기본 사용법

```bash
# 국제표준 PDF
/process-ingest sources/ISO9001_2015.pdf --standard ISO9001 --version 2015

# 국내 법규
/process-ingest sources/개인정보보호법_2024.pdf --standard 개인정보보호법 --version 2024 --category 02

# 기존 조직 문서
/process-ingest sources/현_품질매뉴얼_v3.docx --standard 현_품질매뉴얼 --category 04

# 상태 확인
/process-ingest --status ISO9001

# 완료된 목록 확인
/process-ingest --list
```

---

## 카테고리 분류

`--category` 미지정 시 파일 내용으로 자동 판단한다.

| 카테고리 | 대상 문서 |
|---|---|
| `01` (기본값) | 국제표준 (ISO·IEC·KS·ANSI 등) |
| `02` | 국내외 법령·고시·규정 |
| `03` | 해설서·인증기관 가이드 |
| `04` | 기존 조직 내부 문서 |
| `05` | 업종 프레임워크·체크리스트 |

---

## 실행 흐름

```
Phase 1: Source Intake     — 파일 등록·해시 생성
Phase 2: Text Extraction   — 텍스트 추출 (스캔본 감지 시 즉시 중단)
Phase 3: Structural Parsing — 조항 구조 인식
Phase 4: Requirement Mining — shall/should/may 탐지·분할
Phase 5: Classification    — normative/informative/역할/증적 분류
Phase 6: Traceability      — 요건 ID 부여·원문 위치 매핑
Phase 7: QA Review         — 누락 탐지·품질 점검 → HITL 강제 정지
Phase 8: Handoff           — (confirm 후) inputs/ 패키지 확정
Phase 9: Flow Propose      — [flow-proposer] 업무 시나리오 도출 → HITL 선택 → business_flow.yaml
```

> Phase 9는 Phase 8 완료 후 자동 실행. HITL 선택을 완료해야 `/process-plan` 실행 시 `business_flow.yaml`이 활용된다.

## 스캔본 처리

스캔된 PDF(이미지 기반)는 `--ocr` 플래그로 자동 처리할 수 있다.

### `--ocr` 없는 경우 (기본)

즉시 중단하고 처리 방법을 안내한다.

```
❌ 스캔본 PDF 감지 — 작업 중단

OCR로 자동 처리하려면:
  /process-ingest sources/ISO9001_2015.pdf --standard ISO9001 --ocr

또는 외부 도구로 텍스트 변환 후 재투입:
  1. Adobe Acrobat / ABBYY FineReader 등으로 텍스트 PDF 변환
  2. 변환된 파일을 sources/ 에 배치
  3. /process-ingest 재실행
```

### `--ocr` 있는 경우

`ocrmypdf`(Tesseract 백엔드)를 자동으로 호출하여 처리한다.

**사전 설치 필요**:
```bash
brew install ocrmypdf
```

**사용법**:
```bash
/process-ingest sources/ISO9001_2015.pdf --standard ISO9001 --version 2015 --ocr
```

**내부 동작**:
1. 스캔본 감지 (페이지 평균 100자 미만)
2. `ocrmypdf --force-ocr -l kor+eng` 실행 (한국어·영어 동시 인식)
3. OCR'd PDF에서 텍스트 재추출
4. 이후 파이프라인 동일하게 진행
5. `_state.yaml.ocr_processed: true` 마킹

**주의**: OCR 품질은 스캔 해상도에 따라 달라진다. OCR 후에도 텍스트가 불충분하면 사용자 확인을 요청한다. requirements.yaml 의 QA Review(Phase 7) 에서 반드시 누락 조항을 점검해야 한다.

---

## HITL 검토 (필수)

Phase 7 완료 후 반드시 사람이 결과를 검토해야 한다.

**검토 파일 위치**:
```
inputs/{카테고리}/{standard_id}/qa/review_request.md
```

**검토 내용**:
- 누락 의심 조항 목록
- 모호한 요구사항 목록
- 추출 품질 리포트

**검토 완료 후**:
```bash
/process-ingest --confirm ISO9001
```

> confirm 없이는 `/process-plan`에서 이 표준의 inputs을 사용할 수 없다.

## 출력 파일 구조

```
inputs/01_표준원문/ISO9001/
  ├── structure.yaml       조항 계층 구조
  ├── requirements.yaml    요건 목록 (ID·조항·텍스트·분류·산출물 후보)
  ├── clauses.md           전문 텍스트
  ├── definitions.yaml     용어 정의
  ├── annexes.yaml         부속서
  ├── source_map.yaml      요건 ↔ 원문 페이지·단락 매핑
  └── qa/
      ├── review_request.md         HITL 검토 요청서
      └── extraction_quality_report.md  추출 품질 보고서

inputs/06_목표흐름/
  └── business_flow.yaml   업무 시나리오 목록 (flow-proposer Phase 9 생성)
                           — scenario_groups / scenarios / nodes / edges / mermaid
```

## 표준 개정 처리 (delta 모드)

표준이 업데이트되면 개정사항 문서를 sources/에 등록하고 delta 모드로 실행한다.

```bash
# 개정사항 문서 배치
sources/ISO9001_2024_개정사항.pdf

# delta 모드 실행
/process-ingest sources/ISO9001_2024_개정사항.pdf \
  --standard ISO9001 --mode delta --base-version 2015

# HITL 검토 후 확정
/process-ingest --confirm ISO9001 --mode delta
```

delta 결과는 기존 requirements.yaml에 ADD/MODIFIED/DEPRECATED 상태로 반영된다.

## 복수 표준 → 하나의 적용요건

여러 표준을 ingest한 후 `/process-plan`에서 통합한다.

```bash
# 각각 ingest
/process-ingest sources/ISO9001_2015.pdf --standard ISO9001 --version 2015
/process-ingest sources/ISO27001_2022.pdf --standard ISO27001 --version 2022
/process-ingest sources/개인정보보호법.pdf --standard 개인정보보호법 --category 02

# 통합해서 process-plan 실행
/process-plan "OOO사 통합경영체계"
# → inputs/ 의 모든 표준 요건을 읽어 하나의 적용요건.md로 통합
```

## 플래그 전체 목록

| 플래그 | 설명 |
|---|---|
| `--standard <ID>` | 표준 식별자 (필수) |
| `--version <ver>` | 표준 버전 (권장) |
| `--category <01-05>` | 카테고리 지정 (미지정 시 자동) |
| `--mode delta` | 개정사항 처리 모드 |
| `--base-version <ver>` | delta 기준 버전 |
| `--confirm <standard_id>` | HITL 검토 완료 후 Phase 8 실행 |
| `--status <standard_id>` | 진행 상태 확인 |
| `--list [--category <01-05>]` | 완료된 ingest 목록 |
