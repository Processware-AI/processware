# 04. 프로세스 심사 — `/process-check`

> `/process-do`가 쌓은 실행 기록(REC)과 절차(PRO/WI)의 요건을 AI가 자동으로 대조하여 심사 보고서·부적합(NCR)·KPI 대시보드를 산출한다.

---

## 기본 사용법

```
/process-check start PRO-PPR-101 --auditor "이감사"
/process-check start CMMI-PRJ-PLANNING --auditor "이감사"
```

---

## 독립성 원칙 (ISO §9.2)

**심사자(auditor)와 실행자(executor)는 반드시 달라야 한다.**

같은 사람이 실행하고 심사하면 즉시 abort된다.

```bash
# 본인이 실행한 기록을 본인이 심사 → 오류
/process-check start PRO-PPR-101 --auditor "홍길동"
# ❌ 독립성 위반: trace run-a3f9c2b1 의 executed_by "홍길동"

# PoC·테스트 목적으로만 우회 허용
/process-check start PRO-PPR-101 --auditor "홍길동" --override-independence
```

---

## 전체 흐름

```
Phase 1: audit-planner → 심사 체크리스트 생성
Phase 2: evidence-collector → 증적(REC·trace) 수집
Phase 3: compliance-checker → 요건 ↔ 증적 대조
   ↓ HITL 정지 (심사 매트릭스 확정)
Phase 4 (확정 후):
  ├ audit-reporter → 심사 보고서(REC-AUDIT) 발행
  ├ ncr-drafter → 부적합(REC-NCR) 발행
  └ act-trigger → 차원 4 큐 등록
Phase 5: kpi-collector + kpi-analyzer → KPI 대시보드 갱신
```

---

## 진입 모드

### 1. start 모드 — 신규 심사

```bash
/process-check start PRO-PPR-101 --auditor "이감사"
/process-check start WI-PPR-101-01 --auditor "이감사"        # 단일 WI
/process-check start CMMI-PRJ-PLANNING --auditor "이감사"    # 모듈 전체
/process-check start PRO-PPR-101 --auditor "이감사" --period 2026-01-01..2026-04-30
```

`--period` 기본값: 지난 90일.

### 2. confirm 모드 — 심사 매트릭스 확정

```bash
/process-check --confirm run-a1c2d3e4
/process-check --confirm run-a1c2d3e4 --auditor "이감사"
/process-check --confirm run-a1c2d3e4 --adjust-finding F-002=minor  # 등급 조정
```

AI가 만든 심사 매트릭스를 사람이 검토하고 확정한다. 이때 부적합 등급을 조정할 수 있다.

### 3. close-ncr 모드 — 부적합 종결

```bash
/process-check --close-ncr REC-NCR-PPR-101-2026-001 --capa "REC-CAR-2026-001"
```

시정조치(CAPA) 완료 후 NCR을 종결한다.

### 4. list-ncr 모드 — NCR 목록 조회

```bash
/process-check --list-ncr                   # 전체
/process-check --list-ncr --status open     # 미종결만
/process-check --list-ncr --status closed   # 종결된 것만
```

### 5. kpi 모드 — KPI 측정·대시보드 갱신

```bash
/process-check --kpi start CMMI-PRJ-PLANNING --period 2026-01-01..2026-04-30
```

## 심사 산출물

### 심사 보고서 (REC-AUDIT)

위치: `vault/08_REC_기록/AUDIT/REC-AUDIT-*.md`

포함 내용:
- 심사 범위·기간·심사원
- 요건별 충족/부분/부적합 판정
- Finding 목록과 등급

### 부적합 보고서 (NCR)

위치: `vault/08_REC_기록/AUDIT/REC-NCR-*.md`

NCR 번호 패턴: `REC-NCR-{POL코드}-{PRO코드}-{YYYY}-{NNN}`

### NCR 등급과 SLA

| 등급 | 의미 | 종결 기한 |
|---|---|---|
| **critical** | 심각한 위반 | 발행일 + **20 영업일** |
| major | 중요한 부적합 | 발행일 + 60일 |
| minor | 경미한 부적합 | 발행일 + 90일 |

---

## KPI 대시보드

`vault/90_MAT_통합매핑/MAT-008_KPI_대시보드.md`에 자동 갱신된다.

KPI 판정 4단계:

| 표시 | 판정 | 의미 |
|---|---|---|
| 🟢 | healthy | 목표 충족 + 추세 안정/개선 |
| 🟡 | watch | 목표 충족이지만 추세 나빠짐 |
| 🟠 | recovering | 목표 미달이지만 추세 개선 중 |
| 🔴 | critical | 목표 미달 + 추세 악화 → 차원 4 자동 인계 |
| ⚪ | data_gap | 측정 불가 |

**critical** KPI는 자동으로 차원 4 act queue에 등록된다.

---

## 심사 후 차원 4 인계

심사에서 critical 부적합이나 critical KPI가 발생하면 `act-trigger`가 자동으로 차원 4 큐에 등록한다.

```
심사 완료 → critical finding 3건
  → act_queue: queue-qa1b2c3d4, queue-q9d8c7b6a, queue-qe5f6a7b8
  → /process-act start queue-qa1b2c3d4
```

---

## 옵션 목록

| 옵션 | 효과 |
|---|---|
| `--auditor <이름>` | 심사원 명시 (start 모드 필수) |
| `--period <from..to>` | 심사 기간 지정 (기본: 지난 90일) |
| `--adjust-finding <id>=<grade>` | 부적합 등급 조정 (confirm 시) |
| `--override-independence` | 독립성 검사 우회 (**PoC 전용**) |
| `--dry-run` | 산출물 저장 없이 미리보기 |
| `--strictness strict\|normal\|lenient` | 판정 엄격도 조정 |

---

## 주의사항

1. `/process-check`는 POL/PRO/WI/TMP/EX/REC/MAT-001~005,007 파일을 **절대 수정하지 않는다**.
2. 신규 생성 파일은 `vault/08_REC_기록/AUDIT/`와 `MAT-008`, `MAT-009`만이다.
3. 판정의 근거는 REC에 명시된 값만 사용한다 — AI가 임의로 값을 채우지 않는다.
