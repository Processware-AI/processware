# 06. 역할별 가이드

> 역할에 따라 주로 사용하는 커맨드와 흐름이 다르다.  
> 자신의 역할을 찾아 필요한 부분만 읽어도 충분하다.

---

## 역할 한눈에 보기

| 역할 | 주요 커맨드 | 주요 책임 |
|---|---|---|
| 표준 분석가 | `/process-ingest` | 표준 PDF 전처리·요건 패키지 생성 |
| 프로세스 설계자 | `/process-plan` | 새 프로세스 모듈 빌드·유지 |
| PM (프로젝트 관리자) | `/process-do` (start/approve) | 절차 실행·결과물 승인 |
| QA (품질보증) | `/process-do` (실행) + `/process-check` (심사) | 절차 이행 확인·심사 |
| QMR (품질경영 책임자) | `/process-check --kpi` + `/process-act` | KPI 관리·개정 결정 |
| 심사원 | `/process-check start` + `/process-audit start` | 독립 심사 + 외부 표준 GAP 분석 |
| Process Owner | `/process-act --approve` | PCB 승인·개정 최종 결정 |

---

## 1. 프로세스 설계자

**담당**: 새로운 업무 체계를 빌드하고 유지한다.

표준 PDF가 있으면 먼저 `/process-ingest`로 전처리한 후 `/process-plan`을 실행한다.

### 새 모듈 빌드

```bash
# 1. 필요한 업무 체계를 자연어로 설명
/process-plan "소프트웨어 품질보증 + 형상관리"

# 2. 브랜치 슬러그 확인
# AI 제안: feat/sw-qa-cm-output → 확인 또는 수정

# 3. 표준 PDF 있으면 ingest 먼저 실행 후 진행
# /process-ingest sources/ISO9001_2015.pdf --standard ISO9001 --version 2015
# /process-ingest --confirm ISO9001

# 4. 빌드 완료 후 Obsidian으로 결과 확인
git worktree add ../sw-qa-cm-view feat/sw-qa-cm-output
# Obsidian → ../sw-qa-cm-view/vault 열기
```

### 빌드 중단 후 재개

```bash
/process-plan "소프트웨어 품질보증 + 형상관리" --resume
```

### 특정 단계만 다시 실행

```bash
# WI/TMP/EX만 재작성 (Design은 그대로 유지)
/process-plan "소프트웨어 품질보증 + 형상관리" --from write
```

### 체크리스트

- [ ] 모듈 슬러그 결정 (feat/{슬러그}-output)
- [ ] 입력자료 배치 (없으면 LLM 추정으로 OK)
- [ ] QA ALL PASS 확인
- [ ] origin push 완료
- [ ] Obsidian으로 결과 검토

## 2. PM (프로젝트 관리자)

**담당**: 절차를 실행하고 승인 게이트를 처리한다.

### 절차 실행

```bash
# WI 번호를 알 때
/process-do WI-PPR-101-01

# 업무 이름으로 (자동 매칭)
/process-do "프로젝트 계획 수립"

# 실행자 이름 명시
/process-do WI-PPR-101-01 --executor "홍길동"
```

### HITL 승인 처리

실행 중 승인 게이트에 걸리면 trace_id가 출력된다:
```
⏸ HITL 정지 — trace_id: run-a3f9c2b1
  승인: /process-do --approve run-a3f9c2b1 --approver "홍PM"
  반려: /process-do --reject run-a3f9c2b1 --reason "사유"
```

```bash
# 검토 후 승인
/process-do --approve run-a3f9c2b1 --approver "홍길동"

# 반려 (사유 필수)
/process-do --reject run-a3f9c2b1 --reason "첨부자료 불충분"
```

### 대기 중인 승인 한꺼번에 처리

```bash
/process-do --check-approvals
```

### 체크리스트

- [ ] WI 읽고 내용 이해 후 실행
- [ ] AI 질문에 정확하게 답변 (hallucination 방지)
- [ ] HITL 요청 수신 시 내용 검토 후 승인/반려
- [ ] REC 생성 확인

## 3. QA (품질보증 담당자)

**담당**: 절차 이행을 확인하고 독립 심사를 지원한다.

### 특정 WI 심사 (단독 실행)

```bash
# 주의: 자신이 실행한 기록은 심사할 수 없음
/process-check start WI-PPR-101-01 --auditor "이QA"
```

### 심사 매트릭스 확정

AI가 심사 매트릭스 초안을 만들면 사람이 검토 후 확정:
```bash
/process-check --confirm run-a1c2d3e4 --auditor "이QA"
# 부적합 등급 조정이 필요하면:
/process-check --confirm run-a1c2d3e4 --adjust-finding F-002=minor
```

### NCR 종결 처리

시정조치 완료 확인 후:
```bash
/process-check --close-ncr REC-NCR-PPR-101-2026-001 --capa "REC-CAR-2026-001"
```

### NCR 현황 조회

```bash
/process-check --list-ncr --status open      # 미종결 NCR
/process-check --list-ncr                    # 전체
```

### 체크리스트

- [ ] 독립성 원칙 확인 (내가 실행한 건 심사 안 함)
- [ ] 심사 범위·기간 결정
- [ ] 심사 매트릭스 검토 (AI 판정 신뢰하되 이상한 것 조정)
- [ ] critical NCR → Process Owner에게 즉시 통보
- [ ] SLA 내 NCR 종결 관리

## 4. QMR (품질경영 책임자)

**담당**: KPI를 관찰하고 개정 여부를 결정한다.

### KPI 측정·대시보드 갱신

```bash
/process-check --kpi start CMMI-PRJ-PLANNING --period 2026-01-01..2026-03-31
```

### KPI 대시보드 확인

`vault/90_MAT_통합매핑/MAT-008_KPI_대시보드.md` 열기

🔴 critical KPI → act queue 자동 등록됨

### act queue 현황 조회

```bash
/process-act --list --status pending
```

### 개정 시작 결정

```bash
/process-act start queue-qa1b2c3d4
```

### 체크리스트

- [ ] 분기별 KPI 측정 실행
- [ ] 🔴 critical 항목 즉시 /process-act 트리거
- [ ] 🟡 watch 항목 다음 분기까지 모니터링
- [ ] NCR 종결율·SLA 준수율 확인 (MAT-009)

## 5. 심사원

**담당**: 독립적 입장에서 내부 심사와 외부 표준 GAP 분석을 수행한다.

### 내부 심사 (이행 증적 확인)

```bash
# 반드시 --auditor 지정 (독립성 강제)
/process-check start PRO-PPR-101 --auditor "이감사"

# 심사 후 매트릭스 확정
/process-check --confirm run-a1c2d3e4 --auditor "이감사"
```

### 기간 지정 심사

```bash
/process-check start PRO-PPR-101 --auditor "이감사" --period 2026-01-01..2026-03-31
```

### 심사 엄격도 조정

```bash
# 첫 심사 또는 PoC: 관대하게
/process-check start ... --strictness lenient

# 인증 대비: 엄격하게
/process-check start ... --strictness strict
```

### 외부 표준 GAP 분석 (프로세스 설계 부합성)

```bash
# ISO 9001 대비 GAP 분석
/process-audit start --against ISO9001 --auditor "이감사"

# 초안 검토 후 확정 (N/A 처리 포함)
/process-audit --confirm run-g1a2b3c4 \
  --not-applicable "8.3"="외주 개발 없음"

# 결과 조회
/process-audit --list --against ISO9001
```

> **process-check vs process-audit**: check는 "이행했는가?", audit은 "설계가 표준을 커버하는가?" 를 본다.

### 체크리스트

- [ ] 내부 심사: 심사 대상자가 실행한 기록이 맞는지 확인
- [ ] 내부 심사: 증거(REC·trace)가 충분한지 확인
- [ ] 내부 심사: 부적합 등급 판단 근거 기록
- [ ] GAP 분석: inputs/ 에 표준 요건 파일 있는지 확인
- [ ] GAP 분석: N/A 조항 사유 문서화
- [ ] GAP 분석: critical GAP → Process Owner에게 즉시 통보

## 6. Process Owner

**담당**: PCB 게이트에서 프로세스 자산 개정을 최종 승인한다.

### 개정 승인

```bash
# PCB 승인 요청 수신 후
/process-act --approve run-c1a2b3c4 --approver "김Process-Owner"

# 근본 원인 분석 미흡 → 반려
/process-act --reject run-c1a2b3c4 --reason "5-Why 분석 2단계까지만 진행됨, 추가 분석 필요"
```

### 개정 범위 확인

승인 전 `revision_plan.yaml` 확인:
- 어떤 자산이 개정되는가
- 개정 이유(근본 원인)가 타당한가
- 다른 모듈에 영향은 없는가

### 체크리스트

- [ ] revision_plan.yaml 검토
- [ ] 근본 원인 분석 타당성 확인
- [ ] 영향 자산 범위 적정성 확인
- [ ] 승인 후 /process-plan 재트리거 확인

## 현재 단일 사용자 환경에서

지금은 한 사람이 모든 역할을 수행하는 경우가 많다. 이때:

```bash
# 독립성 검사 우회 (PoC·학습 목적)
/process-check start PRO-PPR-101 --auditor "나" --override-independence

# HITL 자동 승인 (PoC·테스트 목적)
/process-do WI-PPR-101-01 --auto-approve
/process-act start queue-q001 --auto-approve
```

> 실제 운영에서는 `--override-independence`와 `--auto-approve`를 사용하지 않는다.
