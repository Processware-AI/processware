# 02. 프로세스 설계 — `/process-plan`

> 새로운 업무 체계를 Obsidian vault에 자동으로 빌드한다.  
> 표준 코드·PA 그룹·자연어 요건 어떤 형태의 입력도 수용한다.

> **전제**: `/process-plan` 실행 전 표준 PDF 등이 있다면 `/process-ingest`로 먼저 전처리해야 한다.  
> 자연어로 직접 시작할 경우 전처리 없이 바로 실행 가능하다. → [[02-0_표준문서_전처리]]

---

## 기본 사용법

```
/process-plan "프로젝트 계획·추정·리스크 관리"
/process-plan CMMI-PRJ-PLANNING
/process-plan iso9001-quality-management
```

---

## 입력 형태 3가지

| 입력 형태 | 예시 | 브랜치 슬러그 |
|---|---|---|
| 자연어 요건 | `"소프트웨어 품질보증 + 형상관리"` | `feat/sw-qa-cm-output` |
| PA 그룹명 | `CMMI-PRJ-PLANNING` | `feat/cmmi-prj-planning-output` |
| 표준 코드 | `iso9001` (inputs/에 해당 표준 ingest 결과가 있으면 자동 활용) | `feat/iso9001-output` |

> 브랜치 슬러그는 AI가 제안하고, 사용자가 확인 또는 수정한다.

---

## 전체 실행 흐름

```
Phase -2: git 브랜치 생성 (feat/{슬러그}-output)
Phase -1: 이전 실행 상태 확인 (_state.yaml)
Phase  0: 입력자료 확인 (preflight)
Phase  1: Analyze    — 표준·요건 분해
Phase  2: Design     — POL/PRO 설계
Phase  3: Write      — WI/TMP/EX 작성
Phase  4: Trace      — MAT 추적성 갱신
Phase  5: QA         — 구성원칙 준수 검사
→ QA 실패 시 자가수정 루프 (최대 3회)
→ ALL PASS → 완료
```

---

## 단계별 상세

### Phase -2: Git 브랜치 격리

빌드 시작 전 자동으로:
1. main에 표준 정보 등록 (`02_문서번호체계.md`, `07_표준분류레지스트리.md`)
2. `feat/{슬러그}-output` 브랜치 생성
3. 이후 모든 작업은 이 브랜치에서 진행

완료 후 자동으로 브랜치를 origin에 push하고 PR 생성 안내를 출력한다.  
main에 merge는 사용자 결정 사항이다.

### Phase 0: Preflight (입력자료 확인)

`inputs/` 폴더를 스캔한다.

**바이너리 파일 감지 시 즉시 중단**:
```
❌ inputs/ 에 바이너리 파일이 있습니다 — 작업 중단
먼저 /process-ingest 로 전처리 후 재실행하세요.
```

**inputs/ 에 요건 패키지가 있는 경우**:
- `requirements.yaml` 을 자동으로 로드하여 Phase 1 Analyze에 활용
- 여러 표준의 패키지가 있으면 모두 통합

**inputs/ 가 비어 있는 경우**:
- A. sources/ 에 파일을 배치하고 /process-ingest 실행 후 재시작 (권장)
- B. LLM 추정만으로 진행 (첫 시도·교육용에 적합)

### Phase 1: Analyze + 적용요건 정의

`standard-analyzer` 에이전트가:
- inputs/의 requirements.yaml 로드 (있는 경우)
- 없으면 LLM 지식으로 요건 추정
- `vault/02_적용요건/{슬러그}/적용요건.md` 생성
  - 조직 이름 기반 명칭 (예: "OOO사 품질경영체계")
  - 복수 표준 요건 통합 + 출처 조항 매핑 포함
- MAT-002 요구사항 매트릭스 작성

### Phase 2: Design

`process-designer` 에이전트가:
- 3-Tier 프로세스맵 (Management/Core/Support) 매핑
- POL (정책) 1~2종 설계
- PRO (절차) 설계 — RACI·KPI·흐름도 포함
- MAT-003 갱신

### Phase 3: Write

`wi-tmp-writer` 에이전트가:
- 각 PRO의 step별 WI (업무지침) 작성
- WI 대응 TMP (양식) 작성
- 채워진 EX (작성예시) 작성

### Phase 4: Trace

`traceability-mapper` 에이전트가:
- 표준 요구사항 ↔ POL ↔ PRO ↔ WI ↔ TMP 추적성 연결
- MAT-001~006, MAT-011+ (표준별 추적성) 갱신

### Phase 5: QA

`qa-reviewer` 에이전트가:
- 구성원칙 준수 검사 (8종 유형·폴더·파일명·분리 원칙)
- 실패 항목은 자가수정 루프로 자동 재처리

---

## 자가수정 루프

QA 실패 시 AI가 자동으로 실패 항목을 수정하고 QA를 재실행한다.

```
Attempt 1: QA 실패 (3건)
  → 자동 수정 → QA 재실행
Attempt 2: QA 실패 (1건)
  → 자동 수정 → QA 재실행
Attempt 3: QA ALL PASS → 완료
```

수정 불가(`manual`) 항목이 있으면 사용자에게 에스컬레이션한다.

---

## 재개·재시작

진행 상태는 `.claude/states/{슬러그}_state.yaml`에 저장된다.

```bash
# 중단된 빌드 이어서
/process-plan "프로젝트 계획" --resume

# 처음부터 다시
/process-plan "프로젝트 계획" --restart

# 특정 단계부터
/process-plan "프로젝트 계획" --from write
```

---

## 플래그 전체 목록

| 플래그 | 효과 |
|---|---|
| `--restart` | 기존 상태 아카이브 후 처음부터 |
| `--resume` | 현재 phase부터 자동 재개 |
| `--from <phase>` | 지정 phase부터 시작 (preflight/analyze/design/write/trace/qa) |
| `--skip-qa` | QA·자가수정 루프 생략 |
| `--cross` | 교차 표준 통합 분석 포함 |
| `--max-attempts N` | 자가수정 최대 횟수 변경 (기본 3) |
| `--no-branch` | git 브랜치 격리 skip (단일 main 작업 모드) |
| `--commit-per-phase` | 각 phase 종료 시 중간 commit |

## CMMI PA 그룹 빠른 참고

| 그룹 코드 | PA 구성 | 대상 |
|---|---|---|
| `CMMI-ORG-FOUNDATION` | GOV·II·PAD·OT | 조직 기반 (1회 구축) |
| `CMMI-ORG-PERFORMANCE` | MPM | 조직 성과 측정 |
| `CMMI-PRJ-PLANNING` | PLAN·MC·EST·RSK·SAM | 프로젝트 계획·관리 |
| `CMMI-PRJ-ENGINEERING` | RDM·TS·PI·VV·PR | 기술 개발 |
| `CMMI-PRJ-SUPPORT` | CAR·DAR·PCM·PQA | 프로젝트 지원 |

> 전체 CMMI를 한꺼번에 빌드하기보다 그룹 단위로 필요한 것을 조합하는 것이 권장된다.
