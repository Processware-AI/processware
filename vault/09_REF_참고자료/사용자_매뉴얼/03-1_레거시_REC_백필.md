# 03-1. 레거시 REC 백필 — `/process-backfill`

> 기존 조직이 만들어 놓은 문서(Word·Excel·PDF 등)를 표준 REC로 변환한다.  
> `/process-do`가 "앞으로 만들 기록"이라면, `/process-backfill`은 "이미 있는 문서를 기록으로 편입"하는 작업이다.

> **전제**: `/process-plan`으로 WI/TMP가 먼저 빌드되어 있어야 한다.  
> WI가 없으면 매칭 대상이 없어 모든 파일이 ❌ 처리된다.

---

## 언제 쓰는가

| 상황 | 설명 |
|---|---|
| 프로세스 전환 초기 | 표준 도입 전에 자체 양식으로 만든 문서들을 REC로 편입 |
| 감사 준비 | 기존 이행 증적을 표준 REC 형식으로 정비 |
| 시스템 이관 | 다른 도구(Confluence, SharePoint 등)에서 관리하던 기록을 vault로 이전 |

---

## 기본 사용법

```bash
# 폴더 단위 배치 변환
/process-backfill sources/old_docs/

# 단건 변환 (자동 WI 매칭)
/process-backfill sources/sprint_review_2026Q1.docx

# 단건 변환 (수동 WI 지정)
/process-backfill sources/sprint_review_2026Q1.docx --wi WI-CMMI-04-01-03

# 담당자·문서 생성 시점 명시 (권장)
/process-backfill sources/old_docs/ --backfiller "홍길동" --date 2026-03
```

---

## 전체 실행 흐름

```
Phase 1: Scan     — 대상 파일 목록 수집
Phase 2: Extract  — 텍스트 추출 (DOCX/XLSX/PDF/PPTX)
Phase 3: Match    — [backfill-matcher] WI 자동 매칭 + confidence 산출
Phase 4: HITL     — 매핑 결과 테이블 제시 → 사람 확정
          ↓ /process-backfill --confirm run-bXXXX
Phase 5: Map      — TMP 필드에 내용 매핑
Phase 6: Generate — REC 생성 (verdict_type: legacy_evidence)
Phase 7: Index    — MAT-005 §실행기록 등록
```

Phase 3까지는 자동 처리 후 HITL(Phase 4)에서 한 번 정지한다.  
`--confirm`으로 확정해야 REC가 생성된다.

---

## 단계별 상세

### Phase 3: 자동 WI 매칭

`backfill-matcher` 에이전트가 4가지 신호로 WI 후보를 찾는다.

| 신호 | 가중치 | 내용 |
|---|---|---|
| 키워드 오버랩 | 40% | 문서 주요 명사·동사구 ↔ WI keywords |
| 산출물 유형 | 30% | "평가서", "검토표" 등 문서 유형 ↔ WI output_type |
| 제목 유사도 | 20% | 파일명·문서 제목 ↔ WI title |
| 표 헤더 구조 | 10% | 문서 표 헤더 ↔ TMP fields |

**confidence 기준**:

| 범위 | 표시 | 처리 |
|---|---|---|
| ≥ 75% | ✅ | 자동 확정 (HITL에서 수정 가능) |
| 50–74% | ⚠️ | HITL에서 수정 권장 |
| < 50% | ❌ | 자동 제외, REC 미생성 |

### Phase 4: HITL 확정

Phase 3 완료 후 아래 테이블이 출력된다.

```
[Back-fill 매핑 검토] — run-b3f9c2b1
총 4건 | ✅ 자동 1건 | ⚠️ 저신뢰도 2건 | ❌ 매칭 불가 1건

No. 파일                              추천 WI               confidence
────────────────────────────────────────────────────────────────────
1.  sprint_review_2026Q1.docx       WI-CMMI-04-01-03     87%  ✅
2.  peer_review_api.xlsx            WI-CMMI-04-01-05     72%  ⚠️
3.  test_report_2026-03.docx        WI-CMMI-04-02-01     65%  ⚠️
4.  meeting_kick_off.docx           (매칭 불가)           --   ❌

❌ 항목(4번)은 자동 제외됩니다.
⚠️ 항목은 WI 수정 권장. 수정: "2=WI-CMMI-04-01-04" 형식.
그대로 진행: "확정"
```

**응답 방법**:

| 입력 | 효과 |
|---|---|
| `"확정"` | 현재 매핑 그대로 진행 |
| `"2=WI-CMMI-04-01-04"` | 2번 파일의 WI 수정 |
| `"3 제외"` | 3번 파일 제외 (REC 미생성) |
| `"2=WI-CMMI-04-01-04, 3 제외"` | 복합 수정 |
| `"전체 취소"` | 작업 중단 |

수정 후 테이블이 재출력된다. 최종 `"확정"` 입력 시:

```
✅ 매핑 확정 — 3건 진행, 1건 제외
▶ /process-backfill --confirm run-b3f9c2b1
```

실행이 일시 정지(`status: pending_hitl_confirmation`)된다.  
`--confirm` 명령으로 Phase 5부터 재개한다.

## 진입 모드 전체

| 모드 | 명령 | 설명 |
|---|---|---|
| start | `/process-backfill <path>` | 신규 백필 실행 |
| confirm | `/process-backfill --confirm run-bXXXX` | HITL 확정 → REC 생성 |
| resume | `/process-backfill --resume run-bXXXX` | 중단된 trace 재개 |
| status | `/process-backfill --status run-bXXXX` | 진행 상태 조회 |
| list | `/process-backfill --list` | 백필 이력 전체 조회 |

---

## 옵션 전체 목록

| 옵션 | 효과 |
|---|---|
| `--wi <WI번호>` | 수동 WI 지정 (단건 전용, confidence 100% 처리) |
| `--backfiller <이름>` | 백필 담당자 명시 (REC에 기록됨) |
| `--date <YYYY-MM>` | 원본 문서 생성 시점 (미지정 시 오늘 날짜) |
| `--dry-run` | 매핑·미리보기만 실행, REC·MAT-005 저장 생략 |

## 생성되는 REC 특성

백필로 생성된 REC는 일반 REC와 구분된다.

```yaml
# REC frontmatter (일부)
status: backfilled
verdict_type: legacy_evidence
source_doc: sources/sprint_review_2026Q1.docx
source_hash: sha256:abc123...
backfiller: "홍길동"
backfill_date: "2026-03"
backfilled_at: "2026-05-05T14:32:00+09:00"
```

REC 본문 첫머리에 경고 박스가 자동 삽입된다.

```
⚠ 본 기록은 레거시 문서로부터 백필 변환된 기록입니다.
원본 파일: sources/sprint_review_2026Q1.docx
백필 담당자: 홍길동 (2026-03)
verdict_type: legacy_evidence — /process-check 심사 시 partial로 계산됩니다.
```

---

## /process-check 심사 시 처리 방식

백필 REC는 `/process-check` 심사에서 일반 REC와 다르게 처리된다.

| 구분 | 일반 REC | 백필 REC |
|---|---|---|
| verdict | conformant / partial / nonconformant | **legacy_evidence** |
| KPI 계산 | conformant 집계 | **partial 집계** |
| NCR 발행 | 조건 충족 시 발행 | **미발행** |
| 보고서 표시 | 일반 표기 | `⚠️ legacy_evidence` 별도 표시 |
| 집계 | 일반 카운트 | `legacy_evidence_count` 별도 집계 |

**모두 conformant로 처리하고 싶은 경우** (심사원 판단):

```bash
/process-check start PRO-CMMI-04-01 --auditor "이감사" --treat-legacy-as conformant
```

## 지원 파일 형식

| 형식 | 지원 | 비고 |
|---|---|---|
| DOCX / DOC | ✅ | 단락·표 추출 |
| XLSX / XLS | ✅ | 시트별 셀 추출 |
| PPTX / PPT | ✅ | 슬라이드별 텍스트 추출 |
| PDF (텍스트 기반) | ✅ | 텍스트 레이어 추출 |
| PDF (스캔본·이미지 기반) | ❌ | 자동 감지 후 해당 파일 제외 |

---

## 주의사항

- **WI가 없으면 매칭 불가**: 반드시 `/process-plan`으로 WI/TMP를 먼저 빌드해야 한다.
- **백필 REC는 심사에서 partial**: 표준 프로세스를 실제로 이행한 증적이 아니므로 완전 충족으로 인정받지 못한다. 중장기적으로 `/process-do`로 신규 REC를 생성하는 것이 권장된다.
- **원본 파일 보존 권장**: `sources/`에 원본 파일을 보관하면 `source_hash`로 무결성 검증이 가능하다.
