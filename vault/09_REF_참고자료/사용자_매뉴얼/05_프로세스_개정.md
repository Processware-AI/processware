# 05. 프로세스 개정 — `/process-act`

> 심사(Check)에서 발견된 문제를 근본 원인 분석 → 개정 계획 → PCB 승인 → 자동 재빌드까지 처리한다. 4차원 PDCA의 마지막 단계이자 차원 1로 돌아가는 폐쇄 루프.

---

## 기본 사용법

```
/process-act start queue-qa1b2c3d4
```

act queue는 `/process-check`가 자동으로 생성한다. 직접 만들지 않아도 된다.

---

## 전체 흐름

```
차원 3 심사 완료
  → critical 부적합·KPI → act queue 자동 생성
     (예: queue-qa1b2c3d4)

/process-act start queue-qa1b2c3d4
  → rca-analyzer: 5-Why 근본 원인 분석 → root_cause.yaml
  → revision-planner: 개정 범위·영향 자산·명령 결정 → revision_plan.yaml
  → pcb-gatekeeper: PCB 승인 요청 (HITL)
  ↓ 승인 대기

/process-act --approve run-c1a2b3c4
  → act-coordinator: As-Is 입력 파일 작성
  → MAT-001 §개정 이력 갱신
  → act queue status: done
  → 차원 1 재트리거: /process-plan --from write ...
```

---

## 진입 모드

### 1. start 모드 — act queue 처리 시작

```bash
/process-act start queue-qa1b2c3d4
/process-act start queue-qa1b2c3d4 --rca-method both       # 5-Why + Fishbone
/process-act start queue-qa1b2c3d4 --auto-approve           # PCB 자동 승인 (PoC)
```

**다중 큐 일괄 처리** (연관된 여러 부적합을 한 번에):
```bash
/process-act start --batch queue-q001,queue-q002,queue-q003
/process-act start --batch-related queue-qa1b2c3d4          # 연관 큐 자동 펼침
```

### 2. resume 모드 — 재개

```bash
/process-act --resume run-c1a2b3c4
```

### 3. approve 모드 — PCB 승인

```bash
/process-act --approve run-c1a2b3c4
/process-act --approve run-c1a2b3c4 --approver "김프로세스오너"
```

### 4. reject 모드 — PCB 반려

```bash
/process-act --reject run-c1a2b3c4 --reason "근본 원인 분석 미흡 — 재분석 필요"
```

### 5. trigger-rebuild 모드 — 차원 1 재트리거

```bash
/process-act --trigger-rebuild run-c1a2b3c4
/process-act --trigger-rebuild run-c1a2b3c4 --dry-run       # 명령 미리보기만
```

### 6. list 모드 — act queue 목록

```bash
/process-act --list                      # 전체
/process-act --list --status pending     # 대기 중
/process-act --list --status done        # 완료된 것
```

## 근본 원인 분석 (RCA)

`rca-analyzer`가 5-Why 또는 Fishbone 기법으로 분석한다.

```bash
/process-act start queue-q001 --rca-method 5why      # 5-Why (기본)
/process-act start queue-q001 --rca-method fishbone  # Fishbone
/process-act start queue-q001 --rca-method both      # 둘 다
```

분석 결과: `root_cause.yaml`
```yaml
queue_id: queue-qa1b2c3d4
primary_root_cause: "WI-PPR-101-03의 표본 선정 기준이 모호해 담당자마다 다르게 해석"
contributing_factors:
  - "TMP-PPR-101-03-01에 표본 크기 공식 없음"
  - "EX-PPR-101-03-01이 특수 케이스만 예시"
confidence: high
```

---

## PCB (Process Control Board) 승인

표준 자산 개정의 최종 게이트. ISO §9.3 관리검토와 동형.

PCB 승인 요청서(`pcb_approval_request.md`)가 생성되면:
- 내용 검토 후 `--approve` 또는 `--reject`
- 승인 시 act-coordinator가 As-Is 입력 파일 작성 → 차원 1 재트리거

---

## 개정 후 차원 1 재트리거

PCB 승인 완료 후:
```
stdout 출력:
  🔄 차원 1 재트리거 권장 명령:
  /process-plan --from write WI-PPR-101-03 TMP-PPR-101-03-01
```

이 명령을 실행하면 영향받는 자산만 재빌드된다 (전체 재빌드 불필요).

---

## 다중 큐 일괄 처리

관련된 여러 부적합이 같은 원인에서 비롯된 경우, 일괄 처리가 효율적이다.

```bash
/process-act start --batch queue-q001,queue-q002,queue-q003
```

- RCA 통합 분석 → 공통 root cause 한 번에 해결
- 의존성 다이어그램 자동 생성
- PCB 승인 1회로 다수 큐 처리

주의: 의존성 충돌이 있거나 RCA 신뢰도가 낮으면 자동으로 분리 처리 권고.

## act queue 파일 구조

위치: `.claude/queues/process-act/queue-q*.yaml`

```yaml
queue_id: queue-qa1b2c3d4
source_type: ncr          # ncr | kpi | recommendation
source_id: REC-NCR-PPR-101-2026-001
target: WI-PPR-101-03
priority: critical
status: pending           # pending → in_progress → done
dispatched_to: "Process Owner"
```

---

## 옵션 목록

| 옵션 | 효과 |
|---|---|
| `--rca-method 5why\|fishbone\|both` | RCA 기법 선택 (기본: 5why) |
| `--auto-approve` | PCB HITL 자동 승인 (**PoC 전용**) |
| `--approver <이름>` | 승인/반려 응답자 명시 |
| `--reason "..."` | 반려 사유 (reject 모드 필수) |
| `--dry-run` | 재트리거 명령 미리보기만 출력 |
| `--batch <id1,id2,...>` | 다중 큐 일괄 처리 |
| `--batch-related <id>` | 연관 큐 자동 펼침 후 일괄 처리 |

---

## 주의사항

1. `/process-act`는 As-Is 입력 파일과 MAT-001 행만 직접 작성한다.
2. POL/PRO/WI/TMP/EX의 실제 개정은 차원 1(`/process-plan --from write`)의 책임이다.
3. act queue 외 임의 개정 트리거는 금지 — 추적성 손상.

---

## 폐쇄 루프 완성

```
차원 1 Plan → 차원 2 Do → 차원 3 Check → 차원 4 Act
                                                ↓
                              ← ← ← ← ← ← ← 차원 1 재트리거
```

이 루프가 돌 때마다 프로세스가 개선되고, 개선 이력이 MAT-001에 누적된다.
