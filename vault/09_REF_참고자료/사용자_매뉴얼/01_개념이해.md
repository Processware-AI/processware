# 01. 개념 이해

---

## 1. 왜 이 플랫폼인가

표준 프로세스를 만드는 기존 방식의 문제:

| 기존 방식 | 이 플랫폼 |
|---|---|
| ISO/CMMI 표준 전체를 읽고 → 문서 수십 종 수작업 | 필요한 업무 체계를 자연어로 설명 → AI가 문서 자동 생성 |
| 만들고 나면 서랍에 보관, 누가 실행하는지 모름 | 만든 문서를 AI가 직접 실행하며 기록 작성 |
| 심사는 몇 년에 한 번, 사람이 샘플링 | AI 심사원이 모든 기록과 요건을 자동 대조 |
| 문제 발견 → 개선에 수개월 | 심사 결과 → 자동 개정 → 자동 재빌드 |

---

## 2. 5단계 파이프라인 (차원 0~4)

이 플랫폼은 5개의 차원(레이어)으로 동작한다.

```
차원 0: Ingest (/process-ingest) — 표준 문서 전처리·정규화
차원 1: Plan   (/process-plan)   — 프로세스 설계·문서 빌드
차원 2: Do     (/process-do)     — 프로세스 실행·기록
차원 3: Check  (/process-check)  — 프로세스 심사·점검
차원 4: Act    (/process-act)    — 개정·개선·재빌드 트리거
           ↑________________________________|  (폐쇄 루프)
```

차원 0은 표준 PDF 등 원본 문서가 있을 때 실행하는 선행 단계다.  
자연어로 직접 차원 1을 시작할 때는 차원 0을 생략할 수 있다.

각 차원의 산출물이 다음 차원의 입력이 되며, 차원 4의 출력이 차원 1로 피드백되는 **폐쇄 루프**를 형성한다.

---

## 3. 문서 8종 체계

모든 산출물은 8종 유형 중 하나에 속한다.

| 유형 | 영문 | 설명 | 폴더 |
|---|---|---|---|
| **POL** | Policy | 정책 — "무엇을 해야 하는가" (원칙·의무) | `03_POL_정책/` |
| **PRO** | Procedure | 절차 — "누가, 언제, 어떻게" (프로세스 흐름) | `04_PRO_절차/` |
| **WI** | Work Instruction | 업무지침 — 단계별 세부 수행 방법 | `05_WI_업무지침/` |
| **TMP** | Template | 양식 — 빈 칸으로 된 기록 서식 | `06_TMP_템플릿/` |
| **EX** | Example | 작성예시 — 채워진 양식 예시 | `07_EX_작성예시/` |
| **REC** | Record | 기록 — 실제 수행 결과물 (TMP를 채운 것) | `08_REC_기록/` |
| **MAT** | Matrix | 매트릭스 — 추적성·인덱스·대시보드 | `90_MAT_통합매핑/` |
| **REF** | Reference | 참고자료 — 외부 표준·법규·가이드 | `09_REF_참고자료/` |

> **핵심 원칙**: TMP(양식)와 REC(기록)는 절대로 같은 파일이 아니다.  
> TMP는 비어 있는 서식 → REC는 실제 내용이 채워진 결과물.

## 4. 전체 폴더 구조

```
projectroot/
├── sources/                 전처리 소스 파일 (PDF·DOCX 등) — vault 밖
├── inputs/                  ingest 출력 패키지 (YAML·MD만) — vault 밖
│   ├── 01_표준원문/         표준 요건 패키지
│   ├── 02_법규/             법규 요건 패키지
│   ├── 03_해설서/           해설서 요건 패키지
│   ├── 04_AsIs/             기존 조직 자산 패키지
│   └── 05_산업가이드/       산업 가이드 패키지
└── vault/                   Obsidian vault (클라이언트 제공 대상)
    ├── 00_MOC/              목차·인덱스
    ├── 00_공통관리/         문서체계·번호체계·파이프라인 규약
    ├── 01_구성원칙/         표준프로세스 구성원칙 (설계 헌법)
    ├── 02_적용요건/         조직이 정의한 적용요건 (inputs 분석 결과)
    ├── 03_POL_정책/ ~ 07_EX_작성예시/   프로세스 자산
    ├── 08_REC_기록/         실행 기록·심사 보고서·NCR
    ├── 09_REF_참고자료/     이 매뉴얼 포함 참고 자료
    └── 90_MAT_통합매핑/     MAT-001~009 통합 추적성 매트릭스
```

> `sources/`와 `inputs/`는 저작권·기밀 자료를 포함하므로 git에서 제외되며 클라이언트에게 제공하지 않는다.

## 5. Branch-per-Module 아키텍처

```
main (플랫폼)
├── .claude/agents/          AI 에이전트 정의
├── .claude/commands/        5개 커맨드 (/process-ingest/plan/do/check/act)
└── vault/02_적용요건/       적용요건 폴더 (빌드 후 생성)

feat/prj-planning-est-risk-output  (프로젝트 계획 모듈)
├── vault/02_적용요건/...    적용요건 분석 결과
├── vault/03_POL ~ 07_EX     빌드된 프로세스 자산
└── vault/08_REC/            /process-do 실행 기록 누적

feat/iso9001-quality-output  (품질관리 모듈)
└── ...
```

**왜 브랜치로 분리하는가?**
- main은 항상 깨끗 — 하네스(도구)만 있음
- 모듈별로 독립 — 한 모듈을 고쳐도 다른 모듈에 영향 없음
- Obsidian worktree — 여러 모듈을 동시에 열람 가능

## 6. 프로세스 뷔페 개념

**표준 코드가 출발점이 아니다. 실질 업무 요건이 출발점이다.**

```
❌ 기존: "CMMI ML3 인증 받으려면 20개 PA를 모두 구현해야 한다"
           → 필요 없는 것도 전부 만들어야 함

✅ 이제: "우리에게 필요한 것은 프로젝트 계획·추정·품질보증"
           → 필요한 것만 골라서 빌드
           → 나중에 인증이 필요하면: /process-check --against CMMI-ML3 → 갭 분석
           → 부족한 부분만 추가 빌드 → 인증 준비 완료
```

표준(CMMI, ISO 9001, ISO 27001 등)은 **출처 레퍼런스**로 남고, 빌드의 주인공은 **실질 업무 요건**이다.

## 7. 문서 번호 체계

모든 문서는 `{유형}-{영역코드}-{번호}` 형식이다.

예시:
- `POL-QMS-001` — 품질관리 정책 1번
- `PRO-QMS-101` — 품질관리 절차 101번
- `WI-QMS-101-01` — 절차 101의 지침 01번
- `TMP-QMS-101-01-01` — 지침 01의 양식 01번
- `REC-QMS-101-01-01-2026-001` — 2026년 첫 번째 실행 기록

## 8. HITL (Human-In-The-Loop)

AI가 모든 것을 자동으로 처리하지만, **승인이 필요한 단계**에서는 반드시 사람이 개입한다.

HITL 게이트가 발동되는 경우:
- **ingest QA 완료 후** (차원 0) — 추출 결과 검토·승인
- WI에 "승인자" 역할이 명시된 step (차원 2)
- 심사 결과 확정 (차원 3)
- 프로세스 자산 개정 (PCB 승인, 차원 4)

HITL 게이트 흐름:
```
AI 실행 중 → HITL step 도달 → 정지 (pending_approval)
사용자 확인 → /process-do --approve <trace_id>
→ AI 실행 재개 → 완료
```
