# 표준 빌드 워크플로우 가이드 — Worktree 기반

> 본 문서는 `/build-process` 하네스로 새 표준을 편입할 때, **main 격리 정책**과 **branch-per-standard** 정책을 유지하면서 동시에 Obsidian 으로 산출물을 열람·작업하는 방법을 정리한다.

상위 문서: [[표준프로세스_구성원칙]] · [[06_파이프라인_상태규약]] · [[02_문서번호체계]]

---

## 1. 핵심 원칙

| 원칙 | 의미 |
|---|---|
| **main 격리** | main 은 harness + 표준 등록 메타만 누적. 산출물 본체는 main 에 두지 않는다 |
| **branch-per-standard** | 각 표준 산출물은 `feat/{표준코드}-output` branch 에 영구 보존 |
| **worktree 활용** | 같은 머신에서 main 과 표준별 산출물을 동시에 열람·작업하기 위해 worktree 사용 |

---

## 2. 두 가지 워크플로우 방식

| 방식 | 명령 | 디스크 효율 | 추천 상황 |
|---|---|---|---|
| **A. Worktree** | `git worktree add ../ISO9001-view feat/iso9001-output` | .git 공유 (효율적) | 같은 머신에서 동시 작업 — **기본 권장** |
| **B. 별도 Clone** | `git clone <url> ../ISO9001-CMMI` | 완전 독립 (.git 중복) | 다른 머신·팀원·CI 서버 |

같은 PC 에서는 worktree, 다른 PC·CI·팀원은 clone.

---

## 3. 폴더 구조 (권장)

```
~/MyProjects/
├── CMMI-DEV/                   ← main (빌드 본부)
│   ├── .claude/                  Claude Code 하네스
│   ├── vault/                    harness 자체
│   └── ...
├── CMMI-DEV-ML3-view/          ← feat/cmmi-dev-ml3-output (산출물 열람)
├── ISO9001-view/               ← feat/iso9001-output (다음 표준)
├── ISO27001-view/              ← feat/iso27001-output
└── ...
```

| 폴더 | branch | 용도 |
|---|---|---|
| `CMMI-DEV/` | `main` | harness 갱신·표준 등록·`/build-process` 실행 |
| `{표준코드}-view/` | `feat/{표준코드}-output` | 표준별 산출물 Obsidian 열람·운영 |

---

## 4. 새 표준 빌드 절차 (예: ISO 9001)

### Step 1. main 디렉토리에서 빌드 시작

```bash
cd ~/MyProjects/CMMI-DEV
git checkout main
git pull --ff-only
/build-process ISO9001
```

→ `/build-process` skill 이 자동 수행:
1. `vault/00_공통관리/02_문서번호체계.md` + `07_표준분류레지스트리.md` 에 ISO 9001 등록
2. main 에 등록 commit + push
3. `feat/iso9001-output` branch 자동 생성
4. Phase 0 ~ Final 실행 (preflight → analyze → design → write → trace → qa → 자가수정)
5. 종료 시 4 분할 commit + push

### Step 2. 빌드 완료 후 Obsidian 열람용 worktree 추가

```bash
cd ~/MyProjects/CMMI-DEV
git worktree add ../ISO9001-view feat/iso9001-output
```

### Step 3. Obsidian 에서 vault 열기

Obsidian → **File → Open vault** → `~/MyProjects/ISO9001-view/vault/`

기존 vault (CMMI-DEV-ML3-view 등) 와 동시에 열어둘 수 있음.

---

## 5. Worktree 운영 명령

```bash
# 현재 worktree 목록
git worktree list

# 새 worktree 추가 (다음 표준 빌드 후)
git worktree add ../{표준코드}-view feat/{표준코드}-output

# 산출물 worktree 안에서 작업·commit·push
cd ~/MyProjects/{표준코드}-view
# (편집)
git add -A vault/
git commit -m "..."
git push origin feat/{표준코드}-output

# 더 이상 필요 없을 때 제거 (branch 자체는 origin 에 남음)
cd ~/MyProjects/CMMI-DEV
git worktree remove ../{표준코드}-view
```

---

## 6. 제약·주의사항

- **같은 branch 를 두 worktree 에서 동시 열기 불가** — git 이 자동 차단
- **main 디렉토리에서 산출물 직접 작업 비권장** — 산출물 변경은 worktree 또는 `/build-process` 통해서
- **PDF 입력자료는 git LFS 미사용 시 repo 비대화 주의** — 현재는 라이선스 보유 ISACA 자료를 private repo 에 직접 commit 중
- **Force-push 는 worktree 의 feat branch 만** — main 으로의 force-push 절대 금지 (skill 가드 명시)

---

## 7. Clone 방식이 필요한 경우

같은 PC 외 환경:
- **다른 노트북·데스크탑** — 작업 환경 분리
- **다른 팀원 PC** — 기여 환경 격리
- **CI/CD 빌드 서버** — 깨끗한 환경 빌드 후 폐기

```bash
# 다른 머신 또는 팀원 PC
git clone https://github.com/Processware-AI/STD_Process_Builder.git
cd STD_Process_Builder
/build-process ISO9001
```

---

## 8. /build-process skill 과의 정합

본 가이드의 워크플로우는 `.claude/commands/build-process.md` 의 **Phase −2 (Git Branch 격리)** 와 정확히 일치한다:

- −2-1 사전 등록 (main commit) ↔ Step 1
- −2-2 feature branch 생성 ↔ skill 자동
- −2-3 commit 정책 (4 분할) ↔ skill 자동
- −2-4 종료 시 push ↔ skill 자동
- −2-5 안전 가드 (main force-push 금지) ↔ 본 가이드 §6

추가로 본 가이드는 **빌드 후 worktree 추가** 단계 (Step 2~3) 를 명시한다 — Obsidian 작업 흐름.

---

## 9. 한 줄 결론

> **빌드는 main 에서, 열람은 worktree 에서.** 다른 PC 는 clone.

표준이 누적될수록 worktree 가 늘어나며, 각 표준이 독립 영구 보존되는 구조가 자연스럽게 완성된다.
