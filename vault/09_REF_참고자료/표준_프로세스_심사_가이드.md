## 1. 본 하네스가 하는 일

차원 2 (`/do`) 가 누적한 **REC + trace.jsonl + MAT-005 §실행기록** 을 입력으로, AI 심사원이 **PRO/WI 요건 ↔ REC 충족 여부**를 자동 대조하여 **심사 보고서·부적합·KPI(Phase 3)** 를 산출한다.

```
[차원 1]  /build-process ISO9001     → POL/PRO/WI/TMP/EX/MAT/REF (자산 — Plan)
[차원 2]  /do WI-CMMI-04-01-03         → REC + MAT-005 §실행기록    (Do)
[차원 3]  /audit start PRO-CMMI-04-01  → REC-AUDIT + MAT-005 §심사이력 (Check) ← 여기
   ↓ Phase 2 ↓
            /audit + ncr-drafter        → REC-NCR + MAT-009 NCR 관리대장
   ↓ Phase 3 ↓
            /audit + KPI                → MAT-008 KPI 대시보드
[차원 4]  (Phase 4) NCR → 차원 1 재트리거 루프 (Act)
```

## 4. 사용법 (Phase 1)

### 4-1. 7개 진입 모드 (Phase 2: 모두 동작)

| 모드 | 명령 | 용도 |
|---|---|---|
| **start** | `/audit start PRO-CMMI-04-01 --auditor "이감사"` | 신규 심사 — PRO 단위 |
|  | `/audit start WI-CMMI-04-01-03 --auditor "이감사"` | 신규 심사 — 단일 WI (좁은 심사) |
|  | `/audit start CMMI-DEV-ML3 --auditor "이감사"` | 신규 심사 — 표준 전체 |
| **resume** | `/audit --resume run-a1c2d3e4` | 정지된 심사 재개 |
| **confirm** | `/audit --confirm run-a1c2d3e4` | 매트릭스 확정 → 보고서 + NCR 자동 발행 |
|  | `/audit --confirm run-a1c2d3e4 --no-ncr` | 보고서만 발행 (NCR 보류) |
| **reject-finding** | `/audit --reject-finding F-002 --reason "..." --trace run-a1c2d3e4` | 부적합 인정 거부 (오탐 제거) |
| **status** | `/audit --status run-a1c2d3e4` | 현재 진행 단계 확인 |
| **list-ncr** | `/audit --list-ncr [--status open\|closed\|all] [--severity critical] [--overdue]` | NCR 목록 (Phase 2) |
| **close-ncr** | `/audit --close-ncr <ncr_id> --capa <REC>` | 시정조치 종결 (Phase 2) |

### 4-2. 공통 옵션
| 플래그 | 효과 | 적용 모드 |
|---|---|---|
| `--dry-run` | 분석은 하되 REC-AUDIT 미저장·MAT-005 미갱신, 미리보기만 | start, resume, confirm |
| `--auditor "이름"` | 심사원 명시 (start 에서 필수) | start, confirm, reject-finding |
| `--override-independence` | 독립성 위반 무시 (PoC 한정 — Phase 4 에서 제거) | start |
| `--period from..to` | 심사 대상 기간 (REC.executed_at 기준). 기본 지난 90일 | start |
| `--strictness strict\|normal\|lenient` | 부적합 판정 임계 | start |
| `--reason "..."` | 인정 거부 사유 / NCR 종결 사유 | reject-finding (필수), close-ncr |
| `--adjust-finding F-NNN=등급` | 등급 조정 (critical/major/minor) | confirm |
| `--no-ncr` | NCR 자동 발행 보류 (보고서만) | confirm |
| `--capa <REC>` | 시정조치 후속 REC doc_id (필수) | close-ncr |
| `--closed-by <이름>` | 종결 응답자 (미지정 시 시스템 사용자명) | close-ncr |
| `--standard <코드>` | NCR 목록 필터 | list-ncr |
| `--severity <등급>` | NCR 목록 필터 | list-ncr |
| `--overdue` | SLA 기한 경과한 NCR 만 | list-ncr |

### 4-3. 정상 흐름 예시 (PoC: PRO-CMMI-04-01) — Phase 2

```bash
# 1. 신규 심사 시작
/audit start PRO-CMMI-04-01 --auditor "이감사" --period 2026-01-01..2026-04-30
#  → trace_id: run-a1c2d3e4 생성
#  → 독립성 가드 통과 (이감사 ≠ dongseok)
#  → audit_plan.yaml (요건 12건)
#  → evidence.yaml (REC 3건, coverage 40%)
#  → conformity_matrix.yaml (충족 4 · 부분 2 · 부적합 2 · 미평가 4)
#  → confirmation_request.md drop-out
#  → status: pending_confirmation

# 2. 매트릭스 검토 후 확정 (Phase 2: NCR 자동 발행 포함)
/audit --confirm run-a1c2d3e4
#  → audit-reporter 위임
#  → REC-AUDIT-04-01-01-2026-001_프로세스_품질보증_심사보고서.md 발행
#  → MAT-005 §"심사 이력" 1행 append
#  → ncr-drafter 자동 위임 (skip_overridden=true)
#     ├ REC-NCR-04-01-2026-001 (F-001 critical, SLA 2026-05-30)
#     ├ REC-NCR-04-01-2026-002 (F-002 major, SLA 2026-07-01)
#     ├ REC-NCR-04-01-2026-003 (F-003 minor, SLA 2026-07-31)
#     └ REC-NCR-04-01-2026-004 (F-004 critical, SLA 2026-05-30)
#  → MAT-009 §"NCR 발행 현황" 4행 append
#  → REC-AUDIT 의 ncr_refs[] + §4 finding 블록의 NCR 링크 줄 자동 삽입
#  → status: completed

# 3. NCR 목록 확인
/audit --list-ncr                       # open 4건 출력
/audit --list-ncr --severity critical    # critical 만 (NCR-001, NCR-004)
/audit --list-ncr --overdue              # SLA 경과만 (현재 0건 — 2026-05-30 미경과)

# 4. 시정조치 후 NCR 종결
/audit --close-ncr REC-NCR-04-01-2026-003 --capa REC-CMMI-04-01-03-01-2026-002 --closed-by "이팀장"
#  → REC-NCR-...003 frontmatter status: closed + capa_rec / closed_at / closed_by
#  → §7 종결 기록 표 채움
#  → MAT-009 §"발행 현황" 행 제거 → §"종결 현황" 행 추가
#  → SLA 준수 자동 계산 출력
```

```bash
/audit --kpi show CMMI-DEV-ML3                     # 표준 섹션 전체

/audit --kpi check-regressions                     # 모든 표준의 critical+watch 모음

# 4. 다음 분기 측정 (baseline 자동 비교)
/audit --kpi start CMMI-DEV-ML3 --period 2026-04-01..2026-06-30
#  → 회차 2 측정 → kpi-analyzer 가 회차 1 을 baseline 으로 자동 사용
#  → delta 계산 + 회귀 임계 (±5%p default) 비교
#  → verdict 4-tier 갱신 (improved/regressed 신규 부여 가능)
```

### 4-8. KPI Verdict 4-tier

| 표시 | verdict | 조건 | 권고 |
|---|---|---|---|
| 🟢 | healthy | 목표 충족 + 추세 stable/improved | 유지 |
| 🟡 | watch | 목표 충족 + 추세 regressed (≥ ±5%p) | 모니터링 강화 |
| 🟠 | recovering | 목표 미달 + 추세 improved | 시정 진행 중 — 다음 회차 확인 |
| 🔴 | critical | 목표 미달 + 추세 stable/regressed | 차원 4 (Act) 권고 1차 후보 |
| ⚪ | data_gap | 측정 불가 (sample 0 또는 baseline 부재) | 운영 시작 또는 다음 회차 대기 |

### 4-9. 차원 4 인계 큐 흐름 (Phase 4 — act-trigger)

```bash
# 1. confirm + KPI finalize 직후 자동 발행 (default)
/audit --confirm run-a1c2d3e4
#  → audit-reporter
#  → ncr-drafter (NCR 4건)
#  → act-trigger (큐 4 + 권고 통합 1)
#  → MAT-008 §"차원 4 인계" 5행 append

/audit --kpi start CMMI-DEV-ML3 --period 2026-01-01..2026-04-30
#  → kpi-collector + kpi-analyzer
#  → act-trigger from_kpi 모드
#     - NCR 와 연계된 critical KPI 는 NCR 큐에 통합 (KPI-04-01-02 / META-FINDINGS-DENSITY / META-NCR-CLOSURE → queue-qa1b2c3d4)
#     - 연계 없는 critical KPI 만 신규 큐 (META-COVERAGE → queue-q5a6b7c8d)
#  → MAT-008 §"차원 4 인계" 1행 추가

# 2. 큐 조회·관리
/audit --act-queue list                              # status: pending 모두
/audit --act-queue list --priority critical          # critical 만
/audit --act-queue list --kind ncr_capa --priority critical
/audit --act-queue show queue-qa1b2c3d4              # 단일 큐 상세

# 3. 차원 4 dispatch (process_owner 등에게)
/audit --act-queue dispatch queue-qa1b2c3d4 --to "박팀장"
#  → 큐 status: pending → in_progress
#  → dispatched_to / dispatched_at 채움
#  → RBAC: process_owner 또는 qmr 권한 필요

# 4. 차원 4 완료 후 종결
/audit --act-queue done queue-qa1b2c3d4 --capa REC-CMMI-04-01-04-01-2026-003
#  → 큐 status: done
#  → done_at / done_capa_rec 채움
#  → 본 큐가 NCR 연계면 NCR 도 close 권장 (별도 /audit --close-ncr 호출)

# 5. 발행 보류 (필요 시)
/audit --confirm run-a... --no-act-queue            # 큐 미발행, 보고서·NCR 만
/audit --kpi start ... --no-act-queue
```

### 4-10. RBAC 운영 (Phase 4)


```python
def korean_business_days(start: date, n_business_days: int) -> date:
    """KST 영업일 계산. 평일 (월~금) - 한국 공휴일 제외."""
    # 한국 공휴일 14개 (2026 기준): 신정 / 설날 3일 / 삼일절 / 어린이날 / 부처님오신날 /
    #                                현충일 / 광복절 / 추석 3일 / 개천절 / 한글날 / 성탄절
    # 대체공휴일 (일요일 → 월요일) 자동 처리
    holidays_2026 = {
        date(2026, 1, 1),   # 신정
        date(2026, 2, 16),  # 설날 전날
        date(2026, 2, 17),  # 설날
        date(2026, 2, 18),  # 설날 다음날
        date(2026, 3, 1),   # 삼일절 (일요일 — 월요일 대체)
        date(2026, 5, 5),   # 어린이날
        date(2026, 5, 24),  # 부처님오신날
        date(2026, 6, 6),   # 현충일
        date(2026, 8, 15),  # 광복절 (토요일 — 대체 없음)
        date(2026, 9, 24),  # 추석 전날
        date(2026, 9, 25),  # 추석
        date(2026, 9, 26),  # 추석 다음날
        date(2026, 10, 3),  # 개천절 (토요일)
        date(2026, 10, 9),  # 한글날
        date(2026, 12, 25), # 성탄절
    }
    current = start
    days_added = 0
    while days_added < n_business_days:
        current += timedelta(days=1)
        if current.weekday() < 5 and current not in holidays_2026:
            days_added += 1
    return current
```

본 함수는 Phase 4.5 에서 ncr-drafter 와 act-trigger 의 SLA 계산에 정식 도입. 현재 PoC 의 28일 근사값과 비교 검증 필요 (예: 2026-05-02 + 20 영업일 = 2026-05-30 vs 영업일 계산기 결과).
