# 고객 요구사항 도출 (WI-CMMI-03-01-01)

> 상위 절차: [[PRO-CMMI-03-01_요구사항_개발_절차]]

## 1. 업무 목적
이해관계자(고객·사용자·운영자·규제기관·공급자 등)로부터 니즈·기대·제약·인터페이스를 elicit 하고, 이를 우선순위가 부여된 고객 요구사항으로 변환하여 후속 SG2/SG3 활동의 기반을 제공한다.

## 2. 수행 주체
- **주 수행자**: Requirements Engineer
- **검토자**: Chief Engineer, VAL Lead (조기 검증 가능성 확인)
- **승인자**: Project Manager

## 3. 범위
PRO-CMMI-03-01 §5 의 **SP1.1 (Elicit Needs) ~ SP1.2 (Transform Stakeholder Needs into Customer Requirements)** 단계에 적용. SG2 제품 요구사항 수립은 [[WI-CMMI-03-01-02]] 참조.

## 4. 입력 자료 / 산출물
- **Input**
  - 프로젝트 헌장 / 사업 사례 (PRO-CMMI-02-01 산출물)
  - 이해관계자 식별표 (PP SP2.6)
  - 기존 시장 분석·고객 보이스(VOC) 자료
  - 적용 규제·표준 목록
- **Output**
  - [[TMP-CMMI-03-01-01-01_이해관계자_니즈_도출_결과서]] (작성본)
  - [[TMP-CMMI-03-01-01-02_고객_요구사항_명세서]] (작성본 — 우선순위 포함)
  - REQM 인계용 요구사항 베이스라인 후보

## 5. 수행 절차

### 5.1 사전 준비
1. 이해관계자 식별표 확보 — PP SP2.6 산출물 참조. 누락 시 PM 과 협의하여 보완.
2. elicit 기법 선정 — 인터뷰·워크숍·설문·관찰·프로토타입 중 프로젝트 특성에 맞게 2개 이상 조합.
3. elicit 일정·참여자·기록자(서기) 배정.

### 5.2 수행 단계
1. **이해관계자 니즈 elicit (SP1.1)**
   - 모든 라이프사이클 단계(설계·구현·운영·유지보수·폐기)별 니즈 수집.
   - 각 니즈에 출처(이해관계자 ID·세션 일자) 명시.
   - 제약(법규·예산·일정·기술)·인터페이스(외부 시스템·사용자)도 동일 양식으로 수집.
2. **니즈 분류·정제**
   - 중복 병합, 모호 표현 명료화.
   - 분류 축: 기능/비기능/제약/인터페이스/규제.
3. **이해관계자 검증**
   - 정제된 니즈 목록을 원 이해관계자에게 회람하여 확인.
   - 누락·왜곡 보정.
4. **고객 요구사항 변환 (SP1.2)**
   - 각 니즈를 "**누가 / 무엇을 / 어떤 조건에서**" 형식으로 변환.
   - 측정 가능한 수용 기준(Acceptance Criteria) 1개 이상 부여.
5. **우선순위 부여**
   - MoSCoW (Must/Should/Could/Won't) 또는 가중 점수법 적용.
   - 비즈니스 가치·기술 위험·고객 만족도 가중 합산.
6. **고객 요구사항 명세서 작성**
   - [[TMP-CMMI-03-01-01-02]] 양식 작성.
   - 추적 ID 부여(예: CR-001 ~).
7. **고객 확인 결재**
   - 고객(또는 그 대리인) 서명/이메일 확인 첨부.
8. **REQM 인계 준비**
   - 베이스라인 후보 목록을 [[PRO-CMMI-02-03_요구사항_관리_절차]] WI-01 입력으로 송부.

### 5.3 완료 조건 체크리스트
- [ ] 이해관계자 식별표 100% 커버 (모든 식별된 이해관계자로부터 니즈 수집)
- [ ] 모든 라이프사이클 단계별 니즈 수집 (설계~폐기)
- [ ] 각 고객 요구사항에 수용 기준 1개 이상 부여
- [ ] 우선순위 부여 완료 (MoSCoW 또는 점수)
- [ ] 고객(또는 대리인) 확인 결재 첨부
- [ ] 고객 요구사항 명세서 ID 발급 + 추적 ID 부여
- [ ] REQM 인계 송부 기록

## 6. 인터페이스 부서
- **요구사항 엔지니어링팀**: 본 지침 주 수행
- **고객 / 영업**: elicit 세션 참여·회신
- **VAL Lead**: SP3.5 확인 가능성 조기 점검 ([[PRO-CMMI-03-05]])
- **REQM Owner**: 베이스라인 인계 ([[PRO-CMMI-02-03]] WI-01)
- **CM**: 베이스라인 등록 ([[PRO-CMMI-04-01]])

## 7. 주의사항 / 예외 처리

### 7.1 이해관계자 접근 제한
- 핵심 이해관계자가 접근 불가(보안·기밀 등)한 경우:
  - **대체 수단**: 대리인 인터뷰, 과거 산출물 분석, 도메인 전문가 자문.
  - 대체 사용 시 신뢰도 등급(High/Medium/Low) 명시 + Project Manager 결재.

### 7.2 상충하는 니즈
- 이해관계자 간 요구가 상충하는 경우:
  - 충돌 목록을 별도 시트로 분리 → [[WI-CMMI-03-01-04]] SP3.4 균형 분석 입력으로 이관.
  - 본 WI 종료 시점에 미해결 충돌은 "deferred" 상태로 표기.

### 7.3 변경된 이해관계자
- elicit 중 이해관계자가 변경(예: 고객 PM 교체)된 경우:
  - 신규 이해관계자에 재elicit 필수 (이전 결과 자동 승인 금지).
  - 변경 사유·일자·재elicit 결과를 명세서 부록에 기록.
