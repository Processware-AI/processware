# 요구사항 확인 (WI-CMMI-03-01-13)

> 상위 절차: [[PRO-CMMI-03-01_요구사항_개발_및_관리_절차_v1.0]]
> 상위 정책: [[POL-CMMI-03_엔지니어링_정책_v1.0]]

## 1. 업무 목적
확정 직전 요구사항이 **실제 사용자·운영자의 의도된 사용 환경(Intended Operational Environment)** 에서 비즈니스 목표를 충족할지를 시나리오·프로토타입·시뮬레이션·고객 검토 등으로 사전 확인(Validation)하여 후공정의 재작업을 예방한다.

## 2. 수행 주체
- 주 수행자: BA
- 검토자: QA, 운영팀 대표, 아키텍트
- 승인자: PM
- 자문: 핵심 사용자(엔드유저), 고객 SME

## 3. 범위
- 대상: 베이스라인 직전의 모든 이해관계자 요구·고객 요구·솔루션 요구
- 시기: 필요충분성 분석(WI-11) 및 균형 결정(WI-12) 완료 후, 베이스라인 승인 직전
- 제외: 구현 후의 시스템 검증(Verification) 은 PRO-CMMI-03-04 검증·확인 절차의 §5-2 검증 단계 적용

## 4. 입력 자료 / 산출물
- **Input**
  - 베이스라인 후보 요구사항 집합
  - OpsCon 시나리오(WI-CMMI-03-01-08 산출)
  - 필요충분성 분석서(WI-CMMI-03-01-11 산출)
  - 균형 평가서(WI-CMMI-03-01-12 산출)
- **Output**
  - [[TMP-CMMI-03-01-13-01_요구사항_확인_보고서_v1.0]] 작성본
  - 확인 시나리오·프로토타입·고객 검토 의견서
  - 베이스라인 승인 권고서 (PM 결재용)

## 5. 수행 절차

### 5.1 사전 준비
1. 확인 방법(시나리오 워크스루·프로토타입·시뮬레이션·고객 검토 회의·필드 트라이얼) 중 적용 방법을 PM 과 합의한다.
2. 의도된 사용 환경 모델을 정의한다(사용자 페르소나·환경 조건·비즈니스 가정).
3. 확인 참여 사용자·운영자 명단을 확정하고 일정을 통지한다.

### 5.2 수행 단계
1. **확인 시나리오 작성**
   - OpsCon 시나리오를 기반으로 사용자 관점의 확인 시나리오를 도출한다.
   - 정상 흐름·예외 흐름·경계 조건·이상 상황을 빠짐없이 포함한다.
2. **확인 방법 실행**
   - 시나리오 워크스루: 사용자에게 단계별 동작을 설명하고 피드백을 수집한다.
   - 프로토타입 검토: UI 목업·페이퍼 프로토타입으로 사용성·이해도 확인한다.
   - 시뮬레이션: 디지털트윈·도메인 모델로 행동 결과를 사전 시연한다.
   - 고객 검토 회의: 핵심 이해관계자가 모여 시나리오 채택 여부를 결정한다.
3. **결과 수집**
   - 사용자 의견을 수용 가능(Accept) / 조건부(Accept with comment) / 거부(Reject) 로 분류한다.
   - 거부 사유를 구체적으로 기록하고 후속 조치 책임자를 지정한다.
4. **요구사항 갱신**
   - 거부·조건부 의견에 따라 요구사항을 수정·삭제·추가한다.
   - 변경된 요구사항은 양방향 추적성 매트릭스에 즉시 반영한다.
5. **재확인 결정**
   - 변경 규모가 클 경우(영향 요구사항 ≥ 10건 또는 핵심 시나리오 1개 이상) 재확인 회차를 1회 이상 추가 운영한다.
6. **확인 보고서 작성**
   - 확인 방법·참여자·시나리오·결과·갱신 사항을 [[TMP-CMMI-03-01-13-01_요구사항_확인_보고서_v1.0]] 에 정리한다.
7. **베이스라인 승인 권고**
   - 미해결 거부 의견 0건 + 핵심 사용자 서명을 조건으로 PM 에게 베이스라인 승인을 권고한다.

### 5.3 완료 조건 체크리스트
- [ ] 의도된 사용 환경 정의서 첨부
- [ ] 확인 시나리오 도출 완료(정상·예외·경계 포함)
- [ ] 확인 방법 1가지 이상 실행 + 결과 기록
- [ ] 사용자 의견 100% 수집 + 분류
- [ ] 거부 의견 0건 또는 후속 조치 완료
- [ ] 핵심 사용자 서명 수령
- [ ] [[TMP-CMMI-03-01-13-01_요구사항_확인_보고서_v1.0]] 결재 종결
- [ ] 양방향 추적성 매트릭스 최신화

## 6. 인터페이스 부서
- 운영팀: 운영 환경 모델 검증
- 사용자 대표(또는 사용자 패널): 시나리오 워크스루 참여
- 영업·CS: 고객 검토 회의 주관
- 검증팀(QA): 후공정 시스템 검증과의 일관성 검토

## 7. 주의사항 / 예외 처리

### 7.1 사용자 참여 부족
- 확인 회의에 핵심 사용자가 불참한 경우, 영상 인터뷰·서면 검토·대리 검토자 지명 중 하나로 대체한다.
- 대체 검토는 최소 2명 이상의 동료 검토자가 함께 진행하여 단일 의견 의존을 방지한다.

### 7.2 확인 결과 거부 다수 발생
- 거부 의견이 전체 시나리오의 30% 이상이면 요구사항 도출 단계(WI-CMMI-03-01-02)로 회귀한다.
- 회귀 결정 사유와 일정 영향 추정을 PM·CTO 에게 즉시 보고한다.

### 7.3 운영 환경 모델 불확실
- 신규 사업 또는 미경험 도메인이라 운영 환경 모델이 모호한 경우, 가설(assumption) 기반 시뮬레이션을 수행하고 모든 가설을 별도 등록하여 후속 검증 대상으로 추적한다.
- 가설은 PRO-CMMI-02-05 리스크 등록부에 위험 항목으로 추가 등재한다.
