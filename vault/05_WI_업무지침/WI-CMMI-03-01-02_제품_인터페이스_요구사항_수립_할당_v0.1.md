# 제품·인터페이스 요구사항 수립·할당 (WI-CMMI-03-01-02)

> 상위 절차: [[PRO-CMMI-03-01_요구사항_개발_절차]]

## 1. 업무 목적
고객 요구사항을 기술적 언어의 제품·제품컴포넌트 요구사항으로 수립하고, 각 컴포넌트에 할당하며, 컴포넌트 간/외부 시스템과의 인터페이스 요구사항을 식별한다.

## 2. 수행 주체
- **주 수행자**: Requirements Engineer, Architect
- **검토자**: Chief Engineer, TS Lead, PI Lead
- **승인자**: Project Manager

## 3. 범위
PRO-CMMI-03-01 §5 의 **SP2.1 (Establish Product and Product Component Requirements) ~ SP2.2 (Allocate Product Component Requirements) ~ SP2.3 (Identify Interface Requirements)** 단계에 적용. 진입 게이트: [[WI-CMMI-03-01-01]] 완료.

## 4. 입력 자료 / 산출물
- **Input**
  - 고객 요구사항 명세서 ([[WI-CMMI-03-01-01]] 산출물)
  - 초기 아키텍처 가설 (TS 와 협업)
  - 제약 (예산·일정·기술·규제)
- **Output**
  - [[TMP-CMMI-03-01-02-01_제품_컴포넌트_요구사항_명세서]]
  - [[TMP-CMMI-03-01-02-02_인터페이스_요구사항_정의서]]
  - 추적성 매트릭스 (CR → PR → PCR)

## 5. 수행 절차

### 5.1 사전 준비
1. WI-CMMI-03-01-01 산출물(고객 요구사항 명세서) 베이스라인 확정.
2. Architect 와 워크숍 일정 확정 — 컴포넌트 분해 합의.
3. 추적성 도구(예: DOORS, Jama, Excel 매트릭스) 결정.

### 5.2 수행 단계
1. **제품 요구사항 도출 (SP2.1)**
   - 각 CR 을 기술 관점에서 "**~할 수 있어야 한다 / ~를 만족해야 한다**" 형식의 PR 로 변환.
   - 도출 요구사항(derived requirement) — 명시되지 않았지만 기술적으로 필요한 항목 — 별도 표기.
   - 각 PR 에 검증 방법(시험/분석/검사/시연) 부여.
2. **제품컴포넌트 식별·할당 (SP2.2)**
   - 아키텍처 가설로 제품을 컴포넌트로 분해.
   - 각 PR 을 책임 컴포넌트(들)에 할당 — 단일/공유/도출.
   - 할당 충돌(여러 컴포넌트 동시 책임 등) 식별 후 해결.
3. **인터페이스 식별 (SP2.3)**
   - 내부 인터페이스: 컴포넌트 간.
   - 외부 인터페이스: 외부 시스템·사용자·환경.
   - 각 인터페이스에 데이터·프로토콜·SLA·예외 처리 명시.
4. **추적성 매트릭스 수립**
   - CR → PR (제품요구) → PCR (컴포넌트요구) → IR (인터페이스요구) 양방향 추적.
5. **요구사항 충돌·일관성 검토**
   - 충돌(예: 성능 vs 보안)·중복·누락 식별.
   - 미해결은 [[WI-CMMI-03-01-04]] 균형 분석으로 이관.
6. **검토·승인**
   - Chief Eng + TS Lead + PI Lead 합동 검토.
   - PM 승인 후 베이스라인.

### 5.3 완료 조건 체크리스트
- [ ] 모든 CR 에 1개 이상의 PR/PCR 매핑 (커버리지 100%)
- [ ] 도출 요구사항 식별·표기 완료
- [ ] 모든 PR/PCR/IR 에 검증 방법 부여
- [ ] 컴포넌트 할당 충돌 0건 (또는 해결 완료)
- [ ] 인터페이스 정의서에 데이터·프로토콜·SLA·예외 4요소 모두 명시
- [ ] 추적성 매트릭스 양방향 완전
- [ ] PM 승인 결재 첨부

## 6. 인터페이스 부서

- **PI Lead**: 인터페이스 요구 수신 ([[PRO-CMMI-03-03]] SP2.1)

## 7. 주의사항 / 예외 처리

### 7.1 도출 요구사항 (Derived) 과다
- 도출 요구사항이 명시 요구의 30% 초과 시:
  - 누락된 이해관계자 니즈 가능성 → [[WI-CMMI-03-01-01]] 재실시 검토.
  - 또는 과도한 가정 → Architect 와 가설 재정렬.

### 7.2 단일 컴포넌트 제품 (할당 불필요)
- 단일 컴포넌트 제품의 경우 SP2.2 할당은 외부 인터페이스에 한정.
- 미적용 사유를 명세서 부록에 기록.

### 7.3 아키텍처 변경
- 본 WI 진행 중 TS 아키텍처가 변경되면:
  - 변경 영향 평가 후 PCR 할당 재실시.
  - 변경 일자·사유·재할당 결과를 별도 시트에 기록.
