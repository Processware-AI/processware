# WI-ASPICE-01-05-04 — 릴리즈 패키지 빌드 (SPL.2)

## 1. 업무 목적
검증·고객 승인 완료된 베이스라인을 기반으로 OEM 인도용 릴리즈 패키지를 빌드·서명·체크섬 생성하여 변조 방지·추적성·재현성을 확보한다. ASPICE 4.0 SPL.2 활동을 충족한다.

## 2. 수행 주체
- **주 수행자**: Release Engineer
- **검토자**: SW Lead / QA / CM
- **승인자**: Project Manager

## 3. 범위
- 대상: OEM 인도 직전의 모든 SW/HW/문서 산출물
- 활동: 빌드·서명·체크섬·릴리즈 노트·CM 태깅
- 제외: OEM 전달 행위(WI-ASPICE-01-05-05)

## 4. 입력 자료 / 산출물
**입력 자료**
- 검증 완료 베이스라인 (소스/HW/문서)
- 고객 승인서
- 빌드 환경 명세 (Docker/CI/CD)
- 서명 키(HSM 보관)
- 릴리즈 노트 초안

**산출물**
- 릴리즈 패키지 명세서(TMP-ASPICE-01-05-04-01)
- 빌드 산출물 (.bin/.hex/.elf 등)
- SHA-256 체크섬 파일
- 디지털 서명 파일
- CM 베이스라인 tag

## 5. 수행 절차

### 5.1 사전 준비
1. 베이스라인 무결성 검증(이전 단계 산출물 hash 일치).
2. 빌드 환경(Docker 이미지·CI runner) 격리 및 클린 상태 확인.
3. 서명 키 접근 권한·HSM 가용성 확인.
4. 릴리즈 노트 초안과 변경 이력 확정.

### 5.2 수행 단계
1. **클린 빌드 실행** (SPL.2 BP1) — 격리된 환경에서 베이스라인 tag 로 빌드를 수행한다.
2. **산출물 수집·정리** (SPL.2 BP2) — 바이너리·문서·체크섬 파일을 패키지 디렉터리 구조로 정리한다.
3. **체크섬 생성** (SPL.2 BP3) — 모든 파일에 대해 SHA-256 해시를 생성하고 manifest 에 기록한다.
4. **디지털 서명** (SPL.2 BP4) — manifest 에 HSM 키로 서명하고 서명 파일을 첨부한다.
5. **추적성 확인** — 요구사항 ↔ 릴리즈 산출물 매핑이 100% 충족됨을 확인한다.
6. **CM 베이스라인 태깅** (SPL.2 BP5) — 릴리즈 tag(REL-...)을 CM 시스템에 적용한다.
7. **릴리즈 노트 발행** — 변경 내역·결함·잔여 리스크·호환성을 기재.
8. **검토·승인** — 검토자 회람 후 PM 결재.

### 5.3 완료 조건 체크리스트
- [ ] 클린 빌드 환경에서 빌드가 성공적으로 완료되었다.
- [ ] 모든 산출물이 SHA-256 체크섬을 보유한다.
- [ ] manifest 가 HSM 키로 서명되었다.
- [ ] 요구사항 ↔ 산출물 추적성이 100% 충족된다.
- [ ] CM 릴리즈 tag 가 적용되었다.
- [ ] 릴리즈 노트가 발행되었다.
- [ ] 검토·승인 결재가 완료되었다.
- [ ] [[MAT-001_문서관리대장]] 갱신 완료.

## 6. 인터페이스 부서
- 형상관리(CM): 베이스라인 tag·아카이브
- QA: 빌드 무결성·서명 유효성 감사
- SW Lead: 빌드 산출물 기술 검토
- 보안팀: HSM·서명 키 운영
- 법무: 라이선스/수출통제 확인

## 7. 주의사항 / 예외 처리

### 7.1 빌드 비결정성 검출
동일 베이스라인에서 두 차례 빌드 결과 hash 가 다르면 빌드 환경 비결정성으로 보고 원인을 찾을 때까지 릴리즈를 중단한다.

### 7.2 서명 키 접근 거부
HSM 장애·권한 만료 시 보안팀에 즉시 통보하고 키 복구 또는 백업 키 절차를 따른다. 평문 서명 우회 금지.

### 7.3 추적성 누락
요구사항 ↔ 산출물 매핑에 누락이 발견되면 즉시 추적성 매트릭스를 보완하고 누락 사유를 기록한다.

### 7.4 라이선스/수출통제 위반
오픈소스 라이선스 의무(소스 공개·고지) 또는 수출통제 분류(EAR/ITAR) 위반이 확인되면 릴리즈를 중단하고 법무팀 검토를 받는다.
