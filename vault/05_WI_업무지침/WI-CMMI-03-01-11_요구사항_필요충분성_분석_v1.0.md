# 요구사항 필요충분성 분석 (WI-CMMI-03-01-11)

> 상위 절차: [[PRO-CMMI-03-01_요구사항_개발_및_관리_절차_v1.0]]
> 상위 정책: [[POL-CMMI-03_엔지니어링_정책_v1.0]]

## 1. 업무 목적
확정 직전 요구사항 집합이 의도한 솔루션을 구현하기에 **필요(Necessary)** 하고 동시에 **충분(Sufficient)** 한지 정량 분석하여, 누락·중복·모순·과잉 4가지 결함 유형을 베이스라인 이전에 제거한다.

## 2. 수행 주체
- 주 수행자: RDM PO
- 검토자: BA, 아키텍트, QA
- 승인자: PM
- 자문: 도메인 전문가, 운영팀 대표

## 3. 범위
- 대상: PRO-CMMI-03-01 §5-7 솔루션 요구사항이 작성된 시점 ~ §5-13 확인 직전 구간의 요구사항 전체 (이해관계자 요구·고객 요구·솔루션 요구·인터페이스 요구 통합)
- 시기: 베이스라인 직전, 그리고 변경요청 후 베이스라인 재확정 직전
- 제외: 운영 단계에서 발의된 후속 결함은 PRO-CMMI-04-02 형상관리 절차의 변경 통제로 처리

## 4. 입력 자료 / 산출물
- **Input**
  - 요구사항 등록부([[TMP-CMMI-03-01-01-01_요구사항_등록부_v1.0]] 작성본)
  - 양방향 추적성 매트릭스(WI-CMMI-03-01-05 산출)
  - 일관성 점검 결과(WI-CMMI-03-01-06 산출)
  - OpsCon 시나리오(WI-CMMI-03-01-08 산출)
- **Output**
  - [[TMP-CMMI-03-01-11-01_요구사항_필요충분성_분석서_v1.0]] 작성본
  - 결함 시정 요청서(필요 시)
  - 분석 결과 회의록 + 베이스라인 승인 권고서

## 5. 수행 절차

### 5.1 사전 준비
1. 요구사항 등록부의 최신본을 받아 ID·우선순위·상태(승인/검토/보류)별로 정렬한다.
2. 추적성 매트릭스에서 상위(이해관계자 요구) ↔ 하위(솔루션 요구) ↔ 검증(테스트케이스) 연결을 사전 확인한다.
3. 분석 도구(스프레드시트 또는 ALM 쿼리)로 결함 유형 4종 자동 탐지 룰을 점검한다.

### 5.2 수행 단계
1. **필요성(Necessary) 분석**
   - 각 요구사항이 어떤 상위 비즈니스 목표·이해관계자 니즈를 만족시키는지 추적성으로 확인한다.
   - 상위 연결이 없는 요구사항은 "과잉(over-specification)" 후보로 분류한다.
2. **충분성(Sufficient) 분석**
   - OpsCon 시나리오의 모든 사용 사례·예외 흐름이 적어도 하나의 요구사항에 의해 커버되는지 매핑한다.
   - 매핑되지 않은 시나리오 step 은 "누락(missing)" 후보로 등록한다.
3. **중복(Duplication) 분석**
   - 동일 의미를 표현하는 요구사항을 의미 유사도로 묶고, 우선 ID 를 기준으로 통합 권고한다.
4. **모순(Conflict) 분석**
   - 동일 객체에 대해 상반되는 제약·값을 가진 요구사항 쌍을 검출한다.
   - 충돌 검출 시 의사결정 분석(PRO-CMMI-04-04 DAR) 을 트리거한다.
5. **결함 처리 권고 도출**
   - 결함 유형별로 (a) 통합 (b) 추가 도출 (c) 삭제 (d) 표현 정제 중 하나를 권고한다.
   - 권고에는 영향 범위(추적성 상·하 노드 수) 와 변경 비용 추정을 첨부한다.
6. **분석 결과 회의 운영**
   - BA·아키텍트·QA·도메인 전문가가 참여하는 검토 회의를 1회 이상 개최한다.
   - 회의록에 권고 채택 여부와 후속 작업자(owner) 를 기록한다.
7. **베이스라인 승인 권고**
   - 결함 0건 또는 모든 결함이 시정 완료된 경우에만 PM 에게 베이스라인 승인을 권고한다.

### 5.3 완료 조건 체크리스트
- [ ] 분석 대상 요구사항 ID 전수 식별 (목록 첨부)
- [ ] 필요성 미달(상위 연결 없음) 항목 0건 또는 처리 완료
- [ ] 충분성 미달(OpsCon 시나리오 미커버) 항목 0건 또는 추가 도출 완료
- [ ] 중복 항목 통합 또는 사유 기록 완료
- [ ] 모순 항목 DAR 종결 또는 우회 결정 기록
- [ ] 분석 결과 회의록 서명본 보관

## 6. 인터페이스 부서
- 설계팀(아키텍트): 솔루션 요구사항 결함 시정
- 검증팀(QA): OpsCon 시나리오 충분성 보강
- DAR 위원회: 모순 항목 의사결정
- 형상관리(CM): 결함 시정 후 변경요청서 발행

## 7. 주의사항 / 예외 처리

### 7.1 시간 제약으로 정량 분석 불가
- 일정 압박으로 모든 결함 4유형 분석이 어려운 경우, "필요성 + 모순" 두 항목을 우선 처리하고 "중복 + 충분성" 은 다음 베이스라인 직후 14일 이내 보강 분석을 약속한다.
- 보강 약속은 회의록에 기한과 책임자를 명시하여 기록한다.

### 7.2 도메인 전문가 부재
- 의료·금융 등 전문 도메인 요구사항 분석에 사내 전문가가 없는 경우, 외부 컨설턴트 검토를 발주하거나 고객 SME 의 서명 검토를 요청한다.
- 외부 검토 비용은 PRO-CMMI-02-01 프로젝트 계획의 변경 비용으로 반영한다.

### 7.3 결함 시정이 베이스라인 일정을 위협
- 모순·누락이 다수 발견되어 베이스라인 일정 초과가 예상되면 즉시 PM·CTO 에게 보고한다.
- "베이스라인 분할(부분 베이스라인)" 또는 "범위 축소(Scope Cut)" 옵션을 DAR 로 결정한다.
