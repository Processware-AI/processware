# 양방향 추적성·정렬 유지 (WI-CMMI-02-03-03)

> 상위 절차: [[PRO-CMMI-02-03_요구사항_관리_절차]]

## 1. 업무 목적
요구사항 ↔ 하위 산출물 (설계·코드·시험·운영) 의 양방향 추적성 매트릭스(RTM)를 작성·유지하고, 프로젝트 작업과 요구사항의 정렬을 모니터링하여 부적합 발견 시 시정조치한다.

## 2. 수행 주체
- **주 수행자**: Requirements Engineer Lead
- **검토자**: PM, 설계/구현/시험 책임자
- **승인자**: PMO Director (마일스톤 RTM audit 시)

## 3. 범위
PRO-CMMI-02-03 §5 의 **SP1.4(추적성) ~ SP1.5(정렬)**.

## 4. 입력 자료 / 산출물
- **Input**
  - 요구사항 베이스라인 + 변경 이력
  - 하위 산출물: 설계서·코드·시험 케이스·운영 매뉴얼
- **Output**
  - [[TMP-CMMI-02-03-03-01_요구사항_추적성_매트릭스]] (지속 갱신)
  - [[TMP-CMMI-02-03-03-02_정렬_부적합_시정보고]] (부적합 발견 시)

## 5. 수행 절차

### 5.1 사전 준비
1. RTM 도구 (Excel/Jira/DOORS) 확정.
2. 추적 link 카테고리 정의 (Forward/Backward, 강도).

### 5.2 수행 단계
1. **RTM 초기 작성 (SP1.4)** — 요구사항 베이스라인 직후
   - 요구사항 ID × 산출물 ID 매트릭스 작성
   - Forward (Req → 산출물), Backward (산출물 → Req) 양방향 link
2. **지속 유지 (SP1.4)** — 산출물 변경 시
   - 산출물 신설/변경 시 link 추가/갱신
   - 변경 관리 ([[WI-CMMI-02-03-02]]) 반영
3. **정렬 모니터링 (SP1.5)** — 마일스톤별
   - RTM 완전성 (모든 요구사항이 link 보유) 점검
   - 정렬 부적합 (산출물에 미대응 요구사항, 또는 미요구 산출물) 식별
4. **시정조치**
   - 부적합 발견 시 [[TMP-CMMI-02-03-03-02]] 작성
   - PMC SP2 (시정조치) 큐 등록 ([[WI-CMMI-02-02-03]])

### 5.3 완료 조건
- [ ] RTM 작성 완료 (모든 요구사항 link)
- [ ] 산출물 변경 시 RTM 동기 갱신
- [ ] 마일스톤별 RTM 완전성 100%
- [ ] 정렬 부적합 0건 또는 시정조치 등록
- [ ] PPQA audit 통과 (마일스톤 검토 시 RTM 포함)

## 6. 인터페이스 부서
- **설계/구현/시험 책임자**: 산출물 link 정보 제공
- **CM**: RTM 베이스라인 등록 ([[PRO-CMMI-04-01]])
- **PPQA**: RTM 감사 ([[PRO-CMMI-04-03]])
- **PMC**: 마일스톤 검토 시 RTM 입력

## 7. 주의사항 / 예외 처리

### 7.1 RTM 미관리 (자동화 부재)
- 도구 부재 시 최소 Excel + GitHub Issues link.
- RTM 갱신 지연 ≥ 2주 시 PM escalation.

### 7.2 미대응 요구사항 발견 (산출물 없음)
- 마일스톤 검토 전 발견: 시정 (산출물 추가).
- 마일스톤 후 발견: PMC SP2 시정 + 마일스톤 재평가.

### 7.3 미요구 산출물 (overproduction)
- 요구사항 없는 산출물: 본 베이스라인에서 제거 또는 backlog 등록.
- 빈번 발견 (>3건/마일스톤) 시 RD 절차 강화.
