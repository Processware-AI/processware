# 요구사항 이해·약정 (WI-CMMI-02-03-01)

> 상위 절차: [[PRO-CMMI-02-03_요구사항_관리_절차]]

## 1. 업무 목적
RD 가 산출한 요구사항을 받아 평가하고, 수락 기준 통과 시 프로젝트 베이스라인으로 약정을 획득한다.

## 2. 수행 주체
- **주 수행자**: Requirements Engineer Lead
- **검토자**: Project Manager, 고객, 도메인 엔지니어
- **승인자**: PMO Director

## 3. 범위
PRO-CMMI-02-03 §5 의 **SP1.1(이해) ~ SP1.2(약정)**.

## 4. 입력 자료 / 산출물
- **Input**
  - RD 산출 요구사항 명세서 ([[PRO-CMMI-03-01]] 산출물)
  - 고객 요구·시장 분석 자료
  - 수락 기준 표준 (조직 표준)
- **Output**
  - [[TMP-CMMI-02-03-01-01_요구사항_평가_수락_기준]]
  - [[TMP-CMMI-02-03-01-02_요구사항_약정_기록]]

## 5. 수행 절차

### 5.1 사전 준비
1. RD 인계 패키지 (명세서·근거) 확보.
2. Requirement Provider 명단 확인 (고객/내부/표준 등).
3. 수락 기준 체크리스트 준비.

### 5.2 수행 단계
1. **요구사항 이해 (SP1.1)**
   - 각 요구사항을 read-through + Provider 와 Q&A
   - 모호한 표현 (모호어, 검증 불가) 식별
   - 수락 기준 평가: 완전성·검증성·일관성·실현가능성·추적성
   - 미충족 시 Provider 와 재협의
2. **수락 결정**
   - 모든 기준 통과 = 수락 / 일부 미달 = 수정 요청 / 다수 미달 = 반려
3. **약정 획득 (SP1.2)**
   - 수락된 요구사항 집합을 베이스라인 후보로 확정
   - PM·고객·도메인 엔지니어 약정 서명
   - [[TMP-CMMI-02-03-01-02]] 작성

### 5.3 완료 조건
- [ ] 모든 요구사항 평가 완료 — 수락/수정/반려 결정
- [ ] 수정 요청은 Provider 재협의 후 재평가
- [ ] 수락 요구사항 약정 서명 완료
- [ ] CM 베이스라인 등록 ([[PRO-CMMI-04-01]])
- [ ] MAT-001 등록

## 6. 인터페이스 부서
- **RD (Requirements Development)**: 요구사항 공급 ([[PRO-CMMI-03-01]])
- **고객/Stakeholder**: 약정자
- **CM**: 베이스라인 등록
- **PMC**: 변경 모니터링 ([[WI-CMMI-02-02-01]])

## 7. 주의사항 / 예외 처리

### 7.1 모호한 요구사항 (애매어 발견)
- "사용자 친화적", "충분히 빠르게" 등 정량화 불가 표현은 자동 미달 처리.
- Provider 와 정량·측정 가능 표현으로 재작성.

### 7.2 Provider 재협의 거부
- 1회 거부 시 PM escalation.
- 2회 거부 시 sponsor escalation + 해당 요구사항 일시 보류.

### 7.3 베이스라인 후 즉시 변경 요청
- 본 WI 종료 후 1주 내 변경 요청은 [[WI-CMMI-02-03-02]] 변경 관리로 분기.
