# 요구사항 변경 관리 (WI-CMMI-02-03-02)

> 상위 절차: [[PRO-CMMI-02-03_요구사항_관리_절차]]

## 1. 업무 목적
요구사항 베이스라인 후 변경요청을 접수하고, 영향평가를 거쳐 승인·반려를 결정하고, 승인된 변경을 베이스라인에 반영하여 통제된 변경 관리를 보장한다.

## 2. 수행 주체
- **주 수행자**: Requirements Engineer Lead
- **검토자**: PM, CCB 위원 (Tech Lead, PO, CISO 등)
- **승인자**: CCB Chair (PMO Director)

## 3. 범위
PRO-CMMI-02-03 §5 의 **SP1.3 (변경 관리)**.

## 4. 입력 자료 / 산출물
- **Input**
  - 변경요청 (고객·내부·외부 표준 변경 등)
  - 기존 요구사항 베이스라인 + RTM
  - CM 변경 통제 정책 ([[PRO-CMMI-04-01]])
- **Output**
  - [[TMP-CMMI-02-03-02-01_변경요청서]]
  - [[TMP-CMMI-02-03-02-02_변경영향_평가보고서]]
  - 갱신된 요구사항 베이스라인 (CM 등록)

## 5. 수행 절차

### 5.1 사전 준비
1. 변경요청 양식·CCB 위원 명단 확인.
2. RTM 현행화 (가장 최근 상태).

### 5.2 수행 단계
1. **변경요청 접수**
   - 요청자, 변경 내용, 사유, 우선순위 기록
   - 요청 ID 부여 + [[TMP-CMMI-02-03-02-01]] 작성
2. **영향 평가**
   - 영향 영역: 일정·예산·범위·리스크·인접 요구사항·하위 산출물
   - [[TMP-CMMI-02-03-02-02]] 작성
   - 영향 정량화 (effort PM, 비용, 일정 영향일)
3. **CCB 심사**
   - 정기 CCB 회의 (주 1회 권장) 또는 긴급 ad-hoc
   - 심사 결과: 승인 / 조건부 승인 / 보류 / 반려
4. **변경 반영**
   - 승인 시 요구사항 본문 수정 + RTM 업데이트
   - CM 베이스라인 새 버전 등록 (minor/major)
   - Provider 및 다운스트림 PA (RD/TS/PI) 통보
5. **반려·보류 피드백**
   - 요청자에 결정 사유 송부

### 5.3 완료 조건
- [ ] 모든 변경요청 ID 부여 + 추적 가능
- [ ] 영향 평가 완료 (5 영역 모두)
- [ ] CCB 심사 결과 기록 (승인/조건부/보류/반려)
- [ ] 승인된 변경 베이스라인 갱신 + 통보
- [ ] 반려 결정 시 요청자 피드백
- [ ] 변경 리드타임 ≤ 5 영업일 (긴급 외)

## 6. 인터페이스 부서
- **CCB 위원**: Tech Lead, PO, CISO, CM, PM
- **CM**: 베이스라인 새 버전 등록 ([[PRO-CMMI-04-01]])
- **다운스트림 (RD/TS/PI)**: 변경 통보·재조정
- **PMC**: 변경 status 모니터링

## 7. 주의사항 / 예외 처리

### 7.1 긴급 변경 (Emergency Change)
- 운영 정지·법적 의무 등 긴급 변경은 CCB Chair 즉시 결재.
- 24h 내 사후 CCB 심사 의무.

### 7.2 변경 폭주 (분기 ≥ 10건)
- 베이스라인 안정성 문제 시그널.
- 분기 ≥ 10건 시 베이스라인 재수립 검토 (PP 재진입).

### 7.3 영향 평가 의견 불일치
- CCB 위원 간 평가 의견 불일치 시 외부 전문가 의견 청취.
- 합의 도달 못한 사항은 PMO Director 최종 결정.
