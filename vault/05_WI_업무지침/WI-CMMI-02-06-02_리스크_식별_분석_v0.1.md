# 리스크 식별·분석 (WI-CMMI-02-06-02)

> 상위 절차: [[PRO-CMMI-02-06_리스크_관리_절차]]

## 1. 업무 목적
프로젝트의 잠재 리스크를 체계적으로 식별·등록하고, WI-01 의 파라미터로 평가·분류·우선순위화한다.

## 2. 수행 주체
- **주 수행자**: Risk Manager
- **검토자**: PM, 도메인 엔지니어
- **승인자**: Risk Manager (분석 결과), PMO (High/Critical 시 escalation)

## 3. 범위
PRO-CMMI-02-06 §5 의 **SG2 SP2.1~2.2**.

## 4. 입력 자료 / 산출물
- **Input**: 전략서, PP 식별 리스크 (WI-CMMI-02-01-02), 협업 회의·이슈
- **Output**: [[TMP-CMMI-02-06-02-01]], [[TMP-CMMI-02-06-02-02]]

## 5. 수행 절차

### 5.1 사전 준비
1. 전략서·카테고리·파라미터 확인.

### 5.2 수행 단계
1. **리스크 식별 (SP2.1)** — 정기 brainstorming + ad-hoc 입력 (PP·PMC·IPM 회의록 검토).
2. **평가·분류 (SP2.2)** — 확률·영향 평가 → 카테고리 분류 → 우선순위 (Critical/High/Medium/Low).
3. **임계치 초과 alert** — High/Critical 은 즉시 PMO 통보.

### 5.3 완료 조건
- [ ] 등록부 업데이트 (정기 + ad-hoc)
- [ ] 모든 신규 리스크 평가·우선순위 부여
- [ ] High/Critical 24h 내 PMO 통보
- [ ] 우선순위별 통계 보고서

## 6. 인터페이스 부서
- **PM**: 리스크 status 공유 (PMC 입력)
- **WI-03**: 완화 계획 입력
- **OPF**: 시스템적 리스크 패턴 발견 시 ([[PRO-CMMI-01-02]])

## 7. 주의사항 / 예외 처리

### 7.1 리스크 식별 누락 빈발
- 외부 audit 또는 사후 발견 빈번 (≥3건/마일스톤) 시: brainstorming 빈도 증가 + 체크리스트 보강.

### 7.2 우선순위 산정 분쟁
- Risk Manager 와 PM 의견 차이 시 PMO 결정 요청.

### 7.3 같은 리스크 중복 등록
- 카테고리·키워드 자동 중복 점검. 중복 발견 시 통합 (오래된 ID 폐기).
