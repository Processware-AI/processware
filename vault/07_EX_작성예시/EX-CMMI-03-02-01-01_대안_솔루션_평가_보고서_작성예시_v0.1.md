## 2. 대안 후보 (샘플)
| 대안ID | 유형 | 설명 | 주요 가정 |
|---|---|---|---|
| ALT-A | Make | React + 자체 WebSocket 서버 | 프론트팀 React 숙련 |
| ALT-B | Buy | 상용 BI (Power BI Embedded) | 라이선스 비용 수용 |
| ALT-C | Reuse | 기존 사내 대시보드 프레임워크 v3 | 5초 새로고침 가능 검증 필요 |
| ALT-D | Hybrid | 사내 프레임워크 + React 부분 확장 | 인터페이스 안정성 확인 |

## 3. 선정 기준 (샘플)
| 기준ID | 이름 | 가중치(%) | 측정 |
|---|---|---|---|
| K-01 | 기능 충족 | 20 | 요구 매칭율 |
| K-02 | 성능 (p95 ≤ 1.5s) | 20 | 부하 시험 |
| K-03 | CAPEX | 10 | 견적 |
| K-04 | OPEX (3년) | 10 | TCO 모델 |

| K-07 | 리스크 | 10 | 리스크 점수 |
| K-08 | 확장성 | 5 | 아키텍처 리뷰 |
| **합계** |  | **100** |  |

## 6. 민감도 분석 (샘플)
| 항목 | 결과 |
|---|---|
| 분석 시나리오 | K-05(일정) 가중치 15→20%로 변동 |

| 추가 검증 | 일정 추정의 신뢰도(±2주)를 PoC 1주로 검증 |

## 작성 시 유의사항
- 가중치 합 정확히 100% — 99/101 등 오차 시 자동 차단.
- 점수 근거 한 줄 의무 — "감각으로 9점" 금지.
- 민감도 분석에서 순위 변동 발생 시 추가 검증 항목 명시.

## 잘못된 작성 사례
> ❌ 대안 후보 2개로 진행
> ✅ 3개 이상 — 단일 대안만 가능 시 §7.1 절차

> ❌ 가중치 합 95%
> ✅ 합 100% (10진 소수까지 일치)
