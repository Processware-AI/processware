## 3. 명명 규칙 정의
| 항목 | 규칙 |
|---|---|
| CI-ID 패턴 | `CI-{ProjectCode}-{Category}-{Sequence}` (예: CI-ABC-SRC-001) |
| 버전 표기 패턴 | SemVer `vMAJOR.MINOR.PATCH` (예: v1.0.0). HW 는 `HW-v01`. |
| 브랜치 명명 규칙 | `main`(릴리즈), `develop`(통합), `feature/ABC-ADAS-{TicketID}-{slug}`, `release/vX.Y`, `hotfix/vX.Y.Z` |
| 태그 명명 규칙 | `vX.Y.Z` (릴리즈 태그) / `BL-{Milestone}-vX.Y` (베이스라인 태그) |
| 예외 사항 | 외부 라이브러리(EXT) 는 upstream 태그 그대로 사용, 접두사 없음 |

## 작성 요령
- **CI-ID 는 변경 불가**: 한 번 부여한 CI-ID 는 폐기되어도 재사용 금지(추적성 단절 방지).
- **저장소 URL 은 절대경로**: 상대경로·로컬 경로 금지. 누구나 클릭하여 접근 가능해야 함.
- **외부 의존성(EXT) 도 CI**: 누락 시 보안 취약점 추적이 불가능. SBOM 자동화 도구와 연동 권장.
- **도구 자체도 CI**: 빌드 환경 변경이 결과 재현성을 깨뜨리므로 Docker 이미지 또는 Toolchain 동결 필수.

## 잘못된 사례
- CI-ID 없이 단순 파일명만 기재 → 동명 파일이 여러 폴더에 존재 시 추적 불가.
- 명명 규칙을 정의하지 않고 자유 명명 → 자동화 스크립트·검색 효율 저하.
- 외부 라이브러리를 CI 로 등록하지 않음 → 보안 패치 시 영향 범위 파악 불가.
- 저장소 URL 에 사내망 경로(`\\fileserver\proj\...`) 만 적음 → 외부 협력사 접근 불가.
