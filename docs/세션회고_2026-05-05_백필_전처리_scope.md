# 세션 회고 — 2026-05-05 (4)

## 1. 출발점

이전 세션에서 구현한 `/process-backfill`의 Phase 2가 추출 결과를 평문 `.txt`로 저장하고 있다는 문제 제기로 시작했다.

## 2. 핵심 논의 흐름

### 2-1. ingest 전처리 부재 문제

"백필 할 때도 ingest 단계에서의 전처리 과정이 필요할 거 같은데."

현재 문제:
- Phase 2에서 DOCX XML 파싱으로 구조를 추출하지만, `.txt`로 저장하면서 표·섹션·헤딩 구조가 모두 사라짐
- `backfill-matcher` Signal 4(표 헤더 10%)가 평문에서 헤더를 추론해야 함 → 신뢰도 낮음
- Phase 5(Map) TMP 필드 매핑도 구조 없는 텍스트에서 LLM이 알아서 추론

→ **결론**: 추출 결과를 구조 보존 형식으로 저장해야 한다.

### 2-2. 형식 선택 — JSON → YAML → MD

처음 JSON을 제안했으나 "왜 JSON이지?" 지적.
→ 프로젝트 전체가 YAML 기반 → YAML이 맞음.

이어서 "MD 파일로 만들어서 Obsidian vault 구조로 만들면 어떤가?" 제안.

핵심 구분:
- **이 프로젝트의 vault**: processware 시스템 자산 (영구 자산, 오염 금지)
- **sources/legacy/**: 레거시 문서 추출 결과를 담는 **별도 Obsidian vault**

목적: 프로세스 관리자·산출물 작성자가 Obsidian에서 추출 내용을 검토·수정할 수 있도록.

**결론**: `sources/legacy/{경로}.md` — Obsidian 호환 MD, DOCX 표 구조는 MD 표로 보존.

### 2-3. HITL 단일화 확인

HITL 구조 변경 여부:
- Phase 4(WI 매핑 확정) + Obsidian 검토를 **두 개의 게이트**로 분리할 것인가?
- → **하나로 통합**. 매핑 확정 → Obsidian 검토·수정 (선택) → `--confirm`

### 2-4. 디렉토리 구조 보존

"여러 폴더와 서브디렉토리 구조로 되어 있다면? legacy vault 안에 그 구조 그대로 유지했으면."

현재 설계는 파일명만 사용하는 플랫 저장 → **파일명 충돌** 위험.

**결론**: 입력 루트 기준 상대경로 보존.
```
sources/old_docs/2024/Q1/sprint.docx
  → sources/legacy/2024/Q1/sprint.md
```
`backfill-matcher`도 `*.md` → `**/*.md` 재귀 glob으로 전환.

### 2-5. 멀티 legacy → 단일 REC 가능성

"여러 개의 legacy source가 하나의 REC에 담겨서 그 안에서 필터링할 수 있는가?"

두 케이스 구분:
- **케이스 A**: 여러 파일이 동일 WI 실행 1건 구성 (회의록+평가표+승인서) → 원래 1개 REC여야 함. 현재 설계는 미지원.
- **케이스 B**: 독립 실행 여러 건 → MAT-005가 집계·필터 역할. 현재 설계로 대응 가능.

→ 케이스 A(그룹핑)는 이번 세션 미구현. 추후 과제.

### 2-6. 프로젝트 구분 문제

"5개 프로젝트 산출물을 backfill하면 모든 스프린트 리뷰가 개별 REC로 생성되는가? 프로젝트 구분은 어떻게?"

→ 현재 REC 데이터 모델에 프로젝트 컨텍스트 없음. `--project` 플래그 필요.

이어서: "전사 공통 업무나 지원 업무는 어떻게 하나?"

→ `--project`는 너무 좁음. **`--scope`** 로 일반화:
- `--scope "프로젝트-A"`, `--scope "HR팀"`, `--scope "전사-품질"` 모두 동일 플래그

### 2-7. process-plan 자산에도 scope 필요 (핵심 결정)

"process-plan의 결과물인 표준 프로세스 자산에도 scope 개념이 필요하지 않나?"

**결론**: 맞다. `pro_type: core|support`는 가치사슬 역할이고, `scope_type`은 적용 범위 — 다른 차원.

scope_type 3종:
- `project`: 특정 프로젝트·납품 맥락 (스프린트 리뷰, 테스트 등)
- `org`: 조직 단위 관리 기능 (HR, 구매, IT지원 등)
- `common`: 전사 거버넌스·정책 준수 (리스크 관리, 경영검토 등)

**전파 경로**:
```
business_flow.yaml 시나리오 (scope_type)
  → process-designer: PRO frontmatter 상속
  → wi-tmp-writer: WI frontmatter 상속 + steps.yaml 메타 동기화
  → backfill-matcher: WI scope_type 로드 + scope 정합성 보정
```

`backfill-matcher` scope 보정:
- project ↔ org 불일치: **-10 페널티**
- `common` WI: 보정 없음 (전사 공통은 항상 적용 가능)
- scope 미확인: 보정 없음

## 5. 핵심 결정 요약

**"평문 .txt는 구조를 버린다"**  
DOCX XML 파싱으로 추출한 표·섹션을 MD로 보존. backfill-matcher Signal 4(표 헤더)와 Phase 5 TMP 필드 매핑 신뢰도 향상.

**"sources/legacy/는 processware vault가 아니다"**  
임시 아티팩트가 아니라 사람이 Obsidian에서 편집하는 별도 공간. 편집 내용이 Phase 5에 그대로 반영됨.

**"--project는 너무 좁다 — --scope로 일반화"**  
프로젝트뿐 아니라 기능 조직·전사 공통 모두 동일 플래그로 처리. 폴더명 자동 추론이 기본.

**"scope_type은 pro_type과 다른 차원이다"**  
`pro_type`은 가치사슬 역할(핵심/지원), `scope_type`은 적용 조직 범위(프로젝트/조직/전사). business_flow에서 PRO → WI까지 전파되며 backfill 매칭에도 반영.

**"common WI는 scope 페널티 없음"**  
전사 공통 프로세스는 어느 scope의 레거시 문서와도 매칭 가능하므로 보정 불필요.
