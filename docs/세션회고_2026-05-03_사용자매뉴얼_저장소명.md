## 2. 저장소 이름 변경 결정

### 배경
`STD_Process_Builder`라는 이름이 두 가지 오해를 유발:
1. 표준 코드(ISO/CMMI)가 입력이어야 한다는 인상
2. Plan(빌드)만 하는 도구처럼 보임

### 후보 검토 과정

**1차 후보** (process- prefix 유지):

| 이름 | 뉘앙스 | 탈락 이유 |
|---|---|---|
| `process-engine` | 4차원 PDCA 엔진 | 너무 흔함 |
| `process-buffet` | 골라서 쓴다 | 캐주얼, 기업 도구치고 가벼움 |
| `process-mine` | "내 것" + "채굴" 이중 의미 | mine=지뢰 연상 가능 |
| `process-distill` | 표준에서 핵심 증류 | 의미는 좋으나 길다 |
| `process-tailor` | CMMI tailoring 개념 | 조금 좁은 느낌 |

**2차 후보** (단어 하나):

| 이름 | 뉘앙스 | 탈락 이유 |
|---|---|---|
| `Forge` | 만들고 단련 — 폐쇄 루프 이미지 | 최종 후보까지 생존 |
| `Foundry` | 주조소 — 원재료→완성품 | Forge와 유사 |
| `Nexus` | 연결·허브 | Oracle Nexus 선점 |
| `Prism` | 빛 분산 → 실용 요소로 | 추상적 |

### 최종 결정: `processware`

**이유**:
- GitHub 조직명이 이미 `Processware-AI` → `github.com/Processware-AI/processware` 로 일관성
- `-ware` suffix가 "소프트웨어·플랫폼"을 즉각 전달
- 4차원 PDCA 전 주기(Plan·Do·Check·Act)를 커버하는 플랫폼임을 반영
- `Processware-AI` 조직명이 "AI 기반 프로세스 플랫폼"이라는 포지셔닝을 완성

### 후속 조치 (진행 중)

## 3. 플랫폼 포지셔닝 최종 정리

이번 세션에서 명확해진 플랫폼 정체성:

```
Processware-AI / processware

"AI가 프로세스를 설계·실행·심사·개선하는 4차원 PDCA 플랫폼.
표준 코드가 아닌 실질 업무 요건에서 출발하는 프로세스 뷔페(process buffet)."

/process-plan  — 필요한 업무 체계를 자연어로 설명하면 AI가 문서 빌드
/process-do    — 만든 문서를 AI가 실행하며 기록 작성
/process-check — AI 심사원이 요건 ↔ 기록 자동 대조
/process-act   — 근본 원인 분석 → PCB 승인 → 자동 재빌드 → Plan으로 피드백
```
