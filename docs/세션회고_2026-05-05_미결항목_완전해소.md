## 2. 구현 1 — 멀티 소스 REC 그룹핑 (b788f14)

### 문제

동일 WI 실행 1건을 구성하는 여러 파일(예: 평가표 + 회의록 + 승인서)이 각각 별도 REC로 생성되는 구조. "원래 1개 REC여야 할 것"이 분산됨.

### 해결

**그룹핑 기준**: 동일 폴더 + 동일 best WI + confidence ≥ 30%

**backfill-matcher** (Phase 3-7):
- 기준 충족 파일 → `group_id: G1, G2, ...` 부여
- 단일 파일도 `is_singleton: true` 로 G# 부여 (일관된 group_id 체계)
- primary = confidence 최고 파일, 나머지 = supplementary
- `mapping_draft.yaml`에 `groups[]` 섹션 추가

**process-backfill** (Phase 4 HITL + Phase 5/6):
- HITL 테이블에 `그룹` 컬럼 + `★` primary 마커
- 수동 조정 명령: `"N+M group"`, `"N ungroup"`, `"GX primary=N"`
- Phase 5: 그룹 단위로 primary 파일 먼저 필드 추출, supplementary로 빈 필드 보완
- Phase 6: `{group_id}.yaml` 단위로 `backfill_payload.yaml` 저장

**rec-writer** (BF-1/BF-3/BF-4):
- `is_group: false` → `source_doc` / `source_hash` 단일 필드
- `is_group: true` → `sources[]` 배열 (role: primary/supplementary)
- 경고 박스: 단일("레거시 문서로부터 백필 변환된 기록") vs 멀티("{N}개 레거시 문서로부터 병합 백필된 기록") 분기

## 3. 구현 2 — process-audit → process-act 자동 트리거 (511d5e9)

### 문제

`/process-audit --confirm` 완료 후 critical/major GAP이 발견되어도 act 큐에 자동 등록되지 않아 수동으로 처리해야 했음.

### 해결

**act-trigger.md** — `from_gap` 3번째 모드 추가:
- Phase G: critical/major GAP 마다 `kind: gap_remediation` 큐 1건 생성
- priority: GAP severity 그대로 (critical/major)
- 중복 검사: 동일 `(gap_rec_id, clause)` 조합 재발행 방지. done/abandoned 시 재발행 허용.
- MAT-008 §"차원 4 인계" 표 자동 갱신

**process-audit.md** — confirm 흐름에 Phase 4 추가:
- gap-reporter 완료 후 critical + major > 0 이면 act-trigger(from_gap) 자동 호출
- `--no-act-queue` 플래그로 억제 가능
- §3 산출물, §5 옵션, §8 Roadmap 반영

**gap-reporter.md** — Phase F 추가:
- 완료 시 `critical_gaps[]` / `major_gaps[]` 데이터를 caller에게 반환
- 완료 출력에 `🔧 차원 4 자동 트리거: critical N건 · major N건 → act 큐 push 예정` 라인 추가

### 연결된 큐 구조

```yaml
queue_id: queue-q{hex}
kind: gap_remediation      # 신규 kind
priority: critical | major
source:
  gap_rec: REC-GAP-ISO9001-2026-001
  clause: "6.1"
  standard_code: ISO9001
target:
  proposed_action: "PRO-리스크관리 신규 설계 필요"
```

## 4. 구현 3 — 외부 인증기관 보고서 내보내기 (d555070)

### 문제

GAP 분석 결과가 내부 MD 보고서로만 존재 → ISO 심사관 등 외부에 제출할 표준 포맷 없음.

### 해결

**신규 에이전트 `gap-exporter.md`**:
- 입력: `coverage_matrix.yaml` + `REC-GAP-*.md` + 포맷/템플릿 옵션
- Phase A: 데이터 로드
- Phase B: 조항별 한국어 변환 (적합/부분/부적합/해당없음, 의무/권고/선택)
- Phase C-1 (xlsx): 3시트 openpyxl Python 스크립트 생성
  - 시트 1 "커버리지 요약": 기본 정보 + 판정/심각도 집계
  - 시트 2 "부합성 매트릭스": 전체 조항, verdict 색상, 자동 필터
  - 시트 3 "GAP 상세": critical → major → minor 순 정렬
  - 실제 coverage_matrix 데이터 임베드 (stub 금지)
- Phase C-2/3 (html/pdf): Noto Sans KR + A4 landscape print CSS 완성 HTML
- 출력: `vault/08_REC_기록/AUDIT/exports/`

**신규 모드 `/process-audit --export`**:
```
/process-audit --export run-gXXXXXXXX --format xlsx
/process-audit --export run-gXXXXXXXX --format html
/process-audit --export run-gXXXXXXXX --format pdf
/process-audit --export run-gXXXXXXXX --format xlsx --template iso27001
/process-audit --export run-gXXXXXXXX --format xlsx --org "OOO 주식회사"
```

- `status: completed` trace 만 처리
- xlsx: `pip install openpyxl && python {script}` 으로 .xlsx 생성
- pdf: HTML 파일 + 브라우저 인쇄/pandoc/wkhtmltopdf 안내

## 7. 핵심 결정 요약

**"동일 폴더 + 동일 WI = 1개 REC"**  
실무에서 1건의 WI 실행을 여러 파일로 남기는 케이스가 존재. 자동 그룹핑으로 이를 1개 REC에 병합. primary가 핵심 필드를 채우고 supplementary가 빈 필드를 보완하는 구조.

**"GAP 발견 즉시 차원 4 큐로"**  
process-audit confirm 후 critical/major GAP이 자동으로 act 큐에 push됨으로써 4차원 PDCA 루프가 audit → act 자동 연결. `--no-act-queue`로 수동 제어 가능.

**"Claude가 XLSX를 직접 생성할 수 없으면 스크립트로"**  
openpyxl Python 스크립트에 실제 데이터를 임베드하여 `python script.py` 한 줄로 .xlsx 생성. 에이전트가 stub 함수를 남기지 않고 완성 코드를 작성하는 것이 핵심 강제 규칙.
