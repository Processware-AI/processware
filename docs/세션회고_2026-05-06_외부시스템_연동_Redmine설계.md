# 세션 회고 — 2026-05-06 (2차)

## 1. 출발점

vault 산출물을 Redmine, Notion, Codebeamer 같은 외부 ALM/PMS/QMS 시스템과 연동하는 방안 검토.

---

## 2. 멀티 프로젝트·멀티 조직 지원 현황 확인

### 현재 반영 상태

| 필드 | 의미 | 반영 여부 |
|---|---|---|
| `scope_type` | 프로세스 수준 (project/org/common) | ✅ 반영 (flow-proposer → PRO → WI 상속) |
| `scope_code` | 표준 영역 코드 (ISO27001, CMMI 등) | ✅ 반영 (전 파이프라인 추적) |
| `department` / `team` | 부서·팀 단위 | ❌ 미반영 |
| 멀티 조직 vault 격리 | 조직 A / 조직 B 독립 운영 | ❌ 미반영 |

**결론**: `scope_type`은 "어떤 성격의 프로세스인가" 구분이지, "어느 부서/조직인가" 구분이 아님.

## 3. 외부 연동 방안 3가지 분석

### 방안 A — 외부 ALM/PMS/QMS에 자동 등록

- 강점: 기존 팀 도구에 콘텐츠 배포
- 약점: 임피던스 미스매치, 양방향 동기화 불가, vault SSoT 붕괴 위험
- 현실적 용도: 단방향 읽기 전용 배포에만 적합

### 방안 B — Vault 기반 별도 관리 시스템 직접 개발

- 강점: vault 8종 문서 유형·PDCA 사이클 완전 지원
- 약점: 개발 투자 과다, "또 하나의 시스템" 도입 장벽
- 현실적 용도: processware를 SaaS 제품으로 판매할 때 최종 목적지

### 방안 C — Vault를 공통 Repository + Open API 연동 (채택)

- 강점: vault SSoT 유지, 어댑터 단위 확장, 기존 파이프라인 무변경
- 약점: API 서버 운영 필요, 외부 편집 정책 설계 필요

## 4. 추천 모델 결정: Headless CMS

**"vault = Headless CMS"** — vault가 콘텐츠 저장·관리, 화면/시스템은 API로 소비.

```
전통적 CMS: [콘텐츠 저장] + [화면 렌더링] → 웹사이트만 출력
Headless CMS: [콘텐츠 저장(vault)] → API → Redmine도, Notion도, 자체 UI도 소비
```

**5-Layer 아키텍처 확정:**

| Layer | 구성 | 역할 |
|---|---|---|
| 0 | sources/ (PDF/DOCX) | 입력 원천 |
| 1 | AI 파이프라인 (차원 0~4) | 생성·유지 |
| 2 | vault (SSoT, Git) | 편집 원본 |
| 3 | Vault Index Service (SQLite + REST API) | API 게이트웨이 |
| 4 | Push Adapters (Redmine/Notion/CB) | 외부 배포 |
| 5 | 외부 소비 시스템 | 읽기·표시 |

**핵심 원칙:**
- 단방향 Push (vault → 외부). 역방향은 REC 코멘트만 예외
- 외부 시스템 = 배포 뷰, vault = 유일한 편집 원본

## 7. Redmine 연동 설계 완성

### 핵심 구성 요소

```
integrations/
└── redmine/
    ├── config.yaml          # Redmine 접속 정보 + 매핑 룰
    ├── sync.py              # 메인 동기화 스크립트
    ├── transformer.py       # vault MD → Redmine 포맷 변환
    ├── redmine_client.py    # Redmine REST API 래퍼
    └── requirements.txt
.claude/
├── commands/vault-push.md   # /vault-push 스킬
└── states/redmine_sync.db   # vault_id ↔ redmine_id 매핑 (SQLite)
```

### 문서 유형별 매핑

| vault type | status 조건 | Redmine 타겟 |
|---|---|---|
| POL/PRO/WI/TMP/EX/MAT | final | Wiki 페이지 |
| REC (일반) | final | Issue (Closed, 기록용) |
| REC-NCR | open/진행 | Issue (Open, NCR 추적) |
| draft 전체 | draft | skip |

### /vault-push 스킬 인터페이스

```
/vault-push                      # 전체 vault 동기화
/vault-push --changed            # git diff 기준 변경분만
/vault-push WI-ISO27001-001-001  # 특정 문서 1건
/vault-push --type WI            # WI 전체
/vault-push --scope ISO27001     # 특정 표준 전체
/vault-push --dry-run            # 미리보기
/vault-push --status             # 마지막 동기화 상태
```

### 구현 Phase

```
Phase 1: 기반 (1~2일) — config + API 래퍼 + SQLite 스키마 + dry-run
Phase 2: Wiki 동기화 (2~3일) — WI/PRO/POL → Redmine Wiki push
Phase 3: Issue 동기화 (1~2일) — REC/NCR → Redmine Issue
Phase 4: 자동화 (1일) — git post-commit hook + /vault-push 스킬 등록
```

## 8. 핵심 결정 요약

**"vault는 Headless CMS다"**
Markdown + frontmatter = 구조화 콘텐츠 저장소. API 게이트웨이만 붙이면 어떤 시스템도 소비 가능.

**"단방향 Push가 유일한 현실적 시작점"**
양방향 동기화 = vault SSoT 붕괴. 외부 시스템은 projection(배포 뷰), 편집은 vault에서만.

**"Redmine = 첫 번째 어댑터 타겟"**
기존 --export 패턴 확장. integrations/redmine/ 디렉토리 + /vault-push 스킬로 구현.

**"멀티 조직/부서 scope는 미구현 → 향후 과제"**
scope_code(표준 영역)와 scope_type(프로세스 수준)은 있으나, department/team 단위 구분은 없음.
Redmine의 project_mapping(scope_code → Redmine 프로젝트)으로 표준 영역별 격리는 가능.

## 9. Redmine 구조 심화 설계 — Workspace 개념 확정

### 9-1. PRO별 서브프로젝트 구조 채택

scope_code별 서브프로젝트보다 한 단계 더 세분화.
vault의 PRO→WI 계층이 Redmine 프로젝트 계층에 직접 대응되어 REC·NCR을 PRO 단위로 추적 가능.

```
현업 질문 예시:
"리스크관리 절차에서 올해 발생한 NCR 몇 건?"
→ lib-iso27001-riskmanagement Issues 필터링
```

### 9-2. 용어 충돌 발견 및 해소

**문제**: "프로젝트"라는 단어가 세 가지 의미로 혼용됨.
- 현업 업무 프로젝트 (비즈니스 단위)
- Redmine Project (기술적 컨테이너)
- processware scope_type: project (프로세스 적용 수준)

**해결**: Workspace 개념 도입.

> "현업의 업무 단위를 가상의 업무공간(Workspace)으로 만들겠다.
> 그것이 Redmine에서는 여러 개의 Project(컨테이너)로 구성된다."

현업 Workspace 1개 : Redmine Project N개 (1:N 관계)

### 9-3. 확정 용어 체계

| 개념 | 용어 | Redmine prefix |
|---|---|---|
| 표준 프로세스 저장소 | **Process Library** | `lib-{scope_code}` |
| PRO 단위 라이브러리 | **Library Module** | `lib-{scope_code}-{pro}` |
| 현업 업무 단위 가상공간 | **Workspace** | `ws-{name}` |
| Workspace 내 PRO 공간 | **Workspace Module** | `ws-{name}-{pro}` |
| 생성 스킬 | `/workspace-create` | Library Module 복사·조합 |

### 9-4. 3-tier 전체 흐름

```
vault (SSoT)
  → /vault-push
  → Process Library (lib-*)   ← processware 관리, READ ONLY
  → /workspace-create (필요 PRO 선택)
  → Workspace (ws-*)          ← 팀 관리, 자율 편집 가능
```

**Library vs Workspace 정책:**
- Library: vault 변경 시 자동 갱신 (단방향 Push 유지)
- Workspace: 생성 시점의 Library 스냅샷. 이후 독립. 팀이 직접 편집 가능.
- 버전 업데이트 원할 시: `/workspace-sync --ws ws-product-alpha --module riskmanagement`
