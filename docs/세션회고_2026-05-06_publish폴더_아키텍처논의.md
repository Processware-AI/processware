## 2. vault 쿼리 한계 → Document/Record 레이어 분리 논의

### 논의 핵심

"vault의 쿼리에 한계가 있다면, 애초에 DBMS 스키마로 만들면 되는가?"

→ vault는 **사람이 읽는 규범 문서**이기도 하고 **AI agent가 관리하는 프로세스 레코드**이기도 하다. 둘 다 중요하다. 또한 Redmine/Notion 연동뿐 아니라 파일 서버, Excel/Word/PPT 포맷 변환도 필요하다.

→ 결론: **Document Layer / Record Layer** 분리가 정답.

| 레이어 | 대상 | 저장소 |
|--------|------|--------|
| Document Layer | POL / PRO / WI / TMP / EX | vault/*.md (git source of truth) |
| Record Layer | REC / trace / KPI / NCR / MAT | SQLite or PostgreSQL |
| Publish Layer | Redmine / Notion / 파일서버 / XLSX·DOCX·PPT | publish/ 어댑터 |

- 문서는 여전히 마크다운 — Git 감사로그, Obsidian UI, 오프라인 유지
- 레코드는 DB — AI agent가 SQL/API로 쿼리, 집계, 조인
- MAT-005~009 → DB aggregation view → md로 렌더링

## 3. `integrations/` → `publish/` 개명

### 논의 과정

- `integration`은 양방향 동기화 뉘앙스 → 실제 관계는 단방향 (vault → 외부)
- `export`는 GAP 보고서 포맷 변환과 의미 충돌
- `adapters/`는 기술적으로 정확하나 발행 의미 미흡
- `publishers/` → 어렵다 → **`publish/`** 로 최종 결정

### 결정 이유

- 파일 서버 + 외부 시스템 API + 포맷 변환(Excel, Word, PPT) 세 가지를 **발행(publish)** 하나로 통일
- 한국어 "발행"과 일치, processware 전체에서 일관성 유지
- vault-push 스킬 이름과도 방향성 일치

### 변경 범위

- `integrations/` → `publish/` 폴더 이동
- 내부 경로 참조 치환 (dispatcher.py, install-hook.sh)
- `.claude/commands/`, `.github/workflows/`, README, docs/, vault/매뉴얼 전체 치환
- frontmatter YAML 키 `integrations:` → `publish:` (세션회고 설계 예시 포함)

## 4. 핵심 결정 요약

**"vault는 Document Layer, REC·trace는 Record Layer로 분리한다"**
마크다운 source of truth는 유지하되, SQLite를 읽기 전용 인덱스 캐시로 추가. AI agent는 파일 grep 대신 SQL 쿼리를 사용.

**"외부 출력 폴더는 publish/로 통일한다"**
단방향 발행(외부 시스템 + 파일서버 + 포맷 변환) 전체를 하나의 개념으로 묶음.

## 5. 후속 논의 (2026-05-07) — Record Layer 구현 보류 결정

### 재검토 내용

- Document Layer = 사람에게 제시하는 규범
- Record Layer DB = AI 에이전트 자동화 기반 (SQL 집계·조인)
- .md를 source of truth로 유지하고 DB는 읽기 캐시로 인덱싱하는 방안 검토

### 결론: **보류**

- 현재 REC 수가 적어 grep으로 충분 — 가상의 문제
- .md ↔ DB 동기화 자체가 복잡도를 추가함
- **트리거: 실제 운영에서 REC가 대량 누적될 때 재검토**
