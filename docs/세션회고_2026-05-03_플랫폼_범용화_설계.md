## 2. 아키텍처 확인 — 브랜치 = 표준 실행 환경

각 표준 브랜치(`feat/{표준코드}-output`)는 독립적인 실행 환경이다:

```
main (플랫폼)
├── .claude/agents/       ← AI 에이전트 정의
├── .claude/commands/     ← /build-process, /do, /audit, /act
└── vault/02_표준/_scaffold/   ← 빈 템플릿

feat/CMMI-DEV-ML3-output (표준별 브랜치)
├── .claude/              ← main에서 상속
└── vault/
    ├── 02_표준/CMMI-DEV-ML3/   ← 표준 분석 결과
    ├── 03_POL ~ 07_EX          ← 빌드된 프로세스 자산
    └── 08_REC/                  ← /do 실행 시 산출물 누적
```

`git worktree`로 브랜치를 별도 폴더에 마운트하면 Obsidian과 Claude Code CLI가 동일한 파일시스템을 공유한다. 에이전트가 REC를 작성하면 Obsidian 그래프 뷰에 즉시 반영된다.

## 3. Export 레이어 설계 방향

### vault → 외부 시스템

vault 문서는 YAML frontmatter + 정해진 섹션(§RACI, §KPI, §step)으로 구성된 파싱 가능한 구조체다. 이를 기반으로 다양한 외부 시스템으로 변환 가능하다:

| 타겟 | 방법 | 매핑 |
|---|---|---|
| MS Office | pandoc / python-docx | TMP → Word 양식, WI 표 → Excel |
| Notion | Notion API | POL→Page, WI steps→Database rows |
| Jira | Jira REST API | PRO→Epic, WI step→Story, RACI R→Assignee |
| Redmine | Redmine REST API | PRO→Project, WI→Wiki, step→Issue |

### 일감 vs 콘텐츠 판단 기준

| type | 외부 시스템 형태 |
|---|---|
| POL, TMP, EX, REF | 콘텐츠 (Wiki/Page) |
| PRO | 콘텐츠 + Epic (실행 시) |
| WI | 콘텐츠 + Story/Issue per step |
| REC | 일감 (완료) 또는 DB row |
| MAT | 대시보드 / DB view |

보조 판단: `hitl_required: true` → 반드시 일감 / RACI R 존재 → 담당자 자동 배정

### 크로스 플랫폼 연동

`doc_id`를 공통 키로 외부 시스템 ID를 vault frontmatter에 역기록:

```yaml
integrations:
  jira: "PROJ-123"
  confluence: "PAGE-456"
  notion: "abc123def"
```

vault = 단일 진실 공급원, 외부 시스템 = 동기화된 뷰.

## 4. 다중 사용자 / Redmine 진입 시 에이전트 변화

### 현재 한계
단일 사용자가 PM·QA·QMR·Process Owner 역할을 전부 수행. `independence-guard`는 실제로는 같은 사람을 대상으로 한다.

### 다중 사용자 환경에서 달라지는 것

| 에이전트 | 현재 | 다중 사용자 |
|---|---|---|
| `hitl-gatekeeper` | 단일 사용자에게 pause/resume | RACI A 계정으로 외부 이슈 발행 → 응답 감지 |
| `independence-guard` | role 이름 비교 | user_id 비교 (실행자 ≠ 심사자) |
| `process-executor` | 역할 무관 실행 | 세션 ID 주입 — "나는 PM 역할의 홍길동" |
| `rec-writer` | 서명 없음 | executor user_id + timestamp 서명 |

### 핵심 설계 개념: 세션 아이덴티티

```yaml
session:
  user_id: "hong.gildong"
  role: "PM"
  redmine_account: 42
```

에이전트 로직 자체는 크게 바뀌지 않는다. 바뀌는 것은 **"누가 실행하는가"를 인식하는 방식**과 **HITL 게이트의 통지 채널**이다.

## 5. 멀티 프로젝트 관리

### 현재 CMMI vault 상태
TMP/EX 146종이 있으나 모두 **단일 프로젝트 인스턴스** 관점. 포트폴리오·크로스 프로젝트 자원 배분·프로젝트 간 의존성 관련 양식은 없다.

### 추천 구조: 2-tier

```
[Tier 2] PMO 표준 vault (신규, 경량)
    └ 포트폴리오 대시보드 TMP
    └ 조직 자원 배분표 TMP (cross-project)
    └ 프로젝트 간 의존성 매트릭스 TMP
    └ 조직 레벨 성과 통합 보고서 TMP
    └ 프로젝트 등록부

[Tier 1] 프로젝트별 vault
    feat/CMMI-DEV-ML3-{프로젝트A}-output
    feat/CMMI-DEV-ML3-{프로젝트B}-output
    feat/ISO27001-{프로젝트C}-output
```

Redmine의 상위 프로젝트(PMO) → 하위 프로젝트(각 프로젝트)와 자연스럽게 매핑된다.

## 6. CMMI PA 성격 분류 — 조직 vs 프로젝트

CMMI PA를 전부 프로젝트 vault에 넣는 것은 개념적으로 틀리고 낭비다.

| PA | 성격 | 프로젝트 vault에? |
|---|---|---|
| GOV | 전사 거버넌스 | ❌ 조직에 하나 |
| II | 구현 인프라 | ❌ 조직에 하나 |
| PAD | 프로세스 자산 개발 | ❌ STD_Process_Builder 자체가 PAD |
| MPM (Managing Performance and Measurement) | 조직 성과 측정 | ❌ 프레임워크는 조직 |
| CAR / DAR / PCM | 이중 성격 | 🔶 정의는 조직, 실행 REC는 프로젝트 |
| PLAN / MC / EST / RDM / TS / PI / VV / PR / SAM | 프로젝트 실행 | ✅ 프로젝트 vault |

**핵심**: CMMI가 의도하는 OSSP(조직 표준 프로세스 셋) 개념이 이미 이 아키텍처에 구현되어 있다.
- main = OSSP (조직 표준 자산)
- 프로젝트 브랜치 = 테일러링 인스턴스

## 7. PA 그룹화 — 프로세스 모듈화

CMMI 전체를 하나의 표준으로 빌드하는 것은 조직·프로젝트 레벨 PA를 섞는 문제를 야기한다. PA 그룹 단위로 모듈화해서 필요한 것만 조합하는 방식이 맞다.

```
CMMI-ORG-FOUNDATION        (조직 기반, 1회 구축)
  GOV · II · PAD · OT

CMMI-ORG-PERFORMANCE       (조직 성과, 상시 운영)
  MPM

CMMI-PRJ-PLANNING          (프로젝트 계획·관리)
  PLAN · MC · EST · RSK · SAM

CMMI-PRJ-ENGINEERING       (기술 개발)
  RDM · TS · PI · VV · PR

CMMI-PRJ-SUPPORT           (프로젝트 지원)
  CAR · DAR · PCM · PQA
```

프로젝트는 필요한 그룹만 조합 적용. ML 레벨이 올라가면 그룹을 추가하는 방식으로 단계적 확장 가능.

### 배경

### 변경

- 24개 파일 내 모든 참조 일괄 치환

### 새로운 입력 패러다임

```
기존: /build-process CMMI-DEV-ML3
새로: /build-process "프로젝트 계획·추정·리스크 관리"
      /build-process "소프트웨어 품질보증 + 형상관리"
      /build-process "조직 거버넌스 + 성과측정"
```

자연어로 필요한 업무 체계를 설명하면, 관련 표준/프레임워크에서 해당 요소를 추출해 조합한다. 표준 코드는 출처 레퍼런스로 남고, 빌드의 주인공은 **실질 업무 요건**이다.

## 9. 프로세스 뷔페 — 핵심 개념

표준이나 인증 코드가 빌드의 출발점이 아니다. 조직이 필요로 하는 **실질 업무 체계**가 출발점이고, 표준은 그 출처 레퍼런스다.

### 인증과 프로세스의 분리

```
기존: CMMI ML3 인증 목표 → 20개 PA 전부 구현 → 인증 통과
새로: 필요한 프로세스 빌드 → 잘 돌아가는 프로세스 운영
      → 인증 필요 시: /audit --against CMMI-ML3 → 갭 분석
      → 부족한 부분만 추가 빌드 → 인증 준비 완료
```

인증이 필요 없는 조직은 표준 코드를 신경 안 써도 된다. 인증이 필요한 조직은 gap audit 한 번으로 부족한 것만 파악하고 추가한다.

**표준이 목적이 아니라 도구가 된다.**
