## 3. "지원 표준" → "입력 가능 레퍼런스" 재구성 (e3e7425)

"이 프로젝트는 특정 표준에 한정되어 있지 않잖아? 세상의 모든 레퍼런스를 통해서 표준 프로세스를 만들 수 있는 거 아닌가?"

→ 맞다. 기존 README의 "지원 표준" 섹션이 특정 ISO/IEC/IATF 목록을 나열하면서 **마치 그것만 지원하는 것처럼** 오해를 줬음.

### 해결

- "지원 표준" → **"입력 가능 레퍼런스"** 섹션으로 전면 재구성
- 유형별 예시 표: 국제 표준 / 역량 모델(CMMI 포함) / 국내 표준·법규 / 사내 규정 / 고객·계약 요구사항 / 산업 가이드라인
- 핵심 메시지: `/process-ingest sources/{문서}.pdf --standard {레이블}` 한 줄로 어떤 문서든 편입 가능
- 차원 1 특징 줄: "자동차/의료기기 8개 도메인 전용" → **"표준 무관(standard-agnostic)"** 으로 교체

## 4. vault → 외부 시스템 전환 가능성 논의

"이 프로젝트 결과물인 Obsidian vault 구조를 외부 시스템의 컨텐츠 구조·DB·실행 로직으로 전환이 가능하지 않나?"

→ 맞다. vault는 이미 **파일 형태의 데이터 모델**이다.

| vault 구조 | 외부 시스템 전환 |
|---|---|
| frontmatter YAML | DB 컬럼 / JSON 스키마 |
| MD 본문 | Rich text / Content 필드 |
| 파일 간 참조 | Foreign key / 관계 테이블 |
| WI steps.yaml | 워크플로우 엔진 태스크 |
| MAT-001~010 | DB 뷰 / 집계 테이블 |
| act queue YAML | Task queue 테이블 |
| trace.jsonl | Event sourcing / 감사 로그 |
| HITL 게이트 | 승인 워크플로우 노드 |

processware = 설계·생성 도구 / 외부 시스템 = 운영·실행 환경

이 논의가 프레임워크 설계안 §1-B/C, §6 기술적 도전 포인트에 반영됨.

## 6. 이 리포지토리의 성격 인식

"이 프로젝트 리포지토리 자체가 시스템 개발 기획안이 될 것 같은데."

→ 맞다. 이 repo는 동시에:
- **동작하는 PoC** (에이전트 29개, KPI 회복 실증)
- **기술 아키텍처 설계서** (4차원 PDCA 구조)
- **기능 명세서** (커맨드·에이전트 파일 자체가 스펙)
- **사용자 매뉴얼** (7종)
- **제품화 로드맵** (Phase 1~4, CLI → SaaS)

일반적인 기획안이 `문서 → 개발 → 산출물` 순서라면, 이 프로젝트는 `PoC 동작 → 문서 자동 생성` 역전 구조.

## 7. 핵심 결정 요약

**"이 플랫폼은 특정 표준에 한정되지 않는다"**
어떤 구조화 문서든 /process-ingest로 편입 가능. ISO 목록 열거 방식은 오해를 줌.

**"vault는 파일 형태의 DB다"**
frontmatter→컬럼, MD→Content, steps.yaml→워크플로우, trace.jsonl→이벤트 로그. Obsidian은 편집 레이어, DB는 운영 레이어로 공존 가능.

**"설계안은 살아있는 문서다"**
하네스 구현이 누적될수록 설계안도 동기화 필요. 특히 에이전트 수·커맨드 수·검증 완료 항목은 구현 직후 갱신.
