# 표준 프로세스 AI 관리체계 — 4차원 PDCA

> 작성일: 2026-04-23 · 상태: draft

## 1. 개요

국제/국내 표준(ISO·IEC·KS·IATF 등)을 조직에 내재화하는 과정은 **4개 차원**이 순환하는 PDCA 구조로 이루어진다. 각 차원은 개념적으로 분리된 독립 단계이며, AI 에이전트가 각 차원에서 서로 다른 역할을 수행한다.

---

## 2. 전체 구조

```
        ┌─────────────────────────────────────────────────┐
        │                   ACT                           │
        │    개선 과제 → 문서 개정 → 차원 1 재실행        │
        │                                                 │
        ▼                                                 │
[차원 1] PLAN                                            │
  표준 원문 분석                                         │
  POL / PRO / WI / TMP / EX / MAT / REF 생성            │
        │                                                 │
        ▼                                                 │
[차원 2] DO                                              │
  절차 이행 · 기록 생성 · 증적 축적                      │
  담당자: WI 안내 → TMP 채움 → REC 제출 → 승인           │
        │                                                 │
        ▼                                                 │
[차원 3] CHECK                                           │
  심사 · 평가 · 부적합 탐지                              │
  심사원: REC 검토 → 요건 대조 → NCR 발행                │
        │  ※ 심사원 ≠ 이행 담당자 (ISO 독립성 원칙)       │
        │                                                 │
        ▼                                                 │
[차원 4] ACT ────────────────────────────────────────────┘
  근본 원인 분석 → 개정 범위 결정 → 차원 1 재트리거
```

## 3. 차원별 상세

### 차원 1 — PLAN: 프로세스 자산 구축

| 항목 | 내용 |
|---|---|
| **목적** | 표준 요구사항을 조직 맥락에 맞는 프로세스 문서 체계로 변환 |
| **입력** | 표준 원문, 법규, 해설서, As-Is 프로세스, 산업 가이드 |
| **출력** | POL / PRO / WI(MD + steps.yaml) / TMP / EX / MAT / REF |
| **주체** | 품질관리자, 프로세스 오너, 컨설턴트 |
| **빈도** | 표준 도입 시 1회 + 표준 개정·차원 4 피드백 시 재실행 |
| **AI 역할** | 표준 분석 → 설계 → 문서 생성 → 추적성 매핑 → QA (5단계 파이프라인) |
| **현황** | **구현됨** (하네스) · WI 이중 포맷 출력은 **미반영** — 확장 필요 |

**AI 파이프라인:**
```
standard-analyzer → process-designer → wi-tmp-writer → traceability-mapper → qa-reviewer
                                              │
                                    WI 이중 포맷 생성 (확장 예정)
                                    ├── WI-*.md          (사람이 읽는 지침서)
                                    └── WI-*.steps.yaml  (Agent가 실행하는 Step 정의)
```

### 차원 2 — DO: AI Agent 주도 프로세스 이행

| 항목        | 내용                                                  |
| --------- | --------------------------------------------------- |
| **목적**    | AI Agent가 PRO/WI를 읽고 사람과 대화하며 절차를 이행하고 산출물을 작성      |
| **입력**    | PRO/WI (절차·지침), TMP (양식 스키마), Trigger (이벤트·일정·요청)   |
| **출력**    | REC (기록본) — AI가 대화 기반으로 작성, MAT-005 자동 갱신           |
| **주체**    | **AI Agent** (실행) + 사람 (정보 제공·HITL 승인)              |
| **빈도**    | 업무 이벤트 발생 시마다                                       |
| **AI 역할** | PRO/WI 해석 → 대화식 정보 수집 → TMP 자동 채움 → REC 생성 → 승인 라우팅 |
| **현황**    | 미설계                                                 |

#### 기본 실행 모델 — 대화식 (C안)

사람은 절차서(WI)를 보지 않아도 된다. Agent가 WI를 내재화하여 필요한 정보를 대화로 수집하고 산출물을 생성한다.

```
[Trigger: "문서 검토 절차 시작"]

Agent: 검토 대상 문서명을 알려주세요.
사람:  품질방침 개정안 v2.0 입니다.

Agent: 검토자로 지정된 분은 누구입니까?
사람:  김철수 팀장입니다.

Agent: 검토 결과는 어떻습니까? (승인 / 조건부승인 / 반려)
사람:  조건부 승인입니다. 3페이지 용어 수정이 필요합니다.

Agent: [TMP-QMS-001-01-02 검토확인서 자동 작성]
       ── 아래 초안을 확인해주세요 ──
       문서명: 품질방침 개정안 v2.0
       검토자: 김철수 팀장
       결과:   조건부승인
       의견:   3페이지 용어 수정 필요
       ────────────────────────────
       확인하시면 승인 요청을 발송합니다.

사람:  확인했습니다.

Agent: [HITL Gate] 김철수 팀장에게 최종 승인 요청 발송.
       승인 완료 시 REC 자동 확정 → MAT-005 기록.
```

#### 대화 흐름 구조

WI의 각 Step이 대화 질문으로 변환된다. Step에 명시된 입력 항목 = 질문 목록, 출력 항목 = TMP 필드 매핑.

```
WI Step
├── 입력 항목  →  Agent 질문으로 변환
├── 처리 규칙  →  Agent 내부 로직 (판단·계산)
├── 출력 항목  →  TMP 필드 자동 채움
└── HITL 표시  →  사람 승인 게이트 삽입
```

#### HITL Gate 기준

모든 Step을 AI가 자동 처리하지 않는다. 아래 조건 중 하나라도 해당하면 반드시 사람이 개입한다.

| 조건 | 이유 |
|---|---|
| ISO 요건에 "책임자 서명" 명시 | 규제 요구사항 |
| 금전·법적 효력 발생 | 리스크 통제 |
| 전례 없는 예외 상황 | AI 판단 한계 |
| WI에 `hitl: required` 명시 | 프로세스 오너 결정 |

#### 확장 인터페이스 (A·B안)

대화만으로 충분하지 않은 경우, 아래 인터페이스를 MCP Tool로 플러그인한다.

| 확장 | 용도 | 예시 |
|---|---|---|
| **A안 — 내부 데이터** | 파일·DB 조회·기록 | 기존 REC 이력 조회, 문서 버전 확인 |
| **B안 — 외부 시스템** | ERP·Jira·이메일 API | 공급자 데이터 조회, 이슈 자동 생성, 알림 발송 |

A·B안은 차원 2 기본 설계 완료 후 MCP 서버로 순차 추가한다.

### 차원 3 — CHECK: 심사·평가

| 항목 | 내용 |
|---|---|
| **목적** | 이행 결과(REC)가 요건(PRO/WI)을 충족하는지 독립적으로 검증 |
| **입력** | REC (차원 2 산출), PRO/WI 요건, KPI 기준, MAT-002 규제대조표 |
| **출력** | 심사 보고서, NCR (부적합 보고서), KPI 대시보드, 시정조치 요청 |
| **주체** | 내부심사원, QMR, 외부심사원 (인증기관) |
| **빈도** | 정기 (반기·연간) + 수시 (이슈 발생 시) |
| **AI 역할** | REC ↔ PRO/WI 자동 대조, 부적합 패턴 탐지, 심사 보고서 초안, 증적 인덱싱 |
| **현황** | 미설계 |

> **독립성 원칙 (ISO §9.2)**: 심사원은 자신이 수행한 업무를 심사하지 않는다. 플랫폼 구현 시 역할 분리(RBAC)로 강제.

**심사 흐름:**
```
REC 수집 (MAT-005 기반)
    → AI: PRO/WI 요건 항목별 충족 여부 자동 대조
    → 부적합 탐지 → NCR 초안 생성
    → 심사원 검토·확정
    → 심사 보고서 완성 → 차원 4 인계
```

---

### 차원 4 — ACT: 제개정

| 항목 | 내용 |
|---|---|
| **목적** | 심사 결과를 반영하여 프로세스 체계를 개선하고 다음 사이클 품질을 높임 |
| **입력** | NCR, 심사 보고서, 시정조치 완료 보고, 표준 개정판 |
| **출력** | 개정 POL/PRO/WI, 개정 이력 (MAT-001 반영) → **차원 1 재입력** |
| **주체** | 프로세스 오너, PCB (Process Control Board), QMR |
| **빈도** | 심사 주기 후 + 표준 개정 시 |
| **AI 역할** | 근본 원인 분석 보조, 개정 범위 제안, 차원 1 파이프라인 재트리거 |
| **현황** | 미설계 |

**차원 4 → 차원 1 연결 (핵심 루프):**
```
NCR 분석 결과
    → 개정 대상 PRO/WI 식별
    → As-Is 피드백을 _inputs/04_AsIs/ 에 투입
    → /plan-process {표준코드} --from design 재실행
    → 개정판 문서 생성 (초판보다 실운영 피드백 반영으로 품질 향상)
```

## 5. 구현 원칙

### 5.1 개념 분리 · 플랫폼 통합

PDCA 4단계는 **개념적으로 분리**한다. 구현은 하나의 플랫폼 안에서 **역할(Role) 분리**로 제공한다.

```
[단일 플랫폼]
    │
    ├─ Role: 업무 담당자    → 차원 2 워크플로우만 접근
    ├─ Role: 내부심사원     → 차원 3 심사 도구만 접근 (자신의 업무 제외)
    ├─ Role: 프로세스 오너  → 차원 4 개정 승인 권한
    └─ Role: QMR / Admin   → 전 차원 모니터링
```

### 5.2 AI 독립성 보장

차원 2(Do)와 차원 3(Check)의 AI는 같은 LLM을 쓰더라도 **서로 다른 컨텍스트·권한**으로 실행한다. 차원 3 AI는 차원 2의 수행 과정을 실시간으로 알 수 없어야 한다 (사후 검토만 허용).

### 5.3 WI 이중 포맷 — 차원 1·2 인터페이스

차원 2 Agent가 WI를 실행하려면 WI가 **이중 포맷**으로 생성되어야 한다. `wi-tmp-writer`가 두 파일을 함께 출력한다.

| 파일 | 대상 | 역할 |
|---|---|---|
| `WI-{식별번호}_제목_v1.0.md` | 사람 | 기존 형식 — 서술형 지침서 |
| `WI-{식별번호}_제목_v1.0.steps.yaml` | Agent | Step별 질문·입출력·HITL 정의 |

**steps.yaml 구조:**
```yaml
wi_id: WI-QMS-001-01-02
title: 문서 검토 및 승인
version: "1.0"
steps:
  - id: step-01
    name: 검토 대상 문서 확인
    inputs:
      - field: doc_name
        type: text
        question: "검토 대상 문서명을 알려주세요."
      - field: reviewer
        type: person
        question: "검토자로 지정된 분은 누구입니까?"
    outputs:
      - tmp_field: 문서명
        from: doc_name
      - tmp_field: 검토자
        from: reviewer
    hitl: false

  - id: step-02
    name: 검토 결과 판정
    inputs:
      - field: result
        type: enum
        options: [승인, 조건부승인, 반려]
        question: "검토 결과는 어떻습니까?"
      - field: comments
        type: text
        question: "검토 의견을 작성해주세요."
        required_if: "result in [조건부승인, 반려]"
    outputs:
      - tmp_field: 결과
        from: result
      - tmp_field: 검토의견
        from: comments
    hitl: false

  - id: step-03
    name: 최종 승인
    hitl: required
    approver_role: 프로세스 오너
    approval_question: "위 검토 내용을 확인하고 승인해주세요."
```

> **WI 개정 시**: MD와 steps.yaml을 함께 개정한다. steps.yaml 버전이 MD 버전과 불일치하면 차원 3 QA에서 부적합으로 탐지한다.

### 5.4 차원 4 재트리거 조건

모든 개정이 차원 1 재실행을 필요로 하지 않는다. 재실행 기준:

| 개정 범위 | 조치 |
|---|---|
| 오탈자·링크 수정 (v1.n) | 해당 문서만 직접 편집 |
| 소절 추가·책임 변경 | `--from write` 또는 `--from design` 부분 재실행 |
| 구조·범위·정책 변경 | `--restart` 전체 재실행 |
| 상위 표준 개정판 편입 | `/plan-process {코드} --restart` |
