## 2. 본질 재발견 — RFP 는 표준 자산이 아니다

본 메인 repo 의 주력은 **국제표준·법규·산업가이드 → 조직 표준 프로세스 자산(POL/PRO/WI/TMP/EX/MAT)** 구축. 즉 **"조직이 모든 사업에서 항상 따르는 영속 기준"** 만 vault 에 들어간다.

반면 RFP 는:
- 1회성 사업의 발주 요구사항
- 다음 사업 RFP 와 100% 중복되지 않음
- 영속 자산화 부적합 — 사업 종료 시 archive 대상

→ `/process-plan "SI-Project"` 같은 호출은 본 메인 repo 의 주류 흐름과 어긋남. RFP 처리는 **별도 파생 프로덕트의 영역**이라는 인식 확립.

## 3. 3-Vault 비전 정립

사용자 비전 4회 정정을 거치며 다음 구조 확정:

```
┌─ 📦 코어 vault — Processware ─┐
│   국제표준/법규 → POL/PRO/WI  │
└────────────────────────────────┘
                                  ┌─ 🤖 파생 ① — RFP-to-Proposal ─┐
                                  │   RFP → 응시자 제안서(Proposal)│
                                  │   (별도 vault, 코어와 결합 안함)│
                                  └────────────────────────────────┘
                                                ↓ (수주 후)
                                  ┌─ 🛠 파생 ② — Project Asset Gen ─┐
                                  │   파생 ① 제안서 + 코어 vault    │
                                  │   → 사업 산출물                  │
                                  └──────────────────────────────────┘
```

**핵심 정정 4건**:
1. 파생 ② 의 입력은 "정규화 RFP" 가 아니라 **"파생 ① 의 제안서"**
   → 수주된 사업의 산출물 기준은 RFP(요구) 가 아니라 **채택된 제안서(약속)**
2. "제안요청서" 는 오타, 실제는 **제안서(Proposal)** — 응시자가 제출
3. 파생 ① 은 코어 vault 를 입력으로 받지 않음 — **세 vault 가 독립된 구조·내용**
4. 결합점은 파생 ② 하나뿐 (코어 + 파생 ① → 파생 ②)

## 7. 부수 발견 — gitignore negation 규칙의 함정

`inputs/04_AsIs/SI_Project/` 첫 commit 시도 시 macOS 메타데이터 (`._state.yaml`, `._annexes.yaml` 등) 가 staged 에 자동 포함되는 문제 발견.

원인: `.gitignore` 의 `inputs/` 영역 negation 규칙 (`!/inputs/*/*/*.yaml` 등) 이 `**/._*` 일반 ignore 패턴을 무효화. macOS metadata 파일의 확장자가 `.yaml`/`.md` 이라 negation 매칭됨.

해결: negation 규칙 뒤에 명시적 재제외 패턴 추가
```
inputs/**/._*
inputs/**/.__*
subproducts/**/._*
subproducts/**/.__*
```

→ git ignore 우선순위 규칙(나중 패턴 우선) 학습.
