## 2. 핵심 논의 흐름

### 2-1. 백필의 정의와 `/process-do`와의 차이

"이미 존재하는 문서를 표준 REC로 편입"하는 작업.

| | `/process-do` | `/process-backfill` |
|---|---|---|
| 시점 | 앞으로 만들 기록 | 이미 있는 문서를 편입 |
| 증적 성격 | 정식 이행 증적 | 레거시 증적 |
| 심사 처리 | conformant 집계 | partial 집계 |

### 2-2. 3가지 설계 결정

**Q1 — 자동 WI 매칭 필요한가?**  
→ 필수. 파일 수십 개를 수동으로 WI에 연결하는 것은 현실적이지 않음.

**Q2 — no-match 파일 처리**  
→ 제외, 강제 진행 없음. "짝이 없으면 기록이 될 수 없다."

**Q3 — 독립 커맨드 vs `/process-do --import`**  
→ 독립 커맨드(`/process-backfill`). 이유: 파이프라인 구조가 완전히 다름(Extract → Match → HITL → Map → Generate). `/process-do`에 --import를 끼워 넣으면 복잡도만 증가.

### 2-3. confidence 임계값 설계

| 범위 | 표시 | 처리 |
|---|---|---|
| ≥ 75% | ✅ | 자동 확정 (HITL 수정 가능) |
| 50–74% | ⚠️ | HITL 수정 권장 |
| < 50% | ❌ | 자동 제외 |

**4-신호 가중치 구조**:
- 키워드 오버랩 40% + 산출물 유형 30% + 제목 유사도 20% + 표 헤더 10%

### 2-4. `legacy_evidence` verdict 신설

백필 REC를 기존 4가지(conformant/partial/nonconformant/not_assessed)와 구별하기 위해 5번째 타입 추가.

**핵심 특성**:
- KPI 집계: `partial`과 동일하게 계산
- NCR 발행: 없음 (이행 실패가 아니라 증적 형태의 차이)
- 심사 보고서: `legacy_evidence_count` 별도 집계
- 오버라이드: `/process-check --treat-legacy-as conformant` (심사원 판단 시)

### 2-5. HITL 단일화 원칙

파일별 HITL 반복 없이 **전체 매핑 결과를 한 테이블로 한 번만** 제시.  
배치 처리에서 수십 개 파일마다 멈추면 작업이 불가능함.

## 3. 스캔본 PDF 문제와 --ocr 해결

### 3-1. 문제 제기

"스캔본 PDF를 처리하는 방법은 없나?"

기존 설계: 스캔본 감지 → 즉시 중단.  
두 시스템 모두(ingest + backfill) 스캔본 PDF를 처리할 방법이 없었음.

### 3-2. --ocr 플래그 설계

**엔진**: `ocrmypdf` (Tesseract 백엔드), `brew install ocrmypdf`  
**명령**: `ocrmypdf --force-ocr -l kor+eng <입력> <출력>`  
- `-l kor+eng`: 한국어·영어 동시 인식
- `--force-ocr`: 텍스트+이미지 혼합 PDF 전체 재인식

**격리 원칙**: OCR 임시 파일은 `.claude/runs/{trace_id}/ocr/` 에만 저장.  
`sources/` 에는 절대 쓰지 않음.

**품질 체크**: OCR 후에도 페이지 평균 100자 미만이면 스캔 품질 불량으로 제외.

### 3-3. OCR 신뢰도 패널티

OCR 텍스트는 오인식으로 WI 매칭 정확도가 낮아진다.  
→ `backfill-matcher`: `ocr_processed: true` 파일에 **confidence × 0.85** 패널티 적용.

예: 원시 점수 85% → OCR 패널티 후 72% → ⚠️ 저신뢰도

이를 통해 OCR 처리 파일이 HITL에서 수동 확인을 유도받도록 설계.

## 6. 핵심 결정 요약

**"이미 있는 문서는 process-do가 아니라 독립 커맨드로"**  
파이프라인 구조(추출→매칭→HITL→매핑)가 process-do와 근본적으로 다름. 독립 커맨드가 명확함.

**"백필 REC는 partial이지 conformant가 아니다"**  
표준 절차를 실제로 이행한 증적이 아님. `legacy_evidence` verdict로 구별하고, 심사원이 판단할 오버라이드 옵션 제공.

**"HITL은 한 번, 전체 테이블로"**  
파일별 멈춤 없이 배치 전체 결과를 한 테이블로 제시. 확정 후 일괄 처리.

**"OCR 처리 파일은 신뢰도를 낮춘다 — ×0.85 패널티"**  
OCR 오인식이 WI 매칭 품질을 낮추는 현실을 confidence에 반영. HITL에서 수동 확인을 자연스럽게 유도.

**"OCR 임시 파일은 격리 — sources/ 불변"**  
원본 파일 오염 방지. OCR 산출물은 `.claude/runs/` 내에만 존재.
