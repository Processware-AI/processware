Process-Orchestration 스킬에 “전처기 기능”을 별도 모듈로 만든다는 방향으로 다시 정리하면 아래와 같습니다.

정리 방향

기존 process-orchestration 또는 processware-orchestrator 스킬은 Plan / Do / Check / Act 중심의 표준 프로세스 생성·실행·감사·개선 루프를 담당합니다.

다만 국제표준 PDF, 법규 문서, 가이드라인, 심사 기준서 등을 바로 /process-plan에 투입하면 문서 구조가 불안정하고 조항 식별, 요구사항 분류, 추적성 확보가 어렵기 때문에, 본격적인 프로세스 생성 전에 별도의 전처기 기능을 둡니다.

즉, 새로 만들 기능의 목적은 다음과 같습니다.

 원문 표준 문서를 Process-Orchestration이 안정적으로 사용할 수 있는 “정규화된 요구사항 데이터셋”으로 변환하는 전처리 모듈

---

기능명 제안

1안: standard-ingestion-preprocessor

국제표준, 법규, RFP, 가이드라인 등의 원문 문서를 수집·분석·정규화하는 전처기

2안: processware-standard-ingestion

Processware 체계에 투입하기 위한 표준 문서 인입·정규화 모듈

3안: /process-ingest

향후 슬래시 커맨드로 확장할 경우 가장 자연스러운 이름

예:

/process-ingest ISO26262.pdf --standard ISO26262 --version 2018
/process-ingest ASPICE_4.0.pdf --standard ASPICE --version 4.0
/process-ingest IEC62304.pdf --standard IEC62304


---

핵심 개념

기존 구조가 다음과 같다면:

/process-plan
→ 표준 분석
→ 프로세스 설계
→ POL / PRO / WI / TMP / EX / MAT / REF 생성


앞단에 새로운 전처리 단계를 추가합니다.

/process-ingest
→ PDF/OCR 추출
→ 문서 구조 분석
→ 조항 단위 분할
→ 요구사항 식별
→ Normative / Informative 분류
→ 요구사항 ID 부여
→ 추적성 메타데이터 생성
→ 정규화된 입력 패키지 생성

/process-plan
→ 정규화된 입력 패키지를 기반으로 표준 프로세스 생성

전처기 기능의 역할

1. 원문 문서 수집 및 추출

대상 문서:

- 국제표준 PDF
- 국내 법규/고시/지침
- 인증 심사 기준
- 고객 RFP
- 내부 표준 문서
- 컨설팅 산출물
- 체크리스트 및 평가모델

처리 내용:

- PDF 텍스트 추출
- 스캔본 OCR 처리
- 페이지, 장, 절, 표, 그림 식별
- 헤더/푸터/페이지 번호 제거
- 목차 구조 인식

---

2. 문서 구조 정규화

원문 문서는 표준마다 구조가 다릅니다.

예:

- ISO 계열: Clause / Sub-clause / Note / Example / Annex
- ASPICE: Process / Base Practice / Generic Practice / Indicator
- CMMI: Practice Area / Practice / Example Work Product
- IEC 62304: Clause / Requirement / Note
- ISO 26262: Part / Clause / Requirement / Work Product

전처기는 이를 공통 구조로 변환합니다.

예시:

standard_id: ISO26262
version: 2018
source_type: international_standard
document_structure:
  - part: Part 6
  - clause: 7
  - sub_clause: 7.4.3
  - title: Software architectural design


---

3. 요구사항 단위 분할

원문 문장을 단순히 페이지 단위로 넘기지 않고, 실제 프로세스 설계에 사용할 수 있는 요구사항 단위로 쪼갭니다.

분할 기준:

- shall
- should
- may
- must
- required to
- is responsible for
- evidence shall be
- work product shall include
- assessment indicator
- audit criterion

결과 예시:

requirement_id: ISO26262-6-7.4.3-REQ-001
source_clause: ISO 26262-6:2018, 7.4.3
requirement_text: >
  The software architectural design shall be developed in accordance with the software safety requirements.
requirement_type: normative
obligation_level: shall


---

4. Normative / Informative 분류

전처기는 문장을 다음과 같이 분류합니다.

- Normative Requirement: 반드시 준수해야 하는 요구사항
- Recommendation: 권고사항
- Informative Note: 참고 설명
- Example: 예시
- Definition: 용어 정의
- Annex Guidance: 부속서 가이드
- Evidence Requirement: 산출물/증적 요구사항
- Process Requirement: 프로세스 수행 요구사항
- Role Requirement: 책임/역할 요구사항
- Tool/Environment Requirement: 도구 또는 환경 요구사항

이렇게 분류해야 /process-plan이 나중에 POL, PRO, WI, TMP, MAT, REF로 정확히 매핑할 수 있습니다.

전처기 산출물 설명

structure.yaml

문서의 장·절 구조를 정리한 파일

standard_id: IEC62304
version: 2006+A1:2015
clauses:
  - id: "5"
    title: Software development process
  - id: "5.1"
    title: Software development planning
  - id: "5.1.1"
    title: Software development plan


requirements.yaml

프로세스 생성의 핵심 입력 데이터

 requirements:
  - id: IEC62304-5.1.1-REQ-001
    clause: "5.1.1"
    text: "The manufacturer shall establish a software development plan."
    type: normative
    obligation: shall
    target_asset_candidates:
      - PRO
      - WI
      - TMP
    evidence_candidates:
      - Software Development Plan


source_map.yaml

추적성을 위한 원문 위치 정보

source_map:
  - requirement_id: IEC62304-5.1.1-REQ-001
    page: 23
    clause: "5.1.1"
    paragraph_index: 2
    source_text_hash: "..."


extraction_quality_report.md

추출 품질 검토 보고서

내용:

- OCR 신뢰도
- 누락 가능 페이지
- 표 추출 실패 여부
- 조항 번호 인식 실패
- 사람이 확인해야 할 영역

Process-Orchestration과의 연결 구조

전처기 기능은 기존 process-plan을 대체하지 않습니다.  
역할은 명확히 분리됩니다.

/process-ingest
= 원문 표준을 기계가 이해 가능한 요구사항 패키지로 변환

/process-plan
= 요구사항 패키지를 기반으로 표준 프로세스 산출물 생성

/process-do
= 생성된 WI 기반 실행 및 REC 생성

/process-check
= REC와 요구사항/프로세스 간 준수성 검증

/process-act
= 부적합, 개선사항, 변경요구 기반 개정 수행

전처기 내부 단계

Phase 1. Source Intake

- 원문 파일 등록
- 표준명, 버전, 발행기관, 문서 유형 입력
- 파일 해시 생성
- 원문 보존

Phase 2. Text Extraction

- PDF 텍스트 추출
- OCR 필요 여부 판단
- 표/그림/각주 분리
- 추출 로그 생성

Phase 3. Structural Parsing

- 목차 인식
- Clause / Sub-clause 구조 생성
- Annex / Note / Example 구분
- 표준별 구조 패턴 적용

Phase 4. Requirement Mining

- shall / should / may 등 의무 표현 탐지
- 요구사항 후보 추출
- 중복 문장 병합
- 복합 요구사항 분리

Phase 5. Classification

- Normative / Informative 분류
- Process / Evidence / Role / Record / Template 요구사항 분류
- 후속 산출물 후보 매핑

Phase 6. Traceability Mapping

- 요구사항 ID 생성
- 원문 위치 매핑
- 조항-요구사항 매트릭스 생성
- 원문 텍스트 해시 저장

Phase 7. QA Review

- 누락 조항 탐지
- OCR 품질 점검
- 요구사항 분할 오류 탐지
- 사람이 검토해야 할 모호 문장 표시

Phase 8. Handoff Package Generation

- /process-plan에 넘길 입력 패키지 생성
- _state.yaml 업데이트
- MAT-002 Regulatory Requirement Mapping 초안 생성 가능

기존 Processware 8종 문서와의 관계

전처기 산출물은 최종 문서가 아니라, 아래 문서 생성을 위한 원재료입니다.

- POL: 정책 원칙 도출에 활용
- PRO: 절차 요구사항 도출에 활용
- WI: 작업지시 수준의 세부 수행 요구사항 도출에 활용
- TMP: 양식/기록 요구사항 도출에 활용
- EX: 예시와 가이드 문구 생성에 활용
- REC: 향후 실행 증적 구조 설계에 활용
- MAT: 요구사항 추적성 매트릭스 생성에 활용
- REF: 원문 표준 요약 및 참조자료로 활용

---

한 문장 정의

 process-ingest 전처기는 국제표준·법규·RFP 등의 원문 문서를 추출, 정규화, 요구사항화, 추적성화하여 Process-Orchestration의 Plan 단계가 안정적으로 표준 프로세스 산출물을 생성할 수 있도록 준비하는 독립 기능이다.

---

권장 결론

따라서 기능 분리는 다음과 같이 잡는 것이 좋습니다.

- process-orchestration: PDCA 기반 프로세스 생성·실행·감사·개선 엔진
- process-ingest: 원문 표준/문서 전처리 및 요구사항 정규화 엔진

즉, process-ingest는 process-orchestration의 내부 보조 기능이라기보다, 앞단의 독립 전처리 레이어로 설계하는 것이 적절합니다.
