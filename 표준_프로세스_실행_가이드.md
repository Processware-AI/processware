## 1. 본 하네스가 하는 일

`/build-standard` 가 만든 표준 자산(POL/PRO/WI/TMP/EX/MAT/REF)을 **계약서**로 삼아, AI Agent 가 사람과 대화하며 절차를 이행하고 **REC(기록본)** 을 자동 작성한다.

```
[기존]  /build-standard ISO9001       → POL/PRO/WI/TMP/EX/MAT/REF (자산 — 차원 1 PLAN)
[신규]  /do WI-CMMI-04-01-03           → REC + MAT-005 갱신       (이행 — 차원 2 DO)
```

## 4. 사용법 (Phase 2)

### 4-1. 7개 진입 모드 (Phase 3 — `rebuild-catalog` 추가)

| 모드 | 명령 | 용도 |
|---|---|---|
| **start (직접)** | `/do WI-CMMI-04-01-03` | 신규 실행 — doc_id 직접 지정 |
| **start (자연어)** | `/do 작업산출물 평가` | 신규 실행 — process-router 가 자동 매칭 |
| **resume** | `/do --resume run-xxxxxxxx` | 정지·중단된 실행 재개 |
| **approve** | `/do --approve run-xxxxxxxx [--approver "박팀장"]` | HITL 승인 응답 |
| **reject** | `/do --reject run-xxxxxxxx --reason "..."` | HITL 반려 응답 (사유 필수) |
| **status** | `/do --status run-xxxxxxxx` | 정지된 trace 의 현재 상태 조회 |
| **check-approvals** | `/do --check-approvals` | 모든 `approval_request.md` 의 drop-in 응답 일괄 회수 |
| **rebuild-catalog** | `/do --rebuild-catalog [--scope CMMI]` | MAT-007 카탈로그 강제 갱신 [Phase 3 신규] |

### 4-2. 공통 옵션

| 플래그 | 효과 |
|---|---|
| `--dry-run` | REC 파일·MAT-005 갱신 생략, 미리보기만 (start/resume/approve 에 적용) |
| `--executor <이름>` | 실행자(작성자) 명시 (start) |
| `--auto-approve` | HITL 자동 승인 모킹 (PoC·테스트용 — 실운영 금지) |
| `--approver <이름>` | 승인/반려 응답자 신원 명시 (approve/reject) |
| `--reason "..."` | 반려 사유 (reject 필수) |

### 4-3. HITL 정식 흐름 (Phase 2 핵심)

```
사용자: /do WI-CMMI-04-01-04 (auto-approve 없이)
        ↓
   [process-executor]
   step-01 ~ step-N 진행 ...
        ↓
   step-N (hitl: required) 도달
        ↓
   [hitl-gatekeeper] gate_enter 모드
   - state.status: pending_approval
   - approval_request.md drop-out
   - 정지 신호 반환
        ↓
   ⏸ 종료 — 사용자에게 "/do --approve <trace_id>" 안내

      ⏳ (시간 경과 — 같은 세션이든 다른 세션이든)

사용자: /do --approve run-xxxxxxxx --approver "박팀장"
        ↓
   [hitl-gatekeeper] gate_response 모드
   - state.hitl.decision: approved
   - state.status: ready_to_finalize
        ↓
   [process-executor] resume_after_approval 모드
   - 정지된 step 의 derivation·outputs 마무리
   - rec_payload.yaml 작성
        ↓
   [rec-writer] finalize 모드
   - REC 생성, MAT-005 갱신
        ↓
   ✅ 완료
```

### 4-4. 반려 흐름

```
사용자: /do --reject run-xxxxxxxx --reason "표본 부족 — 재수집 필요"
        ↓
   [hitl-gatekeeper] gate_response (rejected)
   - state.hitl.decision: rejected
   - state.hitl.rejection_reason: "표본 부족 ..."
        ↓
   [rec-writer] rejected 모드
   - REC.status: rejected
   - 본문 첫머리 경고문 + 반려 사유
   - MAT-005 상태: ❌ 반려
        ↓
   ❌ 반려 마감 — 시정조치 후 /do {WI번호} 재실행
```

### 4-5. drop-in 응답 (외부 채널 모킹)

승인자가 `/do --approve` 명령 대신 **`approval_request.md` 파일을 직접 편집** 해서 응답할 수도 있다 (외부 알림 시스템 모킹):

```yaml
# .claude/runs/run-xxxxxxxx/approval_request.md frontmatter
---
status: approved          # pending → approved 또는 rejected
approver_name: 박팀장
responded_at: 2026-05-01T18:00:00+09:00
reason: ...               # rejected 시
---
```

저장 후 `/do --check-approvals` 실행 → 모든 drop-in 응답이 자동으로 회수되어 처리.

### 4-6. 자연어 진입 (Phase 3 — process-router 정식)

#### 4-6-1. 동작 모드 3가지

```
/do 작업산출물 평가             → confidence 0.95 → 자동 매칭 → 실행 시작
/do 부적합 보고                 → confidence 0.78 → 후보 3건 제시 → 사용자 [1/2/3]
/do 매우 모호한 키워드          → confidence < 0.3 → 구체화 요청 + 종료
```

| confidence 결과 | mode | UI 동작 |
|---|---|---|
| 1순위 ≥ 0.9 AND 2순위와 차이 ≥ 0.15 | `auto_accepted` | "자동 매칭: WI-X (confidence 0.95)" + Phase A 진입 |
| 위 조건 미충족, 1순위 ≥ 0.3 | `candidates_presented` | top-3 후보 표 + reasoning 1줄 + 사용자 선택 |
| 모든 후보 < 0.3 | `no_match` | "구체화 요청. 어느 영역의 어떤 활동입니까?" + 종료 |

#### 4-6-2. 매칭 입력 컨텍스트

`process-router` 가 LLM 컨텍스트로 사용:
- `vault/90_MAT_통합매핑/MAT-007_프로세스_카탈로그.md` §5 YAML 인덱스 (정밀 인덱싱 항목)
- §3 빠른 검색표 (보조 fallback)
- 동의어 매핑 가이드라인 (process-router §3.2)

#### 4-6-3. 카탈로그 갱신

자동 (차원 1 빌드 시):
```
/build-standard CMMI-DEV-ML3      # → traceability-mapper 가 MAT-007 자동 갱신
```

수동 (차원 2 단독):
```
/do --rebuild-catalog             # 전체 표준
/do --rebuild-catalog --scope CMMI  # 특정 영역만
```

`manual_override: true` 표기된 행은 자동 갱신 시 보존됩니다.

### 4-7. 자연어 매칭 정확도 향상 팁

- 카탈로그가 비어있으면 매칭 실패 — `/do --rebuild-catalog` 먼저 실행.
- 도메인 단서(자동차/의료기기/AI) 를 입력에 포함하면 매칭 정확도 ↑.
- "매우 모호한" 입력은 후보 제시 모드로 폴백 (자동 채택 안 됨) — 정직한 동작.
- 자주 쓰는 표현이 카탈로그에 없으면 `aliases[]` 를 직접 편집 + `manual_override: true` 표기.

## 8. 안전 규칙 (4가지 강제)

1. **자산 영역 무결성** — 본 하네스는 `vault/03_POL/`, `04_PRO/`, `05_WI/`, `06_TMP/`, `07_EX/`, `09_REF/`, `90_MAT/MAT-001~004,006~` 의 어떤 파일도 수정하지 않는다. 위반 즉시 중단.
2. **REC 직접 수정 금지** — 일단 생성된 REC 는 심사 증적이므로 사람이 손대지 않는다. 정정 시 신규 REC 발행 (Phase 2 `--reissue`).
3. **trace.jsonl 누락 금지** — 모든 질문/답변/판정/승인 이벤트가 단일 JSON 라인으로 기록되어야 한다. 누락은 차원 3 심사에서 부적합 사유.
4. **번호 충돌 자동 회피** — 일련번호 동시 발생 시 자동으로 다음 번호 채택. 충돌 사실은 trace 에 `seq_collision_resolved` 메모.
